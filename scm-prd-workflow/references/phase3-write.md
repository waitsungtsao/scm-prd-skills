# Phase 3: 方案输出 — PRD撰写指引

## 目标

基于Phase 1-2的完整需求信息，产出结构完整、描述精准、可直接用于开发评审的PRD文档。

## 写作总原则

### 信息分层

PRD是给不同读者看的，信息需要分层：
- **概览层**（背景/目标/范围）：给决策者和不熟悉业务的人看
- **方案层**（流程/变更/功能）：给开发、测试、业务方看
- **细节层**（字段/接口/规则）：给开发和测试看

### 语言规范

**必须做到**：
- 使用确定性表述："系统应""系统必须"，不用"系统可以""系统尽量"
- 规则量化：不说"及时处理"，说"30分钟内处理"
- 条件穷举：不说"等情况"，逐一列出所有情况
- 主语明确：不说"应处理"，说"WMS系统应处理"

**绝不能出现的表述**：
- "可配置"（配什么？谁配？在哪配？默认值是什么？）
- "灵活处理"（怎么灵活？处理规则是什么？）
- "适当的"/"合理的"（标准是什么？）
- "参考XX系统"（具体参考什么逻辑？）
- "等"字（穷举出来）

### 简洁性规范

PRD 输出必须精准干练，遵循以下标准：

**禁止冗余**：
- 同义反复：同一信息不得用不同措辞重复表述。错误示例："系统自动分配仓库，即由系统根据规则自动选择合适的仓库进行分配"
- 填充句：删除不传递信息的句子。禁止出现："本章是PRD的核心""以下将详细描述""众所周知""不言而喻"
- 过度解释：规则本身已清晰时不附加解释。错误示例："金额精度为2位小数（即保留到分，例如12.34元）"——括号内容删除
- 上下文重复：已在前文或其他章节描述过的内容，用交叉引用（G/C/F/IF/BR-XXX）代替重复叙述

**文体要求**：
- 动词优于名词化：写"系统分配仓库"，不写"系统执行仓库分配操作"
- 表格优于段落：结构化信息（规则、字段、参数）一律用表格，不用文字段落铺陈
- 短句优于长句：单句不超过40字。超过时拆分为多句或用列表
- 删除无信息修饰词："进行""相关的""一定程度上""基本上""总体来说""需要注意的是"

**自检口诀**：每句话问自己——"删掉这句，读者会丢失信息吗？"如果不会，删掉。

### AI建议标注

如果你在撰写过程中基于专业判断补充了用户未明确提出的内容（如建议增加某个异常处理、建议增加某个字段），必须用以下标注：

```markdown
> [建议] 建议增加XXX功能/规则/字段。
> 理由：...
> 如用户同意此建议，请确认，我将移除此标注写入正文。
```

未经用户确认，标注的内容不得出现在正式PRD中。

### `[建议]` 的选项化处理

**批量处理**：PRD撰写完成后存在多条 `[建议]` 时，使用 `AskUserQuestion` 让用户选择：**全部采纳** / **逐条审阅** / **全部拒绝**。

**逐条审阅**：对每条建议使用 `AskUserQuestion`（含建议原文和理由），选项：**采纳** / **拒绝** / **修改后采纳**（进入自由文本）。

## 按需生成章节（requirement_type = update|mixed 时）

### 核心原则

好的PRD是一个好的故事——聚焦于需求变更本身，逻辑闭环，无冗余。当需求类型为"功能优化/变更"时：

1. **只生成有实质内容的章节**：未涉及变更的方面不独立成节。空壳章节（如"本次不涉及XX"）会打断叙事连续性
2. **Ch.2 §2.5 范围中一次性说清**：哪些方面在本次范围内、哪些不在，一目了然
3. **PRD尾部统一说明**：在最后一章末尾，用一段话汇总本次未涉及的方面，替代散落各处的声明

### 章节生成决策

根据 intake.md §4.0 变更范围，决定哪些章节需要生成：

| 章节 | 涉及变更时 | 不涉及变更时 |
|------|-----------|------------|
| Ch.2 需求概述 | 始终生成 | 始终生成 |
| Ch.3 术语定义 | 有新术语时生成 | 无新术语则跳过 |
| Ch.4 业务变更总览 | 始终生成 | 始终生成 |
| Ch.5 业务流程 | 生成完整流程图 | **不生成** |
| Ch.6 功能与规则明细 | 仅描述变更/新增的功能点 | **不生成** |
| Ch.7 §7.2 数据模型 | 新实体/关系变更时生成 | **不生成** |
| Ch.7 §7.3 数据变更规格 | 有字段变更时生成 | **不生成** |
| Ch.7 §7.4-7.8 接口与集成 | 仅描述变更/新增的接口 | **不生成** |
| Ch.8 非功能性需求 | 描述变更的NFR | **不生成** |
| Ch.9 验收标准 | 始终生成（对应实际变更） | 始终生成 |
| Ch.10 待定事项 | 有待定项时生成 | 无待定项则跳过 |

### 尾部汇总格式

在PRD最后一个实质章节之后，附一段汇总：

```markdown
---
**本次需求聚焦于{变更主题}。以下方面沿用现有系统行为，未在本文档中展开描述：{逐一列出未涉及的方面}。**
```

### 系统公约的作用

当 `knowledge-base/system-conventions.md` 存在时，AI以此为参考判断"哪些是标准行为可以不写"。系统公约是AI的内部参考材料，**不在PRD正文中引用或提及**。

## 各章节撰写指引

### 第1章: 文档信息

```markdown
| 项目 | 内容 |
|------|------|
| 文档名称 | PRD-{需求名称} |
| 版本 | V1.0 |
| 作者 | {用户姓名} + AI辅助 |
| 创建日期 | {日期} |
| 状态 | 待评审 |
| 需求编号 | {REQ编号} |
```

### 第2章: 需求概述

- **§2.1 背景与动机**：从 intake.md 提取，包含业务问题和改进动机
- **§2.2 业务现状（AS-IS）**：当前业务流程和系统现状的**唯一完整描述**。后续章节（如第4章变更表）仅用一行摘要回指此节
- **§2.3 痛点**：量化描述问题的影响
- **§2.4 目标与指标**：列出可量化的目标，每个目标使用 G-XXX 编号（如 G-01），包含当前值→目标值和预期价值。目标必须与§2.3痛点对应
- **§2.5 范围**：直接从 clarification.md 提取 in scope / out of scope / 将来考虑
- **§2.6 需求合理性分析**：仅自主模式生成（详见下方"自主模式撰写调整"）

背景和现状描述应基于Phase 1用户确认的内容，不要添加AI的推测。

### 第3章: 术语定义

- 从知识库的 glossary.yaml 中提取本PRD涉及的术语
- 如有本PRD新引入的术语，也在此定义
- 表格形式：术语 | 全称 | 定义 | 首次出现章节

### 第4章: 业务变更总览

这个章节让读者快速知道"改了什么"。每个变更项使用 C-XXX 编号。

**关键原则**：AS-IS 列仅写一行摘要，详细现状回指第2章§2.2，不再重复叙述。

```markdown
| 编号 | 变更项 | AS-IS 摘要（详见Ch.2§2.2） | 目标（TO-BE） | 影响系统 | 影响范围 |
|------|--------|---------------------------|-------------|---------|---------|
| C-01 | 订单分配规则 | 手动选择仓库 | 系统按距离自动分配 | OMS, WMS | 订单创建流程 |
| C-02 | 距离计算服务 | （无） | 新增距离计算 | OMS | 订单创建流程 |
```

AS-IS列为"（无）"即表示新增，AS-IS有值即表示变更，无需额外标记列。

### 第5章: 业务流程

本章描述流程的**整体走向和状态流转**，不包含步骤表。步骤的详细操作和规则在第6章各功能点中描述。

**主流程（§5.1）**：
- 泳道图使用 YAML DSL 描述（规范见 `references/diagram-yaml-schema.md`），泳道横向排列为列，流程从上往下
- 生成 `.diagram.yaml` 后，根据初始化时检测的 Python 环境决定是否转换：
  - `python_available = true` → 调用 `{python_cmd} scripts/yaml2drawio.py` 转换为 `.drawio`（`python_cmd` 为初始化时检测到的命令）
  - `python_available = false` → 仅保留 `.diagram.yaml` 源文件，或按用户选择回退到 Mermaid（参见 SKILL.md 3.2 节降级处理）
- 可选在 PRD 中嵌入简化 Mermaid 版本用于文档内预览
- **流程图节点应标注关联的功能编号**（F-XXX），以便读者跳转到第6章查看详细步骤和规则
- 用简要文字概述流程走向，**不使用步骤表**

**Python 不可用时的降级说明**：当 `python_available = false` 时，回退方式为 AI 直接生成 Mermaid 格式的流程图（使用 `subgraph` 模拟泳道），而非自动将 YAML 转换为 Mermaid。此时不生成 `.diagram.yaml` 文件。

**分支/异常流程（§5.2）**：
- 从主流程中标出可能的分支点
- 每个分支画子流程或在主流程图中标注
- 异常处理必须到"最终处理结果"，不能停在"通知相关人员"

**状态流转（§5.3）**：
- 用Mermaid状态机图表达
- 每个状态转换标注条件和操作

### 第6章: 功能与规则明细

本章是PRD的核心，按功能点逐一描述。每个功能点使用 F-XXX 编号，采用**统一步骤-规则表**格式，将操作步骤、业务规则和异常处理合并在一张表中，避免重复。

```markdown
### F-001 {功能点名称}

**功能描述**：{一句话描述}
**操作角色**：{角色}  |  **前置条件**：{条件}
**关联变更**：C-XX

**操作步骤与规则**：

| 步骤 | 操作 | 业务规则 | 异常处理 |
|------|------|---------|---------|
| 1 | 用户提交订单 | BR-001: 金额>0，必填字段校验 | E-001: 字段缺失→提示补全 |
| 2 | 系统自动分仓 | BR-002: 距离最近+库存充足 | E-002: 无可用仓→通知运营 |

**跨步骤规则**（可选，仅当规则跨多步骤时）：
| 规则编号 | 规则描述 | 涉及步骤 | 触发条件 | 处理逻辑 |

**字段/页面说明**（如涉及前端）：
| 字段名 | 类型 | 必填 | 默认值 | 校验规则 | 说明 |
```

**步骤-规则表拆分指引**：
- ≤2条规则：同一单元格
- >2条：拆为子步骤（2a、2b等）
- 跨多步骤的规则：使用"跨步骤规则"表承载

**ID编号规范**：
- F-XXX：功能点编号，三位数字，如 F-001、F-002
- BR-XXX：业务规则编号，在步骤-规则表中定义
- E-XXX：异常处理编号，在步骤-规则表中定义
- IF-XXX：接口编号，在第7章定义，功能中引用

**注意**：每个功能点的"关联变更"字段标注其实现的变更项（C-XXX），形成与第4章的交叉引用。

### 第7章: 接口与数据集成

本章合并了数据模型、数据流向、接口需求、数据一致性与迁移。

- **§7.1 数据流总览图**：用Mermaid画出数据流向图，说明每条数据流的来源、内容、传输方式、触发时机、频率

- **§7.2 数据模型**（按需生成）：

  **生成触发规则**：
  | 场景 | ER 图 | 实体说明表 | 字段变更清单(§7.3) |
  |------|-------|-----------|------------------|
  | 新建系统（requirement_type=new） | **必须** | **必须** | 不适用 |
  | 新增实体 | **必须** | **必须** | 仅新增字段 |
  | 实体关系变更 | **必须** | **必须** | 变更字段 |
  | 仅字段变更（无新实体、无关系变更） | 不生成 | 不生成 | **必须** |
  | 无数据模型变更 | 不生成 | 不生成 | 不生成 |

  **ER 图格式**：
  - **Mermaid erDiagram**：嵌入 PRD 文档内，用于快速预览
  - **YAML → draw.io**（当实体数 >5 或关系复杂时）：详细 ER 图保存为 `.diagram.yaml` + `.drawio`

  **实体说明表**：对 ER 图中每个实体的补充说明，含主键、所属系统、是否新增。

  **实体关系说明**：用文字描述每条关系的业务含义，辅助理解 ER 图。

  如流程设计或数据模型存在多个备选方案，记录到 decision-log.md。

- **§7.3 数据变更规格**（requirement_type = update|mixed 且涉及数据结构变更时生成）：
  - §7.3.1 字段变更清单：逐个列出新增/修改/删除的字段，每个字段关联 F-XXX
  - §7.3.2 变更影响分析：分析字段变更对接口、报表、计算规则、索引的影响

- **§7.4 接口清单**：每个接口使用 IF-XXX 编号
- **§7.5 接口详细定义**：每个新增/变更的接口：

```markdown
#### IF-001: {接口名称}

- 接口方向: {调用方} → {被调用方}
- 触发时机: ...
- 调用方式: {HTTP/MQ/RPC/文件}
- 是否新增: 是/否（变更）
- 关联功能: F-XXX

**请求参数**:
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

**响应参数**:
| 字段 | 类型 | 说明 |
|------|------|------|

**异常处理**:
- 超时: ...
- 调用失败: ...
- 幂等策略: ...
```

  如接口方案存在技术选型决策，记录到 decision-log.md。

- **§7.6 数据一致性**：事务方案/最终一致性/补偿机制
- **§7.7 数据迁移**（如涉及）：迁移范围、映射关系、执行计划、回滚方案
- **§7.8 上线初始化**：初始数据准备、配置项初始化

### 第8章: 非功能性需求

从 clarification.md 提取，补充具体指标。

### 第9章: 验收标准

每个功能点至少一条验收标准。验收表增加"关联目标"和"关联功能"列，通过 G-XXX 和 F-XXX 编号与第2章目标和第6章功能形成交叉引用：

```markdown
| # | 验收项 | 前置条件 | 操作步骤 | 预期结果 | 关联目标 | 关联功能 |
|---|--------|---------|---------|---------|---------|---------|
| 1 | 正常订单分配 | 存在库存充足的仓库 | 创建订单 | 系统自动分配最近仓库 | G-01 | F-001 |
```

### 第10章: 待定事项与附录

本章合并了待定事项和附录。

- **§10.1 待定问题**：从 clarification.md 提取所有 [待定] 项
- **§10.2 假设清单**：从 clarification.md 提取所有 [假设] 项
- **§10.3 假设总览**：仅自主模式生成（详见下方"自主模式撰写调整"）
- **§10.4 参考资料**：参考文档和链接
- **§10.5 关联需求**：相关PRD链接
- **§10.6 知识库引用**：引用的知识库文件

## 交叉引用 ID 体系

PRD中使用统一的ID编号体系实现交叉引用，确保每个事实只写一次：

| 前缀 | 定义位置 | 示例 | 引用位置 |
|------|---------|------|---------|
| G-XXX | Ch.2§2.4 目标 | G-01: 分仓时间 2h→10min | Ch.9 验收标准"关联目标"列 |
| C-XXX | Ch.4 变更项 | C-01: 订单分配规则变更 | Ch.6 功能说明中"关联变更"字段 |
| F-XXX | Ch.6 功能 | F-001: 自动分仓 | Ch.5 流程图节点标注, Ch.9 验收标准"关联功能"列 |
| IF-XXX | Ch.7§7.4 接口 | IF-001: OMS→WMS出库指令 | Ch.6 功能中引用, Ch.7§7.5 详细定义 |
| BR-XXX | Ch.6 业务规则 | BR-001: 距离优先分仓 | Ch.9 验收标准可引用 |
| DL-XXX | decision-log.md | DL-001: 分仓策略选择 | PRD 正文中以轻量链接引用 |

## ID 分配规划

**在开始逐章撰写前，必须先完成 ID 分配规划**。这一步是避免"边写边编号"导致 ID 跳号、重复或引用断裂的关键保障。

```markdown
## ID 分配方案
- 目标: G-01 ~ G-0N（每个目标一个编号）
- 变更项: C-01 ~ C-0N（每个变更点一个编号）
- 功能点: F-001 ~ F-00N（每个功能一个编号）
- 接口: IF-001 ~ IF-00N（每个新增/变更接口一个编号）
- 业务规则: BR-001 ~ BR-0NN（按功能点分段分配，F-001 用 BR-001~BR-009，F-002 用 BR-010~BR-019，以此类推）
- 异常处理: E-001 ~ E-0NN（编号策略同 BR，按功能点分段）
- 决策记录: DL-001 ~ DL-00N（每个有多方案比选的决策点一个编号）
```

**执行要求**：
1. 此方案在自主模式下作为 PRD 大纲的一部分呈现给用户确认；在交互模式下作为 AI 内部撰写参考
2. 撰写过程中**严格按照规划分配编号**，不得临时新增未规划的编号
3. 如撰写中发现需要新增功能点或规则，先更新 ID 分配方案再继续撰写
4. 撰写完毕后，执行 `check-prd-consistency.py` 验证交叉引用完整性（详见下方"撰写后一致性检查"）

## 决策日志生成规则

在撰写 PRD 过程中，当遇到以下情况时，创建决策记录（DL-XXX）并写入 `decision-log.md`（使用 `templates/decision-log-template.md`）：

**触发条件**：
1. 存在 ≥2 个可行方案且需要取舍时
2. 用户在 Phase 2 / Stage A 中明确讨论过的方案选择
3. AI 在自主模式下推断选择了某方案，且存在其他合理替代方案
4. 涉及业务规则的关键判断（如计算公式选择、异常处理策略选择）

**不触发条件**：
- 仅有一种合理实现方式的技术细节
- 用户明确指定且无替代方案的需求
- 纯格式/结构决定

**PRD 正文引用格式**：

在功能描述或业务规则中涉及方案决策的位置，添加轻量链接：

```markdown
**关联决策**：DL-001（分仓策略选择，选择方案C：综合评分）
```

或在规则说明段落末尾：

```markdown
> 决策背景见 [DL-001](decision-log.md#dl-001)
```

## 撰写后一致性检查

PRD 撰写完毕后、进入自检（Phase 4 / Stage B CK 检查）之前，先运行一致性扫描脚本：

```bash
{python_cmd} scripts/check-prd-consistency.py requirements/REQ-{日期}-{名称}/PRD-{名称}.md
```

脚本检查内容：
1. **ID 交叉引用完整性**：所有 G/C/F/IF/BR 编号是否在定义位置和引用位置成对出现
2. **模糊用语扫描**：检测"大概""可能""适当"等模糊表述
3. **变更覆盖检查**：每个 C-XX 变更项是否在功能点的"关联变更"中被引用

如脚本报告关键问题先修复再进入自检；Python 不可用时跳过，由 AI 在自检中手动执行等效检查。

## 输出要求

> - **仅Markdown**：仅保留Markdown格式
> - **仅Word**：仅生成Word版本，适合正式交付

其他要求：
1. 所有图表保存到 `diagrams/` 目录，文件产出规则：
   - `.diagram.yaml` → `yaml2drawio.py`(**.drawio**) + `yaml2svg.py`(.svg)
   - `.mermaid` → 源文件保留
   - **`.drawio` 为默认产出**：`python_available = true` 时必须自动生成，无需用户确认
   - `.drawio` 始终保留，供用户在 draw.io 中手动编辑
2. **图表导出**：PRD 撰写完成后，调用 `scripts/export-diagrams.py` 统一处理：
   - 阶段1：`.diagram.yaml` → `yaml2drawio.py` → **.drawio**（默认生成）
   - 阶段2：`.diagram.yaml` → `yaml2svg.py` → .svg + .png（需 cairosvg）
   - 阶段3：`.mermaid` → mermaid.ink API → .png（需联网）

3. **降级处理**（详见 SKILL.md "draw.io 生成降级策略 DG-01"）：

   - **降级**：`docx_available = false` 时提示 `pip install python-docx`
   - Markdown 始终是主要交付物，Word 不阻断交付

## 自主模式撰写调整

当处于自主模式（Stage B）时，PRD撰写行为有以下调整：

### PRD 大纲确认

在正式撰写前，先输出 PRD 大纲供用户确认（详见 `references/autonomous-mode.md` 中的"PRD 大纲确认"章节）。用户确认后再开始逐章撰写。

### 一次性生成

- 大纲确认后，10个章节一次性生成，不需要逐章等待用户确认
- 生成过程中不中断、不提问（除非遇到绝对无法推断的红线信息）

### 标记规则变化

自主模式**不使用** `[建议]` 标记，改用三级标记体系：

| 交互模式 | 自主模式 | 说明 |
|---------|---------|------|
| `[建议]` + 等待确认 | （无标记） | 用户明确说过的内容，直接写入 |
| `[建议]` + 等待确认 | `[推断]` | AI基于专业判断补充的行业标准做法 |
| 提问等待回答 | `[待确认]` + 当前假设 | AI不确定的业务决策 |

### 质疑的文档化

交互模式中通过对话进行的质疑，在自主模式中写入文档：

- 在PRD **第2章§2.6**增加"需求合理性分析"段落
- 列出AI对需求的审视意见（是否在解决根因、是否有更简单方案、ROI是否合理等）
- 该段落作为PRD正式内容，不使用 `[建议]` 标记

### 知识库引用

引用知识库内容时标注来源：

```markdown
当前出库流程为"拣货→复核→打包→发运"四步流程。
（参考: knowledge-base/domain-wms.md §核心流程）
```

### 假设总览表

自主模式生成的PRD，在第10章§10.3 中必须包含假设总览表：

```markdown
### 10.3 假设总览（自主模式）

| # | 类型 | 章节 | 假设内容 | 影响程度 | 状态 |
|---|------|------|---------|---------|------|
| 1 | [待确认] | 5.2 | 库存不足时仅通知，不自动采购 | 高 | 待审阅 |
| 2 | [推断] | 6 F-003 | 接口超时重试3次后转人工 | 中 | 如无异议视为确认 |
```

该表是 Stage C 审阅的核心工具，用户可通过此表快速审阅所有AI假设。
