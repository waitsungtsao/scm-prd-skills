# Phase 2: 需求澄清 — 操作指引

## 目标

基于Phase 1的需求摘要，系统化地识别所有模糊点和决策点，逐项与用户确认，产出无歧义的需求规格。

## 澄清问题生成策略

### 输入

读取 `intake.md`，逐段分析以下内容：
- 哪些描述用了模糊词（"大概""可能""一般""适当"）？
- 哪些业务规则只描述了正常场景，缺少异常分支？
- 哪些系统交互只提了方向，没有具体接口和字段？
- 哪些角色操作只提了"能做什么"，没说"不能做什么"？

### 变更范围过滤

当 `requirement_type = update|mixed` 时，intake.md §4.0 已记录本次涉及变更的方面和不涉及的方面。澄清策略：

- **涉及变更的方面**：完整展开澄清（与当前行为一致）
- **不涉及变更的方面**：跳过对应维度的详细澄清。如果在澄清过程中发现某个"不涉及"的方面实际上受影响，主动提醒用户并调整变更范围

`requirement_type = new` 时跳过此过滤，所有维度按需展开。

### 澄清维度清单

按以下维度逐一生成问题。不是每个维度都需要，根据具体需求选择。

#### 维度1: 业务边界

目标：明确的 in scope / out of scope

- 本次需求涉及哪些具体场景？请逐一列出（自由文本）
- 以下场景是否在本次范围内？→ **使用 `AskUserQuestion`**（对每个推测场景逐一确认）：
  > header: "场景范围"
  > 选项：**在范围内** / **排除** / **将来考虑**
- 有没有"将来再做"的内容？需要在本次设计中预留扩展性吗？→ **使用 `AskUserQuestion`**：
  > header: "扩展性"
  > 选项：**需要预留** / **不需要** / **待定**

#### 维度2: 业务数据流

目标：数据从哪来、怎么处理、到哪去

- 本功能的输入数据来自哪个系统/模块？通过什么方式获取？→ **使用 `AskUserQuestion`**：
  > header: "数据获取"
  > 选项：**API接口调用**（系统间标准接口） / **消息队列**（异步事件驱动） / **数据库同步**（直连或CDC） / **文件传输**（FTP/SFTP/对象存储）
- 数据处理的核心逻辑是什么？（计算公式/匹配规则/转换规则）（自由文本）
- 处理后的数据输出到哪里？以什么形式？（自由文本）
- 数据的时效性要求？→ **使用 `AskUserQuestion`**：
  > header: "数据时效"
  > 选项：**实时**（毫秒级同步） / **准实时（秒级）**（消息队列等秒级延迟） / **T+1批处理**（每日定时批量处理） / **按需触发**（用户操作或事件触发时处理）
- 数据量级？（日均/峰值）（自由文本）

#### 维度3: 系统交互

目标：接口级别的清晰定义

- 涉及哪些现有接口的变更？具体变更哪些字段？（自由文本）
- 需要新增哪些接口？调用方和被调用方分别是谁？（自由文本）
- 接口调用的触发时机是什么？（自由文本）
- 接口调用失败时的处理策略？→ **使用 `AskUserQuestion`**：
  > header: "失败处理"
  > 选项：**自动重试**（按策略自动重试N次后告警） / **告警通知人工处理**（直接告警由人工介入） / **降级方案**（启用备选逻辑保证业务不中断） / **直接失败记录**（记录日志后续批量处理）
- 是否需要考虑接口的版本兼容？→ **使用 `AskUserQuestion`**：
  > header: "版本兼容"
  > 选项：**需要（新老并行）**（上线期间新老版本共存） / **不需要**（一次性切换） / **待评估**（需技术团队评估）

#### 维度4: 异常处理

目标：穷举异常场景及处理策略

对intake.md中提到的每个流程步骤，追问：
- 这一步如果失败/超时/数据异常，怎么处理？（自由文本）
- 上游数据缺失或格式错误时，怎么处理？（自由文本）
- 并发操作时的冲突如何解决？→ **使用 `AskUserQuestion`**：
  > header: "并发策略"
  > 选项：**乐观锁（先到先得）**（后提交者收到冲突提示需重新操作） / **悲观锁（锁定操作）**（操作期间锁定记录，他人等待） / **队列化处理**（操作进入队列依次执行） / **业务规避**（通过业务规则避免并发场景）
- 系统不可用时的降级方案？→ **使用 `AskUserQuestion`**：
  > header: "降级方案"
  > 选项：**人工流程兜底**（切换线下人工操作保证业务连续） / **排队等待恢复**（请求排队等系统恢复后处理） / **告警跳过**（记录异常跳过当前步骤） / **无降级方案**（系统不可用时业务暂停）

#### 维度5: 状态与流转

目标：完整的状态机定义

- 涉及的实体有哪些状态？（自由文本）
- 每个状态转换的触发条件是什么？（自由文本）
- 有没有不可逆的状态转换？→ **使用 `AskUserQuestion`**：
  > header: "不可逆状态"
  > 选项：**有**（需明确列出哪些转换不可逆） / **没有**（所有状态转换均可回退） / **不确定**（需进一步梳理）
- 状态转换时需要记录什么日志？（自由文本）
- 有没有自动超时转换的场景？（自由文本）

#### 维度6: 权限控制

- 哪些角色可以执行哪些操作？（自由文本）
- 有没有审批流程？→ **使用 `AskUserQuestion`**：
  > header: "审批流程"
  > 选项：**无需审批**（操作直接生效） / **单级审批**（一个审批人确认即可） / **多级审批**（需多级逐级审批） / **条件审批（按金额等）**（达到条件触发审批，否则直接通过）
- 数据的可见范围？（如仓库A的人能不能看到仓库B的数据）（自由文本）
- 有没有"只读"和"可操作"的区分？（自由文本）

#### 维度7: 非功能性需求

- **性能**：页面响应时间要求？→ **使用 `AskUserQuestion`**：
  > header: "响应时间"
  > 选项：**≤1秒**（高频操作页面） / **≤3秒**（普通操作页面） / **≤5秒**（复杂查询/报表页面） / **无明确要求**（按通用标准）
- **性能**：批处理的时效要求？（自由文本）
- **并发**：预计同时操作的用户数？（自由文本）
- **安全**：有没有敏感数据？需要加密或脱敏？（自由文本）
- **兼容性**：需要支持哪些浏览器/设备？（自由文本）
- **可用性**：系统可用性要求？→ **使用 `AskUserQuestion`**：
  > header: "可用性"
  > 选项：**99.9%**（年停机≤8.8小时，一般业务系统） / **99.99%**（年停机≤53分钟，核心交易系统） / **与现有系统一致**（沿用当前SLA标准） / **无明确要求**（按运维团队标准）

#### 维度8: 风险识别

- 这个需求上线后，最可能出什么问题？
- 如果方案有缺陷，最坏的业务后果是什么？
- 有没有依赖外部团队/系统的部分？他们的配合确定性如何？
- 数据迁移/灰度上线的风险点？

#### 维度9: 数据迁移与初始化

（仅当涉及数据结构变更时）
- 历史数据需要如何处理？→ **使用 `AskUserQuestion`**：
  > header: "历史数据"
  > 选项：**全量迁移**（所有历史数据完整迁移） / **仅迁移活跃数据**（只迁移近期有业务活动的数据） / **不迁移**（新系统从零开始，旧数据归档查询） / **分批迁移**（按时间/业务维度分批次迁移）
- 新老数据格式的映射关系？（自由文本）
- 迁移过程中的业务影响？需要停机吗？→ **使用 `AskUserQuestion`**：
  > header: "迁移方式"
  > 选项：**需要停机**（安排维护窗口，停机迁移） / **不需要（在线迁移）**（双写或CDC方案，不影响业务） / **待评估**（需技术团队评估迁移方案）
- 初始化数据的来源和准备方式？（自由文本）

## 逐项确认的对话策略

### 分批确认

不要一次性抛出所有问题。按维度分批，先用 `AskUserQuestion` 让用户选择维度顺序：

> header: "维度顺序"
> 问题: "澄清问题共涉及以下维度，你希望按什么顺序进行？"
> 选项：
> - **按默认顺序**：按业务边界→数据流→系统交互→…的标准顺序逐维度确认
> - **优先核心维度**：先确认业务边界、异常处理等高影响维度
> - **我来选择**：自行决定每次讨论哪个维度

操作流程：
1. 先呈现问题清单概览（各维度各有多少问题）
2. 按用户选择的顺序进行
3. 每个维度内逐项确认（使用选项化交互，见下方"确认标记"和"维度内领域专属选项"）
4. 每确认完一个维度，使用 `AskUserQuestion` 确认维度完成：

> header: "维度完成"
> 问题: "{维度名称}的澄清已完成，接下来？"
> 选项：
> - **确认，下一维度**：当前维度结论无误，继续
> - **有修改**：需要修改当前维度的部分结论
> - **跳过剩余维度**：已有信息足够，跳过后续维度

### 确认标记

每个问题回答后，使用 `AskUserQuestion` 标记状态：

> header: "状态"
> 问题: "这一项的确认状态？"
> 选项：
> - **已确认**：答案明确，可作为PRD依据
> - **待定**：需要找其他人确认（后续追问需要找谁、预计何时确认）
> - **排除**：不在本次范围内
> - **先假设**：暂按当前理解推进，后续如不成立需重新评估

**批量标记**：当某维度有10个以上问题时，先让用户回答所有问题，再批量标记状态（避免逐条确认的疲劳感）。此时使用一次 `AskUserQuestion`（multiSelect）让用户选出需要特殊标记的项，其余默认为"已确认"。

### 矛盾处理

如果用户在Phase 2的回答与Phase 1的描述矛盾，使用 `AskUserQuestion` 并用 markdown 展示矛盾内容：

> header: "矛盾确认"
> 问题: "发现前后描述不一致，请确认以哪个为准？"
> 选项（每个选项用 markdown 展示对应内容）：
> - **以 Phase 1 为准**：markdown 展示 Phase 1 中的原文
> - **以最新回答为准**：markdown 展示 Phase 2 中的最新回答
> - **都不对，我重新说明**：进入自由文本补充

## 输入文件格式校验

在开始 Phase 2 之前，对 `intake.md` 执行基本格式检查：
- YAML front matter 必填字段：`type`、`requirement_id`、`phase`、`status`
- 章节标题存在性：至少包含"需求概述""痛点""目标"等核心章节
- 如格式校验失败，提示用户检查 intake.md 格式或回退到 Phase 1 修正

## 产出规范

Phase 2完成后，输出 `clarification.md`。使用 `templates/clarification-template.md` 作为模板基础（设置 `mode: interactive`），填充各维度的确认结果。

确认后保存文件，使用 `AskUserQuestion` 询问阶段推进：

> header: "下一步"
> 问题: "Phase 2 需求澄清已完成，接下来？"
> 选项：
> - **进入 Phase 3**：开始撰写PRD文档
> - **继续澄清**：还有需要补充确认的内容
> - **切换自主模式**：基于已有信息自主生成PRD

如有非阻塞的待定事项，可先进入Phase 3，将待定项在PRD中标注。
如有阻塞性待定事项，建议用户先去确认后再继续。
