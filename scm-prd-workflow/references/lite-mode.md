# 轻量模式 — 操作指引

## 概述

轻量模式是PRD工作台的第三种工作模式，专为简单需求、常规改动和团队已有充分共识的场景设计。核心理念是"快进快出"——聚焦"改了什么"。

区别于自主模式（10章PRD + intake.md、clarification.md、review-report.md、PRD.md），轻量模式只产出7章PRD和可选Mermaid图。

---

## 适用场景判断

### 适合轻量模式

- 改个字段默认值、加个筛选条件、调整审批节点等小改动
- 团队（产品/研发/测试）对业务背景已有充分共识
- 不需要严格描述边界、性能、异常处理等
- 需求变更影响范围可控（通常≤5个变更点）

### 不适合轻量模式（应使用自主/交互模式）

- 新增完整业务流程或子系统
- 涉及跨系统接口变更
- 需要数据迁移
- 业务规则复杂、团队对背景缺乏共识
- 加权复杂度总分≥8（详见"复杂度升级机制"）

## 模式确认门控

### 触发时机

Skill被触发后、正式进入任何PRD阶段之前，**统一执行一次模式确认**。AI根据用户输入智能推荐模式，用户选择确认后再正式进入对应流程。

### 推荐逻辑

| 用户输入特征 | 推荐模式 |
|------------|---------|
| 含"轻量""快速""简单""小改动""常规改动"等关键词 | 轻量模式 |
| 含"详细""逐步""交互""一步步"等关键词 | 交互模式 |
| 无特殊关键词 / 直接描述需求 | 自主模式 |

### 模式确认交互（MC-01）

使用 `AskUserQuestion`：

> header: "PRD模式"
> 问题: "请确认PRD生产模式："
> 选项：
> - **{推荐模式}（推荐）**：{推荐模式描述}
> - **{其他模式1}**：{描述}
> - **{其他模式2}**：{描述}

三个模式的描述（含场景示例）：
- **轻量模式**：7章精简PRD，2-5轮对话（约10-15分钟），适合简单需求/常规改动。例如：改字段默认值、加筛选条件、调整审批节点
- **自主模式**：10章完整PRD，5-9轮对话（约30-60分钟），AI自主生成后集中审阅。例如：新增审批流程、增加库存预警功能、新增报表模块
- **交互模式**：10章完整PRD，18-33轮对话（约2-4小时），逐步讨论每个细节，适合关键/复杂需求。例如：设计全新退货流程、重构跨系统结算体系

### 确认后的行为

用户选择后，AI输出一行确认语，然后进入对应流程：

- 轻量模式：
  > 已进入**轻量模式**。请描述你的需求（改什么、改成什么样），我会直接生成精简版PRD。

- 自主模式：
  > 已进入**自主模式**。请描述你的需求背景，我会在收集关键信息后一次性输出完整PRD供你审阅修改。

- 交互模式：
  > 已进入**交互模式**。请简要描述你的需求，我们将逐步讨论每个细节。

### 模式确认只做一次

- Skill触发时执行一次模式确认
- 之后用户可随时通过关键词主动切换，不再重复确认

## Stage L1: 快速录入

**进度播报**：进入本阶段时输出 `▰▱▱ [1/{total}] 快速录入`。

### 目标

用最少的对话（0-1轮）获取足够生成精简PRD的信息。

### 硬性条件（仅2项）

| # | 条件 | 说明 |
|---|------|------|
| 1 | **改什么** | 哪个系统/模块、具体改什么行为/功能 |
| 2 | **改成什么样** | 期望的目标效果 |

对比自主模式的4硬+5软条件，轻量模式只关注最核心的2项。

### 操作流程

1. **分析用户输入**：判断2项硬性条件是否满足
2. **读取知识库**：如存在 `knowledge-base/`，读取 `_index.md` 和相关知识卡片补充上下文（策略同自主模式）
3. **推导变更范围**（仅 `requirement_type = update|mixed`）：从"改什么+改成什么样"自动推导 `change_scope`，不额外追问（保持零追问原则）。推导结果直接嵌入PRD第2章和第3章。例如：用户说"订单列表加一个筛选条件" → `change_scope: [ui_interaction]`，其余为 `unchanged_scope`
4. **条件判断**：
   - 2项均满足 → **跳过追问，直接进入 Stage L2**
   - 有缺失 → **一轮追问**（自由文本，不使用选项式交互）
   - 追问后仍无法判断"改什么" → 建议切换到自主模式
4. **不产出独立 intake.md** — 录入信息直接融入PRD第2章

### 追问模板

当信息不足时，一轮简洁追问：

> 请补充以下信息，我好直接生成精简版PRD：
>
> 1. **改什么**：哪个系统/模块的哪个功能需要变更？
> 2. **改成什么样**：期望变更后的效果是什么？

## Stage L2: 直接生成

**进度播报**：进入本阶段时输出 `▰▰▱ [2/{total}] PRD 生成`。CD-01 确认后如原型选项变化导致 total 变化，**重新输出路线图**。

### 目标

一次性生成7章精简PRD，全程无用户交互。

### 章节详略选择（CD-01 简化版）

轻量模式同样需要让用户主动选择章节精细度，但交互更轻量。AI 先快速分析需求涉及的内容维度，然后使用 `AskUserQuestion`（multiSelect）：

> header: "章节详略选择"
> 问题: "请选择需要**重点展开**的章节（可多选）。未选中的章节精简描述。"
> multiSelect:
> - **功能说明（第4章）**：每个变更点的完整规则 + 异常处理
> - **流程与交互（第5章）**：完整流程图 + 步骤说明
> - **原型演示（PT-01）**：为关键交互生成可点击原型（AI 建议：{是/否}，理由：{触发条件}）

选择结果处理：选中→充分展开，未选中→精简描述或省略（第5章无流程变更时可省略）。原型演示选中→Stage L3 审阅通过后生成 1 屏意图原型。全部不选→所有章节按精简方式撰写。

### 生成流程

1. **CD-01 章节详略选择**（见上方）
2. 读取本文件（`references/lite-mode.md`）确认轻量模式规则
3. 读取 `references/diagram-patterns.md`（如涉及流程变更）
4. 使用 `templates/lite-prd-template.md` 生成 `PRD-{名称}.md`，按 CD-01 选择结果控制各章节精细度
5. 如有流程变更，生成 Mermaid 图（保存到 `diagrams/` 目录）
6. 执行内置自检（CK-L1 ~ CK-L5），发现问题自动修正
7. 进入 Stage L3 呈现给用户

### 章节编号

省略第5章（流程与交互）时，后续章节编号前移（原第6章→第5章，原第7章→第6章），保持从第1章连续编号。撰写时直接使用最终编号。

### 7章PRD结构

| 章节 | 内容 | 对应完整PRD章节 |
|------|------|---------------|
| 第1章 文档信息 | 版本、作者、日期、需求编号 + `mode: lite` 标识 | Ch 1 |
| 第2章 需求概述 | 背景+动机（1段）、目标（1-3条）、范围（列表） | Ch 2 需求概述 |
| 第3章 业务变更点 | AS-IS vs TO-BE 对比表（**核心章节**） | Ch 4 业务变更总览 |
| 第4章 功能说明 | 每个变更点的行为描述 + 业务规则 + **数据变更**（如涉及，简化版） | Ch 6 + Ch 7§7.3 简化合并 |
| 第5章 流程与交互 | Mermaid 流程图/状态图（**可选，仅流程变更时**） | Ch 5 业务流程 简化 |
| 第6章 验收标准 | 关键场景的验收条件 | Ch 9 验收标准 简化 |
| 第7章 备注 | 待确认项汇总（如有）、参考资料 | Ch 14 + Ch 15 合并 |

**省略的章节**（轻量模式不生成）：
- Ch 3 术语定义 — 常规改动不引入新术语
- Ch 7 接口与数据集成 — 如需新接口或数据迁移，说明需求不够"轻量"
- Ch 8 非功能性需求 — 沿用现有系统标准
- Ch 10 待定事项与附录 — 轻量版备注章节已涵盖核心内容

### 撰写原则

沿用自主/交互模式的通用撰写原则（具体优于抽象、穷举优于概括、变更优于全量），但有以下特别调整：

1. **简洁为先**：每个章节尽量精练，避免冗余描述
2. **变更聚焦**：只描述"改了什么"，不展开未变更的现有逻辑
3. **未提及即不写**：用户未提及的方面不生成独立章节或独立描述。由PRD尾部一段话统一说明未涉及的方面（如"本次需求聚焦于XX，以下方面沿用现有系统行为：YY、ZZ"），避免散落各处的空壳声明破坏叙事连续性

> **与叙事驱动的关系**：轻量模式天然符合叙事驱动理念——7 章精简结构、无全局 ID 开销（不使用 BR/E/DL 编号）、只讲该讲的。通过 CD-01 简化版让用户主动选择重点章节，AI 无需显式输出完整叙事规划，但内部应完成"这份 PRD 要重点讲什么"的思考。

### 标记规则

| 标记 | 使用场景 |
|------|---------|
| （无标记） | 默认无标记，AI推断直接写入 |
| `[待确认]` | **仅**用于存在多种合理方案、必须由用户决定的业务歧义（Word 输出时自动渲染为黄色提示块） |

**`[待确认]` 上限：3个**。超过3个触发复杂度升级机制。

`[待确认]` 在轻量模式下的格式（简化版，不需要"如需变更"说明）：

```markdown
> [待确认] {问题描述}
> 当前假设：{AI的假设}
```

### 原型策略（轻量模式例外）

轻量模式通常不触发原型生成——简单字段/配置改动的变更效果从变更点表即可理解。

**例外情况**：当轻量 PRD 的第4章"功能说明"涉及明显的 UI 交互变更（命中 T1/T3/T5 触发条件），且用户在 CD-01 简化版中主动选择"原型演示"时，可生成 1 个屏幕的意图原型。此时：
- 仅生成 1 个屏幕（不超过）
- 精细度固定为"意图原型"
- PRD 审阅（Stage L3）通过后，读取 `references/prototype-planning.md` 进入原型输出流程
- 产出 `prototype-brief.md` + `prototype/bundle.html`

如果需要多屏幕原型，说明需求复杂度已超出轻量模式范围，应先升级到自主模式。

### 内置自检（CK-L系列）

生成PRD后自动执行，不输出独立的review-report.md。发现问题先自动修正，无法自动修正的在交付时附带说明。

| 检查项 | 内容 | 严重度 |
|--------|------|--------|
| CK-L1 完整性 | 7章均存在（第5章可选除外）；第3章变更点表的每一行在第4章有对应描述；第6章验收标准覆盖所有变更点 | 关键 |
| CK-L2 一致性 | 术语前后一致；第5章流程图（如有）与第4章文字描述匹配 | 警告 |
| CK-L3 可测试性 | 每个变更点至少1条验收标准；无模糊用语（"适当""灵活""等"） | 警告 |
| CK-L4 规则无歧义 | 每条业务规则是否有且仅有一种解读（无"或""可以"等歧义表述） | 关键 |
| CK-L5 变更-验收对应 | 第3章每个变更点在第6章是否有对应的验收标准（一一对应检查） | 关键 |

## Stage L3: 快速审阅

**进度播报**：进入本阶段时输出 `▰▰▰ [3/{total}] 审阅`（无原型时为"审阅交付"）。

### 目标

简单反馈循环——用户确认或要求修改，最多2轮迭代。

### 呈现方式

1. 输出完整PRD
2. 如有 `[待确认]` 项（≤3个），在PRD前置摘要提示
3. 使用 `AskUserQuestion` 收集反馈（SL-01）：

> header: "轻量PRD审阅"
> 问题: "精简版PRD已生成，请审阅。"
> 选项：
> - **没问题，交付**：PRD内容确认，准备最终交付
> - **需要小改**：有少量修改意见（进入自由文本）
> - **需要补充细节**：某些部分需要更详细的描述，升级到自主模式扩展

### 迭代规则

- 用户选择"需要小改" → 修改后重新呈现，再次使用 SL-01 收集反馈
- **最多2轮迭代**。2轮后仍有较多修改 → 建议升级到自主模式或交互模式
- 每轮修改后重新执行 CK-L1 ~ CK-L5 自检

### 最终交付

用户选择"没问题，交付"后：

1. 移除所有 `[待确认]` 标记（如用户已在反馈中确认）
2. 更新版本号
3. 输出最终文件列表
4. **不主动提供Word输出选项**（用户可自行要求）

## 复杂度升级机制

### 复杂度评估方式

轻量模式采用**加权复杂度评分**（而非简单计数）判断是否需要升级。每个变更点的复杂度 = 基础分 × 影响系统数 × 规则复杂度因子：

| 因素 | 权重说明 |
|------|---------|
| 基础分 | 每个变更点 1 分 |
| 影响系统数 | ×1（单系统）、×2（2个系统）、×3（3+系统） |
| 规则复杂度 | ×1（无特殊规则）、×2（含审批链/状态机/计算公式/新增实体或关系变更） |

**示例**：
- "修改字段默认值"：1×1×1 = 1分
- "新增跨系统审批流程"：1×2×2 = 4分

**升级阈值**：满足以下任一条件时建议升级：
- 加权复杂度总分 ≥ 8
- **单项复杂度 ≥ 4**（短路规则：即使总分较低，单个高复杂度变更点也应升级）

### 触发条件

当AI在L2生成过程中或L3审阅过程中检测到以下信号时，暂停并建议升级：

| 触发条件 | 检测时机 | 建议方向 |
|---------|---------|---------|
| `[待确认]` 超过3个 | Stage L2 生成中 | → 自主模式 |
| 加权复杂度总分 ≥ 8 | Stage L2 生成中 | → 自主模式 |
| 需要跨系统接口变更 | Stage L2 分析时 | → 自主/交互模式 |
| 需要数据迁移 | Stage L2 分析时 | → 自主模式 |
| 新增实体或实体关系变更 | Stage L2 分析时 | → 自主模式（需完整ER图） |
| 用户连续2轮审阅仍有大量修改 | Stage L3 审阅中 | → 交互模式 |

### 升级交互（SL-02）

使用 `AskUserQuestion`：

> header: "复杂度提示"
> 问题: "这个需求的复杂度可能超出轻量模式的覆盖范围。"
> markdown: |
>   **评分明细：**
>   | 变更点 | 基础分 | 影响系统数 | 规则复杂度 | 小计 |
>   |--------|--------|-----------|-----------|------|
>   | {变更点1} | 1 | ×{N} | ×{M} | {分} |
>   | ... | ... | ... | ... | ... |
>   | **合计** | | | | **{总分}** |
>
>   **触发原因**：{总分≥8 / 单项≥4 / 跨系统接口 / 数据迁移}
> 选项：
> - **切换自主模式**：基于已有信息生成完整10章PRD
> - **继续轻量模式**：我理解限制，继续出精简版
> - **切换交互模式**：逐步讨论每个细节

### 升级时的状态保留

已生成的轻量PRD内容作为完整PRD的种子，映射关系：

| 轻量章节 | 完整PRD章节 | 升级处理方式 |
|---------|-----------|------------|
| 第2章 需求概述 | → Ch 2 需求概述 | **直接复用**，补充§2.2现状详述和§2.6合理性分析 |
| 第3章 业务变更点 | → Ch 4 业务变更总览 | **直接复用**，AS-IS列简化为摘要回指Ch.2 |
| 第4章 功能说明 | → Ch 6 功能与规则明细 | **结构重写**：轻量版"业务规则列表"需重构为"统一步骤-规则表"格式（规则和异常使用局部序号） |
| 第5章 流程与交互 | → Ch 5 业务流程 | **直接复用**，可选升级为泳道图 |
| 第6章 验收标准 | → Ch 9 验收标准 | **直接复用**，增加G-XXX和F-XXX关联列 |

轻量PRD中未覆盖的章节（Ch 3/7/8/10）由自主模式补充生成。

**升级提示**：SL-02 升级交互中告知用户："第4章功能说明将重构为统一步骤-规则表格式，其余章节在已有基础上扩展。"

## 模式切换

### 进入轻量模式

| 来源 | 触发方式 | 行为 |
|------|---------|------|
| 模式确认门控 | 用户选择轻量模式 | 输出确认语，进入 Stage L1 |
| 自主模式任意阶段 | 用户说"够了，出精简版就行" | 基于已收集信息生成7章精简PRD |
| 交互模式任意阶段 | 用户说"太细了，出精简版" | 基于已收集信息生成7章精简PRD |

### 从轻量模式切出

| 目标 | 触发方式 | 行为 |
|------|---------|------|
| 自主模式 | 复杂度升级 / 用户说"补充完整版" | 保留内容，扩展为10章 |
| 交互模式 | 用户说"逐步讨论" | 保留内容，进入 Phase 2 |

### 切换时的保留原则

- 所有已收集的信息和已生成的内容均保留
- 不重复询问已确认的信息
- 升级时只补充缺失的章节和维度

└── diagrams/                      # 仅在流程变更时生成
    └── *.mermaid                  # Mermaid 图（仅此格式）

**不产出的文件**（对比自主/交互模式）：
- `intake.md` — 录入信息融入PRD第2章
- `clarification.md` — 无独立澄清阶段
- `review-report.md` — 自检内置执行，不输出报告
- `PRD-{名称}.docx` — 不主动提供Word版本
- `*.diagram.yaml` / `*.drawio` — 不使用YAML→draw.io
- 独立的 §7.2/7.3 数据模型章节 — 数据变更简化嵌入第4章功能说明
- `decision-log.md` — 轻量模式不记录方案决策过程
- `prd-changelog.md` — 轻量模式最多2轮迭代，变更记录不单独持久化

## 与自主模式的对比速查

| 维度 | 自主模式 | 轻量模式 |
|------|---------|---------|
| 用户交互 | 5-9轮（约30-60分钟） | 2-5轮（约10-15分钟） |
| PRD章节 | 10章 | 7章 |
| 产出文件 | intake + clarification + review-report + PRD + 图表 | PRD + 可选图表 |
| 标记体系 | `[推断]` + `[待确认]` | 仅 `[待确认]`（上限3个） |
| 澄清阶段 | AI推断9维度 | 无 |
| 自检 | CK-0 ~ CK-9（60+项） | CK-L1 ~ CK-L5（5项） |
| 复杂度判断 | 4硬+5软条件 | 加权复杂度评分（≥8分升级） |
| 图表格式 | Mermaid + YAML→drawio | 仅 Mermaid |
| Word输出 | 默认提供 | 不提供（可要求） |
| 审阅迭代 | 无限轮 | 最多2轮 |
