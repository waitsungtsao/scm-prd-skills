# 轻量模式 — 操作指引

读取时机：MC-01 选择轻量模式后。

## 核心理念

轻量模式是简单需求的快速通道。**轻的是用户表达和输出形式，不是 AI 对质量的把控**。心智模型与自主模式一致——鉴别诊断、叙事驱动、质疑精神一样适用，只是方案大概率不复杂、需要确认的点少。

产出：3 章叙事 PRD（背景/系统需求/注意事项），支持单需求和批量需求。

---

## 适用场景

**适合**：改字段默认值、加筛选条件、调整审批节点、修改规则参数等。团队对背景有共识，不需要从零建立认知。

**不适合**（应使用自主模式）：新增完整流程/子系统、跨系统接口变更、数据迁移、业务规则复杂。加权复杂度 ≥12 时触发升级。

---

## Stage L1: 快速录入

**进度播报**：`▰▱▱ [1/{total}] 快速录入`

### 目标

用最少的对话（0-1 轮）获取足够生成 PRD 的信息。

### 操作流程

1. **分析用户输入**：判断"改什么"和"改成什么样"是否明确
2. **读取知识库**（如存在）：补充上下文
3. **快速鉴别诊断**（内部）：形成假设，识别有没有关键鉴别点
4. **条件判断**：
   - 信息充分 → **直接进入 Stage L2**
   - 有关键缺失 → **一轮追问**（自由文本）
   - 追问后仍不清晰 → 建议切换自主模式
5. 不产出独立 intake.md

### 批量模式

用户一次性提供多个需求草稿时（列表/会议纪要/多段描述），AI 解析为 N 个独立需求，对每个执行快速鉴别诊断。如有需要补充的信息，**所有需求的问题合并为一次追问**（按需求分组）。

---

## Stage L2: 直接生成

**进度播报**：`▰▰▱ [2/{total}] PRD 生成`

### 目标

一次性生成 3 章 PRD，全程无用户交互。

### 三章结构

#### 第1章：背景

一段话回答"为什么做"：痛点→目标→上下文。不需要 G-XX 编号，不需要范围声明。

#### 第2章：系统需求

用融合叙事讲完所有变更点。读完后开发知道要做什么。

**允许的结构化元素**（服务叙事，不是强制格式）：
- 分段（按逻辑段落自然分）
- 列表（规则、字段变更等适合枚举的内容）
- 小标题（涉及多个相关但独立的变更时）
- 简单表格（字段变更对比、状态转换等）

**不需要的**：F-XXX 编号、步骤-规则表、独立的异常处理章节。异常处理自然融入叙事（"如果库存不足，则..."）。

#### 第3章：注意事项

待确认项、已知风险、依赖、回归范围。**没有则不写此章**。

`[待确认]` 格式：
```markdown
> [待确认] {问题描述}
> 当前假设：{AI的假设}
```

### 生成流程

1. AI 内部完成叙事思考（不输出叙事规划，但遵循同样的原则）
2. 使用 `templates/lite-prd-template.md` 生成 PRD
3. 如用户要求画图或 AI 判断非常必要 → 生成 Mermaid 图
4. 运行 `check-prd-consistency.py`（模糊用语检测）
5. AI 快速通读确认叙事连贯性（不生成 review-report）
6. 进入 Stage L3

### 批量输出结构

```markdown
---
title: "需求批次 — {日期} {主题}"
mode: lite-batch
count: {N}
date: YYYY-MM-DD
---

# 需求批次 — {日期} {主题}

## 1. {需求简称}

### 背景
{一段话}

### 系统需求
{融合叙事}

### 注意事项
{如有}

---

## 2. {需求简称}
...
```

### 标记规则

| 标记 | 使用场景 |
|------|---------|
| （无标记） | 默认，AI 推断直接写入 |
| `[待确认]` | 存在多种合理方案、必须由用户决定的业务歧义 |

不使用 `[推断]`。每个 `[待确认]` 计入加权复杂度 +2 分。

### 图表策略

- 默认不画图
- 用户要求 → 画（Mermaid 优先，复杂度超出可用 YAML → draw.io）
- AI 判断非常必要（如涉及状态流转、多步分支流程）→ 画
- 图表保存到 `diagrams/`

### 自检

| 层 | 执行者 | 内容 |
|---|--------|------|
| 硬逻辑 | `check-prd-consistency.py` | 模糊用语检测 |
| 软逻辑 | AI 快速通读 | 背景→系统需求因果成立？注意事项覆盖风险点？ |

不生成独立 review-report.md。发现问题先自动修正，无法修正的标注在注意事项中。

## Stage L3: 快速审阅

**进度播报**：`▰▰▰ [3/{total}] 审阅交付`

### 目标

用户审阅反馈，最多 2 轮大改迭代。

### 呈现方式

1. 输出完整 PRD（批量模式则一次性输出全部需求）
2. 如有 `[待确认]`，前置摘要提示
3. 邀请自由文本反馈：

> "PRD 已生成，请审阅。直接回复修改意见即可，未提到的内容视为确认。"

**不使用 AskUserQuestion**——与自主模式 Stage C 一致的批量反馈模式。

### 迭代规则

- 最多 2 轮大改（≥4 处实质性变更）。2 轮后仍有大量修改 → 建议升级
- 小改（≤3 处文案/格式调整）不计入迭代上限
- 每轮修改后重新执行自检

### 交付

1. 移除已确认的 `[待确认]` 标记
2. 生成 Word（`.docx`）— 批量模式注意标题层级：H1=文档标题、H2=每个需求、H3=章节
3. 输出最终文件列表
4. 更新约束索引（简化版：仅 `prd_id`、`version`、`scope_in`）
5. 如有知识发现 → 提示导入 curator

### 产出文件

单需求：`PRD-{名称}.md` + `.docx` + 可选 `diagrams/*.mermaid`
批量：`PRD-batch-{日期}-{主题}.md` + `.docx`

不产出：intake.md、clarification.md、review-report.md、decision-log.md、prd-changelog.md

## 复杂度升级机制

### 加权复杂度评分

| 因素 | 权重 |
|------|------|
| 基础分 | 每个变更点 1 分 |
| 影响系统数 | ×1（单系统）、×2（2系统）、×3（3+系统） |
| 规则复杂度 | ×1（普通）、×2（审批链/状态机/计算公式/新实体） |
| `[待确认]` | 每个 +2 分 |

**升级阈值**：总分 ≥12 或 单项 ≥4 或 跨系统接口 或 数据迁移。

### 升级交互（SL-02）

AI 直接说明情况（不使用 AskUserQuestion 的复杂评分表）：

> "这个需求比预想复杂一些——{具体原因，如"涉及跨系统状态同步"/"审批规则有3种可能方案需要确认"}。我先对齐一下理解——{叙事大纲}。需要补充：{问题}。补充后我生成完整版 PRD。"

用户可以选择继续轻量（"不用，就按当前的出"）或同意升级。

### 升级时的映射（3→10 章）

| 轻量 | 自主模式 | AI 处理 |
|------|---------|--------|
| 背景 | → Ch.2 需求概述 | 扩展为 §2.1~§2.6 |
| 系统需求 | → Ch.5+Ch.6 | 拆分为流程+功能点，补规则和异常 |
| 注意事项 | → Ch.10 + 分散到各章 | 待确认项分发到对应章节 |
| （无） | Ch.1/3/4/7/8/9 | AI 补充生成 |

AI 静默完成转换，用户看到的是完整版 PRD。不向用户暴露"升级"或"重构"过程。
