# 方案输出 — PRD撰写指引

## 目标

基于Stage A的完整需求信息，产出**重点清晰、逻辑连贯、可直接用于开发评审**的PRD文档。PRD是讲故事，不是填模板——先想清楚"这份文档要怎么讲"，再动笔。

---

## 叙事规划（撰写前的产品经理思考）

### 为什么需要叙事规划

PRD的质量不取决于章节是否齐全，而取决于读者能否快速理解"改了什么、为什么改、怎么改"。直接按模板逐章填写会导致：
- 每章都写满，信号被噪音稀释
- 模块之间靠标号关联而非逻辑连接
- 行业标准做法重复描述，本需求特有的重点反而不突出

叙事规划是撰写前的**全局思考**——像产品经理一样，先想清楚 PRD 的目标、逻辑主线、重点，再决定写什么、不写什么、重点展开什么。

### 叙事规划的产出

```markdown
## 叙事规划

### 核心叙事（一段话）
{什么问题} → {怎么解决} → {为什么这样做} → {预期价值}

### 方案评估

在确定核心叙事后、展开文档结构前，退后一步审视方案本身。这是 AI 构思整个故事的关键时刻——不是先写完再检查，而是在构思时就确保故事逻辑完整。

**因果链自检**：从 intake 痛点到当前方案，因果链是否每一步都成立？
- {痛点} → 导致 → {问题} → 所以需要 → {方案} → 预期达到 → {效果}
- 如果任何一步逻辑跳跃，**修改方案或完善表达使逻辑完整**——不是打补丁加解释段落

**方案合理性**：
- 这是不是当下最合理的方案？有没有更简单的替代方案？
- 如果有更简单的方案但选择了当前方案，这个选择的逻辑是什么？（体现在核心叙事中，不是单独解释）
- 方案的复杂度与问题的严重程度是否匹配？

**成本与权衡**：
- 这个方案要牺牲什么？（开发成本/系统复杂度/用户学习成本）
- 这些牺牲是否值得？
- intake 中用户提到的担忧/约束/历史包袱，方案是否已经回应？

**执行方式**：
- 方案评估的结果**融入核心叙事和维度选择**中——如果评估导致方案调整，核心叙事直接反映调整后的版本
- AI 能自行调整的（如发现可以简化方案）：直接修改，用户看到的是评估后的版本
- 需要用户输入的（如发现信息缺失或矛盾）：暂停，向用户呈现具体问题后继续

### 已知背景推断

**目的**：识别"所有目标读者都已经知道的"，PRD 中引用但不展开——减少补偿性冗余。

**注意**：PRD 的读者不只是撰写者，还包括开发、QA、架构师、BA、决策者。"已知"是指**所有目标读者都知道**，不是作者知道就省略。如果作者知道但开发不知道，PRD 仍需描述。

**执行方式**：AI 基于 knowledge-base、system-conventions、intake 内容，**主动推断**一组已知背景假设，写入叙事规划。不向用户提问"你知道什么"（知识的诅咒——描述已知且非核心的事情对人类很痛苦），而是列出推断让用户确认。

```markdown
### 已知背景（AI 推断，`[推断]` 标记）
以下是 AI 判断为团队共识、PRD 中引用但不展开的内容：
- {系统现有的消息队列架构} — 来源：knowledge-base/domain-oms.md
- {字段命名规范} — 来源：system-conventions.md §2.3
- {当前退货流程的 AS-IS 状态} — 来源：intake.md 用户描述
如有需要在 PRD 中特别展开的项，请在审阅时指出。
```

**各模式差异**：
- 自主模式：写入 intake.md 的 `known_context` 字段，Stage C 审阅时确认
- 轻量模式：不显式处理（轻量本身就是"团队有共识"的场景）

### 表达维度与注意力预算

表达维度由两个独立维度交叉决定：**叙事主次**（这个维度在故事中的角色）和**读者认知**（目标读者对这个维度的了解程度）。

**维度 A — 叙事主次**（从故事逻辑来）：

| 叙事角色 | 处理方式 |
|---------|---------|
| 核心变更点 — 故事的主线 | 重点展开 |
| 支撑核心的上下文 — 让主线成立的前提 | 标准描述 |
| 非核心但必要的 — 术语、NFR 等 | 简述 |
| 与故事无关的 | 省略 |

**维度 B — 读者认知**（从 Theory of Mind 来）：

| 读者认知判断 | 处理方式 |
|------------|---------|
| 所有目标读者都不知道 | 需要在 PRD 中建立认知 |
| 部分读者知道（如开发知道但 BA 不知道） | 简明描述，不过度展开技术细节 |
| 行业共识（所有供应链从业者都知道） | 一句话带过，不展开原理 |
| 团队已知（`已知背景` 中确认的项） | 引用不解释 |
| 影响小且读者无需关心 | 不写 |

**交叉决策表**：

| | 读者不了解 | 部分读者了解 | 团队共识/行业共识 |
|---|---|---|---|
| **叙事核心** | ★★★ 重点展开 | ★★ 标准描述 | ★ 简述（点到即可） |
| **支撑上下文** | ★★ 标准描述 | ★ 简述 | 引用不解释 |
| **非核心** | ★ 简述 | 引用或省略 | 省略 |

**叙事规划模板**：

| 维度 | 叙事角色 | 读者认知 | 权重 | 理由 |
|------|---------|---------|------|------|
| {审批流程} | 核心变更 | 部分读者了解 | ★★ | {本次需求核心，BA 熟悉但开发需要了解规则} |
| {字段默认值} | 非核心 | 团队共识 | — | {改动明确，无歧义，所有人都知道} |

### 快速线框（可选，命中 ≥3 条触发条件时）

当原型策略命中 **≥3 条** T1-T5 触发条件时，在叙事规划中追加一个"快速线框"段落——不是完整原型（那在 PRD 确认后才做），而是 1-2 个关键屏幕的**文字线框描述**，帮助在 PRD 撰写前验证交互假设。

**快速线框格式**（纯文字，不需要代码/HTML）：

```markdown
### 快速线框 — {屏幕名称}

**布局**：
- 顶部：{面包屑 / 页面标题 / 操作按钮}
- 左侧：{导航 / 筛选面板 / 无}
- 主区域：{列表 / 表单 / 仪表盘 / 详情}
- 底部：{分页 / 操作栏 / 无}

**关键交互**：
1. 用户点击 {按钮} → 系统 {行为} → 显示 {结果}
2. 当 {条件} 时 → {状态变化}

**核心字段**（仅列变更/新增字段）：
| 字段 | 类型 | 来源 |
|------|------|------|
| ... | ... | F-XXX |

**不做什么**：{明确排除的交互/区域}
```

**作用**：
- 用户在 CD-01 阶段看到快速线框后，可以提前发现交互假设偏差（如"这个不应该是列表页，应该是看板视图"）
- 快速线框中发现的问题反馈到叙事规划，避免 PRD 写完才发现方向错误
- 如果用户在 CD-01 中选择了"原型演示"，快速线框将作为后续 prototype-design.md 的基础

**不触发快速线框的情况**：命中 <3 条触发条件、用户在 CD-01 中取消了原型选项、轻量模式。

### 叙事规划反思（NP-01 ~ NP-05）

叙事规划草稿完成后、呈现给用户（CD-01）前，**静默完成以下反思**。发现问题则修正后再输出——用户只看到最终版本，不感知反思过程。

**三个核心问题**（叙事反思的 guideline）：

1. **这份 PRD 的核心故事是什么？** — 不是"做了什么功能"，而是"解决了谁的什么问题、为什么用这个方案"。如果用一句话说不清楚，说明还没想透。
2. **哪些章节承载主线、哪些是支撑？** — 主线章节应该浓墨重彩，支撑章节精简到位，不相关的维度果断省略。如果所有章节同等篇幅，说明没有表达策略。
3. **读者读完后能记住什么？** — 每个目标读者（开发、QA、BA、决策者）读完后应该能复述 2-3 个要点。如果想让人记住所有东西，等于什么都记不住。

**好的叙事规划会自然满足以下检查**（NP-01~05 作为验证性自检，而非目的）：

| ID | 反模式 | 如何识别 | 说明 |
|----|--------|---------|------|
| NP-01 | 泛化叙事 | 核心叙事含"提升效率""优化体验"等万能句 | 核心问题 1 没想透 |
| NP-02 | 平铺罗列 | 所有维度标为同一详略级别 | 核心问题 2 没想透 |
| NP-03 | 图表堆砌 | 图表数 > 功能点数，或图无表达意图 | 图是为叙事服务的，不是装饰 |
| NP-04 | 脱离 intake | 提及了 intake 范围外的模块/功能 | 超范围内容删除或标注 |
| NP-05 | 无主次 | 功能点清单无核心/复杂标注 | 核心问题 3 没想透 |

### 撰写前结构预校验（NP-06 ~ NP-08）

NP-01~05 聚焦叙事质量，NP-06~08 聚焦**结构完整性**——确保即将撰写的 PRD 在 ID 覆盖、章节映射、术语一致性上不会产生事后才发现的问题。

| ID | 检查目标 | 检查方法 | 修正方式 |
|----|---------|---------|---------|
| NP-06 | **ID 覆盖完整** | 叙事规划中的功能点清单是否覆盖 intake.md 中的每个痛点/目标（G-XX）；update/mixed 类型下每个变更点（C-XX）是否有对应功能 | 列出未覆盖的 intake 项，补充到功能点清单或标注为"不在本次范围（原因：{原因}）" |
| NP-07 | **章节映射无遗漏** | CD-01 选中的重点章节是否覆盖所有核心功能点；标为"省略"的维度是否确实不含核心功能 | 发现核心功能被分配到"省略"维度时，建议将该维度升级为"标准"或"重点" |
| NP-08 | **术语预检** | 叙事规划中使用的系统名称、角色名、状态名是否与 `glossary.yaml`（如存在）一致；是否引入了 glossary 中未定义的新术语 | 不一致的术语标注高亮，新术语记录到待补充列表（交付时写入 knowledge-discoveries.md） |

**NP-06~08 仅在以下条件满足时执行**：
- NP-06：始终执行（ID 覆盖是核心质量保证）
- NP-07：CD-01 完成后执行（需要知道用户的章节选择）
- NP-08：`glossary.yaml` 存在时执行（不存在则跳过）

### 方案评估验证（NP-09 ~ NP-11）

NP-09~11 验证"方案评估"步骤是否被认真执行（防止 AI 跳过直接进入文档结构规划）。如不通过，**回到方案评估步骤重新思考**，而不是在叙事规划文档上打补丁。

| ID | 验证什么 | 不通过的信号 | 修正方式 |
|----|---------|-----------|---------|
| NP-09 | **因果链完整** | 核心叙事中存在跳跃——痛点和方案之间缺少"为什么选这个方案"的逻辑 | 回到方案评估，重新构建因果链，修改核心叙事 |
| NP-10 | **方案已被评估** | 叙事规划没有体现方案选择的逻辑——看起来像直接把 intake 需求翻译成了功能清单，读者无法追溯"为什么是这个方案" | 回到方案评估，思考方案合理性后调整叙事 |
| NP-11 | **约束已被消化** | intake 中有用户提到的担忧/约束，但叙事规划中完全没有回应 | 混合：信息遗漏→暂停询问用户；信息已有但叙事未体现→AI 自行调整 |

**执行规则**（NP-01~11 统一）：
- 逐条检查，命中则立即修正
- 最多自修正 **2 轮**（防止无限循环）。2 轮后仍有未满足项，带当前版本进入 CD-01
- NP-01~05 在 CD-01 前执行；NP-06 在 CD-01 前执行；NP-07~08 在 CD-01 后、撰写前执行
- NP-09~11 与 NP-01~05 同时执行（方案评估完成后、CD-01 前）
- 全部通过后，开始逐章撰写

### 叙事规划确认 + 重点标注（CD-01）

叙事规划输出后，通过 CD-01 让用户标注 **2-3 个重点章节**。其余章节由 AI 根据需求复杂度自动判断详略。目的是让用户表达"我最关心什么"，而非逐章分配详略。

**各模式的执行方式**：

| 模式 | 时机 | 交互方式 |
|------|------|---------|
| **自主模式** | Stage B 步骤 3，PRD 大纲确认时 | 详见 `autonomous-mode.md` 的"PRD 大纲确认" |
| **轻量模式** | Stage L2 生成前 | 简化版 CD-01（详见 `lite-mode.md`） |

**CD-01 交互**：

AI 先输出叙事规划（含核心叙事、AI 建议的重点、叙事反思结论），然后使用 `AskUserQuestion`（multiSelect）：

> header: "重点标注"
> markdown: （展示叙事规划 + AI 建议的重点章节及理由）
> 问题: "我建议重点展开以下章节（已预选），你可以增删。未标注的章节我会根据需求信息自动判断详略。"
> multiSelect（AI 预选 2-3 项）:
> - **业务流程（Ch.5）**：完整流程图 + 步骤详解
> - **功能与规则（Ch.6）**：步骤-规则表逐项展开，含完整异常处理
> - **接口与数据（Ch.7）**：接口规格 + 字段级定义
> - **非功能性需求（Ch.8）**：性能/安全/可用性量化描述
> - **原型演示（PT-01）**：为关键交互生成可点击原型（AI 建议：{是/否}，理由：{触发条件}）

**选择结果的处理**：

| 用户选择 | 章节处理 |
|---------|---------|
| **选中** | 按"重点"详略撰写，充分展开 |
| **未选中** | AI 自动判断：与需求相关→标准/精简，无关→省略（PRD 尾部说明） |
| **全部不选** | 所有章节按"标准"详略撰写 |
| **原型演示 选中** | PRD 审阅确认后进入原型输出阶段 |

AI 根据用户选择更新叙事规划的"表达维度与详略"表格，据此指导后续撰写。

## 写作总原则

### 叙事优先

**文档层面**：
- 每个模块服务于全局叙事，不为写而写
- 行业标准做法、系统默认行为**不展开描述**——只写本需求特有的内容
- 模块之间用逻辑连接（"因为A，所以需要B"），而非仅靠标号关联
- 规整的格式和表格不等于好表达，符合人类认知的流畅叙述才是目标

**章节层面**：
- 每个章节在动笔前问自己：这章回答读者什么问题？如果答不出来，可能不需要写
- 叙事规划中标为"省略"的维度不生成独立章节，由 PRD 尾部一句话统一说明

### 信息分层

PRD 是跨角色的沟通工具——撰写者、开发、QA、架构师、BA、决策者都是目标读者。信息按读者需求分层：
- **概览层**（背景/目标/范围）：给决策者和不熟悉业务的人看
- **方案层**（流程/变更/功能）：给开发、测试、业务方看
- **细节层**（字段/接口/规则）：给开发和测试看

### 注意力 ROI 自检（分层执行）

撰写过程中，每完成一个层级即自检该层级的信息 ROI。从宏观到微观：

**章节级**：这章是否服务叙事主线？删掉整章后，读者对"问题→方案→验证"的理解会断裂吗？
- 不会断裂 → 省略或合并到其他章
- 会断裂 → 保留，但检查权重是否匹配（核心章节篇幅应 > 非核心）

**段落级**：这段是否传递了本章其他段落没有的新信息？
- 是在重述知识库/系统公约/已知背景中的内容 → 改为引用（"沿用现有 X 机制"）
- 是行业共识所有供应链从业者都知道 → 一句话带过或删除
- 是本需求特有的新信息 → 保留

**句子级**：这句是否有信息增量？删掉后上下文是否仍然连贯完整？
- 属于冗余修饰（"本章将详细描述""下面我们来看"）→ 删除
- 属于过度解释（规则已清晰但仍附加冗余示例）→ 删除
- 属于补偿性内容（AI 不确定读者是否知道所以写的"安全网"描述）→ 对照已知背景判断，已知则删除

**关注点级**：各关注点之间的逻辑关系是否清晰？
- 相邻段落是否"A 说了但 B 没衔接" → 补充过渡逻辑
- 不同章节是否"A 和 B 重复说了同一件事" → 保留信息密度更高的那个，删另一个

### 语言规范

**必须做到**：
- 使用确定性表述："系统应""系统必须"，不用"系统可以""系统尽量"
- 规则量化：不说"及时处理"，说"30分钟内处理"
- 条件穷举：不说"等情况"，逐一列出所有情况
- 主语明确：不说"应处理"，说"WMS系统应处理"

**绝不能出现的表述**：
- "可配置"（配什么？谁配？在哪配？默认值是什么？）
- "灵活处理"（怎么灵活？处理规则是什么？）
- "适当的"/"合理的"（标准是什么？）
- "参考XX系统"（具体参考什么逻辑？）
- "等"字（穷举出来）

### 简洁性规范

**文档级**：
- 叙事规划中标为"省略"的维度不写独立章节
- 叙事规划中标为"精简"的维度只写关键差异，不铺陈完整设计
- 系统公约已涵盖的标准行为不在 PRD 中重复

**段落级**：
- 同义反复：同一信息不得用不同措辞重复表述
- 填充句：删除不传递信息的句子。禁止："本章是PRD的核心""以下将详细描述""众所周知"
- 过度解释：规则本身已清晰时不附加解释
- 上下文重复：已在前文描述过的内容，用章节回指代替重复叙述

**句子级**：
- 动词优于名词化：写"系统分配仓库"，不写"系统执行仓库分配操作"
- 表格优于段落：结构化信息（规则、字段、参数）一律用表格
- 短句优于长句：单句不超过40字
- 删除无信息修饰词："进行""相关的""一定程度上""基本上""总体来说""需要注意的是"

**自检口诀**：每句话问自己——"删掉这句，读者会丢失信息吗？"如果不会，删掉。

### 表注

对表格字段、规则的少量补充说明，使用 `^` 前缀标记为**表注**。表注紧跟表格，与表格主体不留间隔，与下方后续内容留标准间距。

**写法**：

```markdown
| 字段 | 类型 | 说明 |
|------|------|------|
| order_no | String | 订单号 |

^ **数据来源说明**：出库单的车牌号取OMS出库单已有字段。
^ 报关字段的标准码值由关务系统维护。
```

**规则**：
- `^` 后跟一个空格，然后是注释内容
- 支持行内格式（`**加粗**`、`` `代码` ``）
- 表注应紧凑（1-3 行），大段说明应作为正文段落
- Word 输出时表注渲染为 9pt 灰色小字（caption 样式），紧贴表格底部

**何时使用表注 vs 正文**：
- 字段取值范围、数据来源、枚举说明 → 表注
- 响应参数格式补充、异常处理规则 → 表注
- 完整的业务逻辑描述 → 正文段落

### 提示块

在正文中使用提示块高亮关键信息，帮助读者快速识别需要注意的内容。提示块在 Word 输出中渲染为带色条的视觉区块。

**四种类型**：

| 语法 | 用途 | 示例场景 |
|------|------|---------|
| `> [!INFO]` | 补充说明、背景知识 | 业务上下文补充、术语解释、知识库引用 |
| `> [!CAUTION]` | 需要注意但不阻断 | 数据一致性风险、性能影响、向后兼容提醒 |
| `> [!WARNING]` | 严重风险或硬性约束 | 资金安全、数据丢失风险、法规合规要求 |
| `> [!TIP]` | 最佳实践、正向建议 | 推荐做法、行业经验、优化建议 |

**写法示例**：

```markdown
> [!WARNING] 库存扣减必须在同一事务中完成，否则可能出现超卖。
> 相关功能：F-001 步骤3。

> [!INFO] 当前出库流程为"拣货→复核→打包→发运"四步流程。
> （参考: knowledge-base/domain-wms.md）

> [!TIP] 建议在审批节点增加金额分级，5万以下自动通过，减少审批等待。
```

**自动映射**：PRD 标记自动渲染为对应提示块样式，无需额外使用 `[!XXX]` 语法：

| PRD 标记 | 渲染为 | 色条颜色 |
|---------|--------|---------|
| `[待确认]` | CAUTION（注意事项） | 黄色 |
| `[推断]` | INFO（信息提示） | 蓝色 |

**使用原则**：
- 每页不超过 2 个提示块，避免视觉噪音
- 正文中能自然表述的信息不需要用提示块包裹——提示块用于**需要读者停下来注意的内容**

## 按需生成：章节裁剪与详略控制

### 核心原则

好的PRD是一个好的故事——聚焦于需求本身，逻辑闭环，无冗余。**不论 requirement_type 是 new 还是 update**，都应根据叙事规划裁剪章节：

- 叙事规划中标为"省略"的维度 → 不生成独立章节
- 叙事规划中标为"精简"的维度 → 生成但精炼描述
- 叙事规划中标为"重点"的维度 → 充分展开

**requirement_type = new 时的注意事项**：新建类需求同样需要叙事规划——不是所有 10 章都要写满。例如一个新增子模块可能不需要 Ch.8（NFR 沿用标准）和 Ch.3（术语已知），但需要重点展开 Ch.5（全新流程）和 Ch.7（新接口设计）。不生成的章节不需要"尾部汇总"说明（该汇总仅适用于 update/mixed）。

### 章节裁剪决策（requirement_type = update|mixed 时的额外规则）

在叙事规划的基础上，update/mixed 类型还遵循以下裁剪规则：

| 章节 | 涉及变更时 | 不涉及变更时 |
|------|-----------|------------|
| Ch.2 需求概述 | 始终生成 | 始终生成 |
| Ch.3 术语定义 | 有新术语时生成 | 无新术语则跳过 |
| Ch.4 业务变更总览 | 始终生成 | 始终生成 |
| Ch.5 业务流程 | 生成流程图 | **不生成** |
| Ch.6 功能与规则明细 | 仅描述变更/新增的功能点 | **不生成** |
| Ch.7 接口与数据集成 | 仅描述变更/新增部分 | **不生成** |
| Ch.8 非功能性需求 | 描述有特殊要求的项 | **不生成** |
| Ch.9 验收标准 | 始终生成（对应实际变更） | 始终生成 |
| Ch.10 待定事项 | 有待定项时生成 | 无待定项则跳过 |

### 尾部汇总

在PRD最后一个实质章节之后，附一段汇总：

```markdown
---
**本次需求聚焦于{变更主题}。以下方面沿用现有系统行为，未在本文档中展开描述：{逐一列出}。**
```

### 系统公约的作用

当 `knowledge-base/system-conventions.md` 存在时，AI以此为参考判断"哪些是标准行为可以不写"。系统公约是AI的内部参考材料，**不在PRD正文中引用或提及**。

### PRD 章节自定义

如果团队有特定的 PRD 章节结构需求（如增加"合规审查"章节、删除"非功能性需求"章节），可在 `knowledge-base/system-conventions.md` 中增加 `prd_structure` 配置段：

```yaml
# system-conventions.md 末尾（可选配置）
prd_structure:
  extra_chapters:            # 在标准10章之后追加的自定义章节
    - title: "合规与审计"
      position: after_ch8    # 插入位置：after_ch{N}
      description: "数据合规、审计日志、监管要求"
  skip_chapters:             # 默认省略的章节（用户仍可在 CD-01 中选回）
    - ch8                    # 非功能性需求（团队有独立 NFR 文档）
  rename_chapters:           # 章节重命名
    ch10: "风险与遗留项"     # 原"待定事项与附录"改为"风险与遗留项"
```

**执行逻辑**：
- 叙事规划阶段读取此配置，自动调整章节列表和编号
- `extra_chapters` 追加到 CD-01 的选项中
- `skip_chapters` 对应的章节在 CD-01 中默认不选（但用户可主动选回）
- 如无此配置段，按标准 10 章结构生成

## 各章节撰写参考

> **注意**：以下是各章节的**参考指引**，不是必须逐项填满的模板。每章是否生成、详略程度，由叙事规划决定。模板结构见 `templates/prd-template.md`。

### 第1章: 文档信息

- **叙事角色**：标识文档身份，始终生成
- **何时省略**：不省略
- **详略判断**：固定格式，无需调整

### 第2章: 需求概述

- **叙事角色**：回答"为什么要做这件事"，让不了解背景的人快速进入语境
- **何时省略**：不省略
- **详略判断**：始终是 PRD 的重要章节。§2.2 业务现状是 AS-IS 的**唯一完整描述**，后续章节回指此处

各小节要点：
- **§2.1 背景与动机**：从 intake.md 提取，基于用户确认的内容，不添加推测
- **§2.2 业务现状（AS-IS）**：当前流程和系统现状的唯一完整描述
- **§2.3 痛点**：量化描述问题的影响
- **§2.4 目标与指标**：每个目标使用 G-XX 编号，包含当前值→目标值。目标必须与§2.3痛点对应
- **§2.5 范围**：直接从 clarification.md 提取 in scope / out of scope / 将来考虑
- **§2.6 需求合理性分析**：仅自主模式生成

### 第3章: 术语定义

- **叙事角色**：确保读者对关键术语有统一理解
- **何时省略**：无新术语且读者对领域熟悉时
- **详略判断**：从知识库 glossary.yaml 提取本PRD涉及的术语，新术语也在此定义

### 第4章: 业务变更总览

- **叙事角色**：让读者快速知道"改了什么"，是方案层的入口
- **何时省略**：requirement_type = new 时可简化为功能清单
- **详略判断**：每个变更项使用 C-XX 编号。AS-IS 列仅写一行摘要，详细现状回指§2.2

### 第5章: 业务流程

- **叙事角色**：回答"事情是怎么运作的"，帮助读者建立全局动态认知
- **何时省略**：纯规则/配置变更、无流程变化时
- **详略判断**：叙事规划中"业务流程"维度的详略决定本章深度

本章描述流程的**整体走向和状态流转**，不包含步骤表（步骤详情在第6章）。

**主流程（§5.1）**：
- 泳道图使用 YAML DSL（规范见 `diagram-yaml-schema.md`），或简单流程用 Mermaid
- **流程图节点标注关联的功能编号**（F-XXX），便于跳转到第6章
- 用简要文字概述流程走向，**不使用步骤表**

**分支/异常流程（§5.2）**：
- 从主流程中标出分支点，异常处理必须到"最终处理结果"

**状态流转（§5.3）**：
- 用Mermaid状态机图，每个转换标注条件和操作

### 第6章: 功能与规则明细

- **叙事角色**：回答"每个功能具体怎么做、有什么规则"，是开发实现的核心依据
- **何时省略**：叙事规划中"功能规则"维度标为省略时（极少见）
- **详略判断**：**只写本需求特有的规则**，行业标准做法、系统默认行为不写

每个功能点使用 F-XXX 编号，以 `####` 级标题呈现。Word 输出时，以 ID 前缀开头的 `####` 标题自动渲染为 Feature Heading（彩色左边条 + 浅色背景），不以 ID 前缀开头的 `####` 标题仍渲染为普通 H3。此特性由 `scripts/md2docx.mjs` 自动识别，Markdown 侧无需额外标注。

**Feature Heading 配色规格**：

| ID 前缀 | 类型 | 色系 | 深色（文字+边条） | 浅色（背景） |
|---------|------|------|-----------------|------------|
| F-XXX | 功能点 | 蓝色 | #1E3F6F | #E8EEF5 |
| IF-XXX | 接口 | 绿色 | #155E3D | #E6F4ED |
| G-XX | 目标 | 琥珀 | #8E5B08 | #FDF3E3 |
| C-XX | 变更点 | 紫色 | #5B3A8C | #F0EBF8 |

**格式根据内容复杂度选择**：

**格式一：步骤-规则表**（多步骤操作、规则≥3条时推荐）

```markdown
### F-001 {功能点名称}

**功能描述**：{一句话描述}
**操作角色**：{角色}  |  **前置条件**：{条件}
**关联变更**：C-XX（仅 update/mixed 类型需要）

**操作步骤与规则**：

| 步骤 | 操作 | 业务规则 | 异常处理 |
|------|------|---------|---------|
| 1 | 用户提交订单 | 金额>0，必填字段校验 | 字段缺失→提示补全 |
| 2 | 系统自动分仓 | 距离最近+库存充足 | 无可用仓→通知运营 |
```

规则和异常在表内使用**局部序号**（规则1/2/3），不使用全局 BR/E 编号。跨步骤规则可用"跨步骤规则"补充表。

**格式二：叙述式**（单步骤、1-2条规则时）

```markdown
### F-002 {功能点名称}

**功能描述**：{一句话描述}
**操作角色**：{角色}  |  **前置条件**：{条件}

系统在订单创建时自动校验收货地址的经纬度信息。地址缺少经纬度时，
提示用户补充后方可继续提交。
```

**格式三：变更对比表**（纯配置/参数变更时）

```markdown
### F-003 {功能点名称}

**功能描述**：{一句话描述}

| 配置项 | 变更前 | 变更后 | 说明 |
|--------|--------|--------|------|
| 审批金额阈值 | 5万 | 10万 | 提高阈值减少审批频次 |
```

**字段/页面说明**（如涉及前端，可附加）：

| 字段名 | 类型 | 必填 | 默认值 | 校验规则 | 说明 |
|--------|------|------|--------|---------|------|

### 第7章: 接口与数据集成

- **叙事角色**：回答"系统之间怎么协作、数据怎么流转"
- **何时省略**：无新增/变更接口、无数据模型变更时
- **详略判断**：**新增接口需完整定义；变更接口只写变更部分**；标准接口引用现有文档

本章合并了数据模型、数据流向、接口需求、数据一致性与迁移。

- **§7.1 数据流总览图**：用Mermaid画数据流向
- **§7.2 数据模型**：按需生成（新实体/关系变更→必须生成ER图；仅字段变更→用§7.3清单即可）
- **§7.3 数据变更规格**：仅 update/mixed 且涉及数据结构变更时生成
- **§7.4-7.5 接口**：每个新增/变更接口使用 IF-XXX 编号

接口详细定义格式：

```markdown
#### IF-001: {接口名称}

- 接口方向: {调用方} → {被调用方}
- 触发时机 / 调用方式 / 是否新增
- 关联功能: F-XXX

**请求/响应参数**：（表格）
**异常处理**：超时/失败/幂等策略
```

- **§7.6-7.8**：数据一致性、迁移、上线初始化 — 按需生成

### 第8章: 非功能性需求

- **叙事角色**：告诉读者本需求对系统有哪些**超出现有标准**的特殊要求
- **何时省略**：沿用现有标准、无特殊要求时。尾部汇总中一句话说明即可
- **详略判断**：**只写有具体指标且不同于现有标准的项**，"沿用标准"的不展开

### 第9章: 验收标准

- **叙事角色**：回答"怎么证明需求做对了"
- **何时省略**：不省略
- **详略判断**：**聚焦高风险和易出错场景**，显而易见的正向验收不占篇幅

验收表保留"关联目标"（G-XX）和"关联功能"（F-XXX）列，确保需求可追溯：

```markdown
| # | 验收项 | 前置条件 | 操作步骤 | 预期结果 | 关联目标 | 关联功能 |
|---|--------|---------|---------|---------|---------|---------|
| 1 | ... | ... | ... | ... | G-01 | F-001 |
```

### 第10章: 待定事项与附录

- **叙事角色**：收纳未决问题、假设和参考信息
- **何时省略**：无待定项、无假设时可简化
- **详略判断**：有内容就写，没有就跳过

## 交叉引用与 ID 体系

### 三层 ID 体系

PRD中使用分层的ID体系，**根据叙事规划选择需要的层级**：

| 层级 | ID | 定义位置 | 引用位置 | 使用条件 |
|------|-----|---------|---------|---------|
| **第一层（始终使用）** | G-XX | Ch.2§2.4 目标 | Ch.9 验收"关联目标"列 | 始终 |
| | F-XXX | Ch.6 功能标题 | Ch.5 流程图节点、Ch.9 验收"关联功能"列 | 始终 |
| **第二层（按需使用）** | C-XX | Ch.4 变更项 | Ch.6 功能"关联变更"字段 | 仅 update/mixed |
| | IF-XXX | Ch.7§7.4 接口 | Ch.6 功能中引用、Ch.7§7.5 详细定义 | 有新增/变更接口时 |
| **第三层（局部编号）** | 规则1/2/3 | Ch.6 步骤-规则表内 | 表内跨步骤引用 | 表内使用，不纳入全局引用 |
| | 异常1/2/3 | Ch.6 步骤-规则表内 | 表内引用 | 同上 |

**不再使用全局编号的**：
- ~~BR-XXX~~：规则在 F-XXX 的表格内用局部序号
- ~~E-XXX~~：异常处理在 F-XXX 的表格内用局部序号
- ~~DL-XXX~~：决策记录用标题引用（如"见决策记录：分仓策略选择"）

### 正文引用风格

**正文中用叙事逻辑连接**，而非标号串联：

```markdown
# 避免这样写：
系统应自动分配仓库（F-001），按照 BR-002 距离优先规则执行，
调用 IF-001 获取距离数据，对应变更项 C-01，支撑目标 G-01。

# 应该这样写：
系统自动分配仓库时，优先选择距离最近且库存充足的仓库。
距离数据通过调用地图服务接口获取（详见§7.5 IF-001）。
```

- **F-XXX**：保留在章节标题中（`### F-001 自动分仓`），流程图节点标注
- **G-XX**：保留在目标表和验收表中
- **C-XX / IF-XXX**：在定义处和首次引用处出现，正文中不反复引用
- 模块间关联优先用**章节回指**（如"详见§5.1"）或**语义引用**（如"如前述分仓流程"）

## 决策日志生成规则

在撰写 PRD 过程中，当遇到以下情况时，创建决策记录并写入 `decision-log.md`（使用 `templates/decision-log-template.md`）：

**触发条件**：
1. 存在 ≥2 个可行方案且需要取舍时
2. 用户在 Stage A 中明确讨论过的方案选择
3. AI 在自主模式下推断选择了某方案，且存在其他合理替代方案
4. 涉及业务规则的关键判断（如计算公式选择、异常处理策略选择）

**不触发条件**：仅有一种合理实现方式、用户明确指定无替代方案、纯格式决定

**PRD 正文引用格式**：

在涉及方案决策的位置，用语义引用：

```markdown
> 决策背景见 [决策记录：分仓策略选择](decision-log.md#分仓策略选择)
```

## 逐章实时一致性检查

**目的**：在撰写过程中（而非全部写完后）及时发现结构性问题，降低返工成本。

**执行时机**：每完成一个主要章节（§2 需求概述、§5 业务流程、§6 功能与规则、§7 接口与数据、§9 验收标准）后，静默执行以下轻量检查子集：

| 章节完成后 | 检查内容 | 等效 CK 项 |
|-----------|---------|-----------|
| §2 需求概述 | G-XX 目标是否全部来自 intake.md；目标数量与 intake 痛点数对应 | CK-0.2 子集 |
| §5 业务流程 | 流程图中的步骤是否与功能点清单（叙事规划）一一对应；每个 decision 节点是否有明确判断条件 | CK-2.2 子集 |
| §6 功能与规则 | 每个 F-XXX 是否有异常处理描述；规则是否有明确触发条件（无模糊语言） | CK-1.4 + CK-3.1 子集 |
| §7 接口与数据 | IF-XXX 引用的实体是否在 §6 中有定义；字段名是否与 §6 一致 | CK-2.5 子集 |
| §9 验收标准 | 引用的 F-XXX / G-XX 是否都已在前面章节定义 | CK-2.6 子集 |

**执行规则**：
- **静默修复**：发现问题立即修正（如 F-XXX 缺少异常处理则补充），不告知用户
- **不可修复问题记录**：无法自动修复的问题（如需要用户提供业务信息）记录到内部待处理列表，在 Stage C 集中呈现
- **轻量模式不执行**：轻量模式一次性生成，不拆章检查
- **性能约束**：每次检查仅针对当前章节 + 已完成章节的交叉引用，不扫描未写章节

   - Markdown 始终是主要交付物，Word 不阻断交付

## 自主模式撰写调整

当处于自主模式（Stage B）时，PRD撰写行为有以下调整：

### PRD 大纲确认

在正式撰写前，先输出叙事规划 + PRD 大纲供用户确认（详见 `references/autonomous-mode.md` 中的"PRD 大纲确认"章节）。用户确认后再开始逐章撰写。

### 一次性生成

- 大纲确认后，一次性生成全部章节，不逐章等待用户确认
- 生成过程中不中断、不提问（除非遇到绝对无法推断的红线信息）

### 标记规则

自主模式使用三级标记体系：

| 标记 | 说明 |
|------|------|
| （无标记） | 用户明确说过的内容，直接写入 |
| `[推断]` | AI基于专业判断补充的行业标准做法 |
| `[待确认]` + 当前假设 | AI不确定的业务决策 |

### 质疑的文档化

在PRD **第2章§2.6**增加"需求合理性分析"段落，列出AI对需求的审视意见。该段落作为PRD正式内容，直接写入。

### 知识库引用

引用知识库内容时标注来源：

```markdown
当前出库流程为"拣货→复核→打包→发运"四步流程。
（参考: knowledge-base/domain-wms.md §核心流程）
```

### 假设总览表

自主模式生成的PRD，在第10章§10.3 中必须包含假设总览表，是 Stage C 审阅的核心工具。
