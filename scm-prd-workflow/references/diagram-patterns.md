# 图表选型与模式速查

本文档帮助决定**画哪种图、用什么格式、什么时候不画**。配合 [`diagram-yaml-schema.md`](./diagram-yaml-schema.md)（YAML 语法）使用。

---

## 1. 工具选择：Mermaid vs YAML DSL

| 工具 | 特点 | 适用 |
|---|---|---|
| **Mermaid** | 纯文本可版本控；Markdown 内嵌；多数文档平台原生渲染 | 状态图、时序图、数据流、≤12 节点的简单流程 |
| **YAML DSL → draw.io** | 真正的泳道容器、精确布局；`.drawio` 可在 draw.io 编辑器二次编辑 | 泳道图、>12 节点复杂流程、>5 实体 ER 图 |

**判断阈值**：
- 节点数 **≤ 12** + 角色 **≤ 1** → Mermaid
- 节点数 **> 12** 或角色 **≥ 2** + 跨角色协作 → YAML DSL
- 状态机 / 时序图 → 始终用 Mermaid（YAML DSL 不擅长）

---

## 2. 图表类型选择表

| 场景 | 图表类型 | 推荐格式 | 判断规则 |
|------|---------|---------|---------|
| 跨角色 / 跨系统的业务流程 | 泳道图 | **YAML → draw.io** | 涉及 ≥2 个角色/系统的协作流程 |
| 复杂流程（>12 节点 + 交叉连线） | 流程图 | **YAML → draw.io** | 节点数 >12 或存在跨泳道交叉连线 |
| 简单泳道（≤3 泳道 + ≤12 节点） | 泳道图 | Mermaid（可选） | 泳道少且节点少时可用 Mermaid 简化 |
| 实体状态变化 | 状态机图 | Mermaid | `stateDiagram-v2` |
| 系统间数据流 | 数据流图 | Mermaid | `graph LR` |
| 复杂时序交互 | 时序图 | Mermaid | `sequenceDiagram`，用 alt/opt 表分支 |
| 系统架构概览 | 架构图 | Mermaid | `graph TD` + `subgraph` |
| 简单流程（≤12 节点） | 流程图 | Mermaid | `graph TD` |
| 数据模型（实体关系） | ER 图 | Mermaid (≤5 实体) / **YAML → draw.io** (>5) | 涉及新实体或关系变更 |

完整 Mermaid 模式示例代码见 [`./examples/diagram-all-patterns.md`](./examples/diagram-all-patterns.md)。

## 3. 泳道图布局约定

所有 YAML 泳道图遵循统一规则：

- **泳道横向排列**：每条泳道是一个竖直的列，从左到右
- **流程从上往下**：节点在泳道列内纵向排列，跨泳道连线在列之间横向连接
- **泳道之间有清晰分隔**：每条泳道列有独立边框
- `lanes` 列表的定义顺序 = 从左到右的列顺序

颜色板（YAML DSL 内建）：`blue` / `green` / `orange` / `purple` —— 用于区分不同角色 / 系统。

## 4. 模式速查表

| 模式 | 适用场景 | 格式 | 关键约定 |
|------|---------|------|--------|
| 泳道图 | ≥2 角色协作 | YAML → draw.io | lanes 顺序=从左到右；`style: error` 标异常路径 |
| 状态图 | 实体生命周期 | Mermaid `stateDiagram-v2` | 标注触发条件和不可逆转换 |
| 数据流 | 系统间数据传递 | Mermaid `graph LR` | 标注数据格式和触发时机 |
| 时序图 | 系统间调用时序 | Mermaid `sequenceDiagram` | 用 alt/opt 表分支，标注超时/重试 |
| ER 图 | 数据模型 | Mermaid(≤5 实体) / YAML(>5) | 标注关系基数和关键字段 |

## 5. 每张图的成本收益判断

**核心原则**：一张图 = 一个表达意图。先确定要回答什么问题，再决定画什么内容。

| 判断问题 | 答"是" → 画 | 答"否" → 不画 |
|---------|------------|-------------|
| 这里的逻辑用文字是否散乱难懂，图能一目了然？ | 画 | 文字足够则不画 |
| 这张图是否传递了文字无法替代的信息（空间关系、并行性、状态转换）？ | 画 | 与文字重复则冗余 |
| 读者是否需要反复参照这张图来理解后续内容？ | 画（高复用） | 一次性信息用文字 |

**别画**的常见情形：
- 用图重复描述刚刚说过的文字
- 流程只有 3-4 个线性步骤，画图反而拖沓
- 试图把"概览"和"细节"塞进同一张图

---

## 6. 各类图的适用时机

| 图表类型 | 什么时候画 | 什么时候不画 |
|---------|----------|------------|
| **泳道流程图** | 跨系统协作 + 有分支/交互；变更类需求需展示改动在全局中的位置 | 纯线性传递、单系统简单步骤 |
| **状态图** | 实体状态 ≥3 个，状态转换有条件约束 | 两态开关、转换逻辑简单 |
| **时序图** | 需表达系统间调用的时间先后和并行性 | A→B→C 同步链，流程图已清晰 |
| **ER 图** | 新增实体 或 实体关系变更（如 1:N→N:M） | 仅字段增减，无新实体或关系变更 |
| **数据流图** | 需展示数据在系统间的流向和转换 | 与流程图信息重复 |

---

## 7. 变更类（Update）场景的图表范围

当用户的诉求是"在现有流程上加/改一段"，图的范围**不是整个系统全流程**，而是**变更点 + 周边上下文**：

- **小改动**（节点增加一个判断）：只画该节点所在的局部流程段 + 上下游各 1-2 步，用 `style: highlight` 标注变更点
- **中等改动**（新增子流程嵌入现有流程）：画从切入点到回归点的完整路径；现有步骤正常色，新增步骤 highlight
- **大改动**（全新跨系统流程）：画完整流程，按新建处理

示例见 [`../examples/flow-change-local.diagram.yaml`](../examples/flow-change-local.diagram.yaml)。

---

## 8. 图内重组优先于拆图

当一张图连线复杂但仍服务于同一表达意图时，**先尝试在图内重组**，而非拆分：

- **策略 1：展开耦合节点**——一个节点同时触发多个系统 → 拆成发送/接收两个节点各归其泳道
- **策略 2：增加泳道分离职责**——多个系统的逻辑混在同一泳道 → 增加泳道让各系统步骤各在自己列中
- **策略 3：区分主次流程**——主流程用正常样式，辅助/回调流程用 `style: async`（虚线），视觉上区分主干和支线

## 9. 何时拆图（多信号柔性判断）

**不拆的原则**：如果一张图在回答同一个问题，即使节点多也不拆——拆开会丢失上下文。

**节点数不是充分必要条件**——12 是工程经验值不是硬限：10 节点全线性的图毫无问题，8 节点带 6 个决策的图就该拆。判断是否拆图用**多信号组合**：

| 信号 | 触发条件 |
|---|---|
| 节点数偏多 | >12（评估信号） |
| 跨泳道边占比高 | >50% 的边跨泳道 |
| 决策密度高 | 决策节点 ≥3 个 |
| 嵌套子流程 | 某节点本身是另一个独立流程的起点 |
| 多叙事并存 | 一张图同时回答 ≥2 个独立问题 |
| 回环 / 反向边 | YAML 中存在 to 指向更早节点的边（引擎不支持） |

**决策规则**：
- 任 2 个信号命中 → **强烈建议拆图**
- 回环信号命中 → 必须重构（引擎不支持）
- 嵌套子流程命中 → 必须拆

详细背景与能力边界见 [`layout-philosophy.md`](./layout-philosophy.md)。

**拆图结构（两层封顶，不更深）**：
- 概览图（1 张）：subprocess 节点代表可展开的子流程，≤10 节点
- 子流程图（2-3 张）：标明入口 / 出口，不再拆分
- 命名：概览 `{name}.diagram.yaml`，子流程 `{name}-{sub}.diagram.yaml`

## 10. 反模式速查

| 反模式 | 表现 | 后果 | 正确做法 |
|---|---|---|---|
| 决策只有"是"没有"否" | 判断节点只画一条出边 | 异常路径丢失，实现方遗漏错误处理 | 决策节点必须穷举分支 |
| 只画 happy path | 异常路径完全不画 | 实现脆弱、测试覆盖不足 | 主流程 + 关键异常路径都要画，异常用 `style: error` |
| 概览与细节混画 | 一张图同时含高层步骤和底层细节 | 读者抓不住重点 | 拆为概览 + 子流程两层 |
| 用流程图表达状态 | 用 process 节点表达状态变化 | 状态语义不清；漏掉转换条件 | 改用 Mermaid `stateDiagram-v2` |
| 节点名是名词 | "订单创建"、"通知发送" | 读起来像数据，不像动作 | 用动宾短语：动词+宾语，"创建订单""发送通知" |
| 子图标题用"步骤 N" | 泳道命名为"步骤 1""阶段 2" | 不传达角色/系统信息 | 用真实的系统/角色名："OMS""WMS""质检员" |
| 一张图节点 >25 | 试图把所有事都塞进一张图 | 视觉过载、连线混乱 | 评估能否图内重组；不行就拆图 |

---

## 11. 节点命名与标注约定

- **节点命名**：动宾短语（"创建订单"而非"订单创建"），保持主动语态
- **判断分支**：必须穷举，标注条件文字（"合格"/"不合格"/"超时"）
- **异常路径**：用 `style: error`（红色）标注；超时、降级、回滚都属异常
- **异步通知**：用 `style: async`（虚线）标注非阻塞调用
- **泳道命名**：用真实系统/角色名（"OMS""仓库""客户"），不用"步骤 1"

---

## 12. 图与故事的关系

图不是"把所有想到的画出来"，而是"用空间结构表达流程的核心逻辑"。先用自然语言复述流程对齐故事，再落地为 YAML——避免"图画完了用户才说不对"的返工。

完整 YAML 语法见 [`diagram-yaml-schema.md`](./diagram-yaml-schema.md)。
