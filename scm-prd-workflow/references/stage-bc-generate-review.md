## Stage B: 自主生成

**进度播报**：叙事规划阶段输出 `▰▰▱▱▱ [2/{total}] 叙事规划`；PRD 撰写开始时输出 `▰▰▰▱▱ [3/{total}] PRD 撰写`。CD-01 确认后如原型选项变化导致 total 变化，**重新输出路线图**。

### 目标

一次性生成完整PRD及全部附属文件，全程无用户交互。

### 生成顺序

1. **整理信息** → 输出 `intake.md`（使用 `templates/autonomous-intake-brief.md`）
2. **推导假设** → 输出 `clarification.md`（AI基于信息缺口自主推导，标记假设依据）
3. **叙事规划 + PRD 大纲** → 输出叙事规划和大纲供用户确认（详见 `writing-narrative.md`）
4. **撰写PRD** → 读取 `writing-chapters.md` 和 `diagram-patterns.md`，按叙事规划详略策略逐章撰写
5. **记录决策** → 多方案选择时同步写入 `decision-log.md`
6. **绘制流程图** → 保存到 `diagrams/`，详见 `diagram-patterns.md`
7. **导出图表** → 调用 `scripts/export-diagrams.py`
8. **生成 Word** → `docx_engine = "js"` 时调用 md2docx.mjs
9. **脚本自检（硬逻辑）** → `check-prd-consistency.py`，结果写入 review-report.md
10. **AI 自检（软逻辑）** → 读取 `review-guide.md`，仅执行语义检查
11. **生成报告** → 输出 `review-report.md`

### PRD 大纲确认（叙事规划）

叙事规划 + PRD 大纲（1页以内）供用户确认。**大纲核心是表达策略，不是盘点数量。**

完整格式参考 `writing-narrative.md`。叙事规划自检（NP-01~11）详见 `writing-narrative.md`。
章节详略选择（CD-01）详见 `writing-narrative.md`。

### 标记使用

标记体系详见 `core-conventions.md`。自主模式使用三级标记（无标记/`[推断]`/`[待确认]`）。

#### 绝不假设的红线

以下信息必须在 Stage A 中从用户获得，不可假设：
- 核心业务问题的定义（用户到底要解决什么）
- 系统模块范围（到底改哪些系统）
- 主要用户角色（到底谁在用）
- 基本期望目标（到底要达到什么效果）

如果这些信息在 Stage A 未获得，不得进入 Stage B。

### 质疑的文档化

质疑写入 PRD §2.6 需求合理性分析。

### 知识库引用规范

引用知识库内容时标注来源：`（参考: knowledge-base/domain-xxx.md §章节）`

### clarification.md 的自主生成

由AI自动生成，设置 `mode: autonomous`、`status: ai-drafted`，按 9 维度框架逐维度填充。

## Stage C: 审阅迭代

**进度播报**：进入本阶段时输出 `▰▰▰▰▱ [4/{total}] 审阅迭代`。多轮迭代期间步骤编号不变。

### 目标

将完整PRD交给用户审阅，通过批量反馈高效迭代。

### 核心交互原则

**静默确认机制**：`[推断]` 项在呈现后，若用户未在当轮对话中提出异议，自动视为确认。

### 呈现顺序

Stage B 完成后，按以下顺序向用户呈现：

**第一步：假设总览表（一次性呈现）**

假设按影响程度分两层，高影响用加粗标注。影响程度判断标准：

| 等级 | 判断标准 | 典型场景 |
|------|---------|---------|
| 高 | 影响核心业务逻辑、数据安全或资金准确性，或涉及≥2个下游系统 | 库存扣减规则、金额计算精度、审批阈值 |
| 中 | 影响次要流程、操作效率或单系统内部行为 | 异常处理策略、通知方式、批量操作上限 |
| 低 | 仅影响展示/体验层面 | 字段排序、默认排序方向、提示文案 |

**第二步：批量反馈（SC-01）**

假设总览表 + 自检报告摘要 + 完整 PRD 一次性呈现后，以自由文本方式邀请用户反馈：

> "以上是完整 PRD 和假设总览。请在下方一次性反馈需要修改的假设和其他调整意见（引用编号即可），未提到的假设视为确认。"

**不使用 AskUserQuestion**，不设"确认"按钮——用户直接回复修改意见。

**第三步**：自检报告摘要。**第四步**：完整PRD文档。

### 用户反馈处理规则

AI 解析用户自由文本中的编号引用和修改意见，批量处理：

| 反馈类型 | 识别方式 | 处理 |
|---------|---------|------|
| 假设修改 | 引用了编号 | 更新对应假设，移除标记 |
| 假设确认 | 未提到的编号 | 视为确认，批量移除标记 |
| 章节调整 | 提到章节号或内容区域 | 对指定章节补充细化 |
| 新增需求 | 描述了 PRD 中没有的功能/规则 | 补充到对应章节，标记为新增 |
| 局部深入 | 某维度反馈量大或不确定 | AI 主动建议深入讨论（SC-05） |

### 变更摘要输出与持久化

变更摘要格式详见 `review-guide.md` "变更摘要"章节。

### 迭代效率

- 每次修改后，追加变更记录到 prd-changelog.md（含影响传播分析）
- 每次修改后，仅对受影响的章节重新执行自检
- 假设确认后，批量移除标记
- 支持"先确认大方向，再细化"的渐进式审阅

### 局部深入讨论（SC-05）

当用户的批量反馈中某一维度 >50% 的假设被修改，或用户对某个区域表示不确定时，AI 主动建议深入讨论。

### 最终交付（SC-06）

交付流程详见 `review-guide.md` "最终交付"章节。自主模式额外：
- CD-01 选中"原型演示" → 进入 Stage D
- 更新约束索引
