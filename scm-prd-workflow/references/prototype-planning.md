# 原型规划参考文档

## 概述

本文档是 PRD 原型规划阶段的操作参考，指导 AI 判断是否需要生成原型、确定原型范围和精细度、以及将 PRD 内容转化为可执行的原型规范。

**核心理念**：原型是 PRD 的补充表达手段，不是必选项。目的是让评审者 3 分钟内理解"改了什么"，不是做 UI 设计稿。不做比乱做好。

---

## 全流程与门控总览

原型流程嵌入 PRD 流程中，共 3 个用户门控（标 🚧），均通过 `AskUserQuestion` 强制确认，AI 不可跳过。

```
Stage B: 叙事规划
│
├─ AI 评估触发条件 T1-T5
├─ 输出原型 Brief（嵌入叙事规划）
│
▼
🚧 PT-01（嵌入 CD-01 multiSelect）
│  用户选中"原型演示" → 启用原型流程
│  用户未选中 → 原型流程结束，后续无任何原型动作
│
▼
[PRD 撰写 → PRD 审阅 → PRD 确认]（正常 PRD 流程，不受原型影响）
│
▼
Stage D / Post-review: 原型设计与生成
│
├─ AI 从 PRD 提取数据，生成 prototype-design.md 初稿
│
▼
🚧 PT-02（设计方案迭代，可多轮）◄───────┐
│  ├─ "确认，开始生成" → 继续            │
│  ├─ "需要调整" → 用户反馈 → AI 更新 ──┘
│  └─ "跳过原型" → 原型流程结束
│
├─ CK-PT 自动检查（关键项不通过 → 自动修正后重检）
├─ 调用 web-artifacts-builder 生成代码 + 打包
│
▼
🚧 PT-03（原型结果确认）
   ├─ "满意，完成交付" → 交付，原型流程结束
   ├─ "需要修改" → 回到 PT-02 调整设计方案
   └─ "有小问题，直接改代码" → 代码层微调 → 交付
```

### 门控清单

| ID | 名称 | 类型 | 时机 | 选项 | 强制？ |
|----|------|------|------|------|--------|
| PT-01 | 原型方向确认 | CD-01 的 multiSelect 扩展选项 | 叙事规划后、PRD 撰写前 | 选中=启用 / 不选=跳过 | 是（嵌入 CD-01，CD-01 本身不可跳过） |
| PT-02 | 设计方案确认 | AskUserQuestion 单选 | prototype-design.md 每轮迭代后 | 确认/调整/跳过 | 是（不确认不生成代码） |
| PT-03 | 原型结果确认 | AskUserQuestion 单选 | 原型 HTML 生成后 | 满意/修改设计/改代码 | 是（不确认不交付） |

**关键约束**：
- PT-01 未选中 → 后续不出现任何原型相关输出或提问
- PT-02 "跳过原型" → 等价于事后撤回 PT-01，流程干净退出
- PT-02 迭代不限轮次（设计对话没有收敛压力，用户说确认才算确认）
- PT-03 "需要修改" 回到 PT-02 而非直接改代码（设计方案是 source of truth）
- **原型必须零外部依赖**：产出的 `bundle.html` 必须是完全自包含的单文件，所有 JS/CSS 内联，不依赖任何外部 CDN（如 jsdelivr、tailwindcss.com 等）。用户双击即可在浏览器中打开，无需联网、无需安装任何工具。使用 `web-artifacts-builder` 打包可自动满足此要求；如不使用打包工具直接生成 HTML，必须手动内联所有依赖

## 触发判断

### 触发标准（命中任一即建议做原型）

| # | 触发条件 | PRD 中的信号 | 原因 |
|---|---------|-------------|------|
| T1 | 新增用户交互界面 | F-XXX 有字段/页面表 + 多步操作 | 纯文字无法传达交互感受 |
| T2 | 复杂状态机 | Ch.5 状态图 ≥4 状态 | 状态转换在表格中极难推演 |
| T3 | 屏幕级 AS-IS→TO-BE | C-XX 涉及 UI 字段/操作变更 | 前后对比在可视原型中最直观 |
| T4 | 多步分支流程 | F-XXX 步骤-规则表 ≥4 步 + 异常分支 | 分支逻辑在文字中容易丢失 |
| T5 | **新增**列表/批量操作 | F-XXX 新增列表页、新增批量动作、新增筛选维度（对已有列表的小修改不触发） | 列表信息密度无法用文字描述 |

### 反触发（不做原型的场景）

- 轻量模式的简单字段/配置改动（除非用户显式要求）
- 纯后端变更（接口/数据迁移/计算规则，无 UI 表面）
- 叙事规划中标为"省略"的维度
- 全新系统尚无基线（无上下文可对比，除非用户要求）

### 判断流程

```
叙事规划完成
  │
  ├─ 扫描 F-XXX 字段/页面表 → 有？→ 检查 T1
  ├─ 扫描 Ch.5 状态图 → ≥4 状态？→ 检查 T2
  ├─ 扫描 C-XX 表 → 涉及 UI？→ 检查 T3
  ├─ 扫描 F-XXX 步骤表 → ≥4 步+分支？→ 检查 T4
  └─ 扫描 F-XXX 描述 → 列表/批量？→ 检查 T5
  │
  ├─ 命中 ≥2 条 + 无反触发 → 建议生成原型
  ├─ 命中 1 条 → 不主动建议，但在 CD-01 中保留选项供用户主动选择
  └─ 无命中 或 全命中反触发 → 不生成原型
```

## 精细度判断

| 级别 | 适用场景 | 特征 | 屏幕数 |
|------|---------|------|--------|
| **意图原型** | 大多数需求 | 只做关键变更点的可交互演示；未变更区域灰色占位 | 1-3 |
| **高保真原型** | 重大项目/客户演示/新系统 | 完整页面布局 + 真实数据密度 + 导航结构 | 3-7 |

**选择依据**：

| 因素 | 意图原型 | 高保真原型 |
|------|---------|-----------|
| 受众 | 内部研发/测试评审 | 高层决策者/外部客户 |
| 变更范围 | 局部改动（1-3个功能点） | 新流程/新模块/重大重构 |
| 现有系统 | 有，评审者熟悉 | 无，或评审者不熟悉 |
| 叙事规划标记 | 标为"重点"的维度 ≤2 | 标为"重点"的维度 ≥3 |

## 屏幕类型定义

供应链系统的原型屏幕归纳为以下 5 种类型：

### list-view（列表视图）

**场景**：订单列表、库存查询、运单列表、对账单列表等
**典型结构**：筛选栏 + 数据表 + 行操作侧边栏
**shadcn/ui 组件**：Table, Badge, Select, Input, Button, Sheet, Checkbox, DropdownMenu
**关键交互**：
- 筛选条件联动
- 状态 Badge 颜色区分
- 行选择 + 批量操作
- 点击行展开详情（Sheet 侧边栏）

### form（表单）

**场景**：订单创建、规则配置、审批提交等
**典型结构**：分区表单 + zod 校验 + 确认弹窗
**shadcn/ui 组件**：Form, Input, Select, Calendar, Textarea, Label, Button, Dialog
**关键交互**：
- 字段级校验（必填/格式/范围）
- 联动显隐（选择类型A显示字段X，类型B显示字段Y）
- 提交确认弹窗

### dashboard（仪表盘）

**场景**：运营看板、库存概览、异常监控等
**典型结构**：KPI 卡片 + 状态分布 + 最近活动表
**shadcn/ui 组件**：Card, Tabs, Badge, Progress, Table
**关键交互**：
- 标签页切换维度
- KPI 卡片数值变化
- 状态分布条/环图

### detail-panel（详情面板）

**场景**：订单详情、运单跟踪、库存明细等
**典型结构**：侧边栏/弹窗 + 多标签 + 明细表
**shadcn/ui 组件**：Sheet/Dialog, Tabs, Table, Badge, Separator
**关键交互**：
- 标签页切换（基本信息/操作日志/关联单据）
- 状态流转按钮
- 关联数据跳转

### flow-viz（流程可视化）

**场景**：审批进度、订单履约状态、拣货流程等
**典型结构**：步骤指示器 + 当前状态 + 可用操作
**shadcn/ui 组件**：自定义 React 状态机 + Badge + Button
**关键交互**：
- 当前步骤高亮
- 可用操作按钮
- 状态流转演示

## PRD → 原型数据提取规则

### 数据契约提取（从 Ch.6 字段/页面表）

PRD 中 F-XXX 的字段/页面表包含：

| PRD 字段 | 原型用途 |
|---------|---------|
| 字段名/标签 | 表头/表单 label |
| 字段类型 | 输入组件选择（String→Input, Enum→Select, Date→Calendar, Boolean→Switch） |
| 必填 | zod 校验规则 |
| 默认值 | 表单初始值 |
| 取值范围/枚举 | Select 选项 / Badge 变体 |
| 只读/可编辑 | 是否为 disabled 状态 |

### 状态枚举提取（从 Ch.5 状态图）

Mermaid `stateDiagram-v2` 中的：
- 状态名 → Badge 变体（不同颜色）
- 转换条件 → 操作按钮的 label
- 转换方向 → 按钮可用性逻辑

### 交互提取（从 Ch.6 步骤-规则表）

| 步骤表列 | 原型对应 |
|---------|---------|
| "用户操作"类步骤 | 可点击的按钮/链接 |
| "系统处理"类步骤 | 点击后的状态变化/Toast 提示 |
| 异常处理 | 错误状态的 UI 表现 |
| 业务规则 | 校验逻辑/联动逻辑 |

### 校验规则提取（从 Ch.9 验收标准）

高优先级的验收标准（有明确输入→预期输出的）可映射为原型中的校验行为：
- "必填字段为空时提示" → required 校验 + 红色错误文字
- "金额>0" → min 校验
- "提交后状态变为XX" → 按钮点击后 Badge 变化

## 模拟数据规范

### 数据量

| 精细度 | 列表行数 | 表单字段数 | 状态数 |
|--------|---------|-----------|--------|
| 意图原型 | 8-12 行 | 按 PRD 字段表 | 按 PRD 状态图 |
| 高保真原型 | 15-20 行 | 按 PRD 字段表 | 按 PRD 状态图 |

### 数据内容要求

- **全中文**：标签、数据、提示信息均使用中文
- **产品名**：使用真实产品类别（"蓝牙耳机 BT-500""充电宝 PB-2000""无线鼠标 MX-300"）
- **订单号**：遵循 PRD 定义的编号规则，无定义时使用 `{系统缩写}-{YYYYMMDD}-{序号}` 格式（如 "OMS-20260402-0001"）
- **金额**：合理范围（如订单金额 50-5000，运费 5-50）
- **时间**：使用近期日期，时间跨度合理
- **状态分布**：列表中各状态应有分布（不全是同一状态），至少覆盖 PRD 定义的每种状态
- **角色名**：使用 PRD 中定义的角色名（"仓库主管""客服专员""物流调度"）

## 质量标准

| 标准 | 要求 |
|------|------|
| 运营系统质感 | B2B 内部系统风格：高信息密度、克制配色、中文标签、左侧导航栏（即使是静态的） |
| 数据真实感 | 数据量充足、中文内容、真实格式、合理分布 |
| 状态可点 | PRD 定义的状态流转在原型中可操作（点按钮切换 Badge） |
| 校验可见 | PRD 中的必填/格式/范围校验规则在表单中可触发并显示错误 |
| 批量可演 | 复选框选择 + 批量操作按钮能批量更新行状态 |
| Delta 聚焦 | 未变更区域灰色占位 + "不在本次范围" 标注，不分散注意力 |

---

## 反模式

| 禁止 | 正确做法 |
|------|---------|
| 每个 F-XXX 都做一个屏幕 | 只做命中触发条件的功能 |
| 未原型化的导航项可点击 | 标灰 + "(不在本次范围)" |
| 花哨动画/渐变/紫色主题 | 零动画，即时状态变化，克制配色 |
| 英文占位数据 / lorem ipsum | 全中文标签和数据 |
| 发明 PRD 中没有的 UI 模式 | 每个交互元素追溯 PRD 引用 |
| 为了"完整"而原型化未变更区域 | 灰色占位 + 标注 |
| 深层组件嵌套/过度工程 | 扁平组件结构，硬编码模拟数据 |
| 居中布局/卡片式首页 | 左导航 + 顶栏 + 内容区的 B2B 布局 |

## 两步流程

原型产出分两步，节奏与 PRD 自身的"叙事规划→撰写"对应：

```
Step 1: 原型 Brief（随 PRD 大纲一起出）
  · 方向性判断：做不做、做哪些屏幕、什么精细度
  · 用户在 CD-01 中确认方向
  · 产出：嵌入叙事规划，不产出独立文件

Step 2: 原型设计方案（PRD 确认后独立阶段）
  · 从 PRD 提取完整细节，构建组件结构和交互链
  · 多轮迭代打磨设计方案
  · 确定后调用 web-artifacts-builder 生成代码
  · 产出：prototype-design.md + prototype/bundle.html
```

### 为什么分两步

- **Brief 阶段** PRD 还没写，没有字段表、步骤表、状态图——只能做方向性判断
- **设计方案阶段** PRD 已确认，所有细节可提取，且设计方案本身需要和用户迭代（"详情用 Sheet 还是 Dialog""批量操作要不要确认弹窗"这些是设计对话）

---

## Step 1: 原型 Brief（嵌入叙事规划）

**时机**：Stage B 叙事规划输出时，紧跟"图表策略"之后。

**内容**：使用 `templates/prototype-brief-template.md` 的结构，嵌入叙事规划输出中（不产出独立文件）。包含：
- 意图（一句话）
- 屏幕清单（表格：屏幕、类型、来源、要表达什么）
- 精细度选择
- 体验路径（一段话）
- 不做什么

**用户确认**：作为 CD-01 的一个选项（PT-01），用户选中"原型演示"即确认方向。

---

## Step 2: 原型设计方案（PRD 确认后）

**进度播报**：原型设计阶段输出 `▰...▱ [{N}/{total}] 原型设计`；原型生成阶段输出 `▰...▱ [{N+1}/{total}] 原型生成`。具体步骤编号取决于模式（自主=5/6），参见 SKILL.md 步骤表。PT-02 多轮迭代期间步骤编号不变。

### 时机

- 自主模式：Stage C 审阅迭代结束后
- 轻量模式：Stage L3 审阅结束后（仅在罕见触发时）

### 前置检查（PT-00）

进入原型设计阶段前，**先检测** `web-artifacts-builder` skill 是否已安装（尝试读取其 SKILL.md 或调用其初始化脚本）。

- **可用** → 继续进入 Step 2.1
- **不可用** → 使用 `AskUserQuestion` 告知用户：

  > header: "原型依赖"
  > 问题: "原型生成需要 web-artifacts-builder skill，但当前未检测到该 skill。"
  > 选项：
  > - **跳过原型**：本次不生成原型，仅保留 PRD
  > - **仅输出设计方案**：生成 prototype-design.md 但不打包为 HTML（后续安装 skill 后可手动生成）

### 步骤

#### 2.1 生成设计方案初稿

使用 `templates/prototype-design-template.md` 模板，从已确认的 PRD 提取：

| PRD 来源 | 提取到设计方案的位置 |
|---------|-------------------|
| Ch.5 状态图 | §2 状态机 + 状态→Badge 颜色映射 |
| Ch.6 字段/页面表 | §3 各屏幕数据契约 |
| Ch.6 步骤-规则表 | §3 各屏幕交互设计（触发→过程→反馈→结果） |
| Ch.9 验收标准 | §3 各屏幕空状态与边界处理 |
| Brief 中的屏幕清单 | §3 屏幕列表和类型 |
| Brief 中的体验路径 | §4 推荐体验路径 |

同时补充 PRD 中没有但原型必须决定的内容：
- 全局设计决策（页面架构、详情展示方式、信息密度、Delta 标记方式）
- 组件结构（具体 shadcn/ui 组件选型和嵌套关系）
- 反馈设计（Toast/Dialog/Badge 变色等交互反馈链）
- 模拟数据（中文 SCM 领域数据，覆盖所有状态）
- 设计决策记录（选择及理由）

产出 `prototype-design.md`。

#### 2.2 设计方案迭代（PT-02）

呈现设计方案后，使用 `AskUserQuestion`：

> header: "原型设计方案"
> 问题: "原型设计方案已生成（{N} 个屏幕，{M} 个关键交互），请审阅。"
> 选项：
> - **确认，开始生成**：设计方案无异议，开始生成原型代码
> - **需要调整**：修改设计细节（进入自由文本，多轮迭代）
> - **跳过原型**：本次不生成原型

**迭代规则**：
- 用户每轮反馈后，AI 更新 `prototype-design.md`，追加迭代记录
- 重新呈现变更摘要 + PT-02
- 不限迭代轮次（原型设计是创意对话，不像 PRD 审阅有收敛压力）
- 常见迭代话题：组件选型调整、交互流程优化、增减屏幕、数据补充

#### 2.3 执行 CK-PT 检查

设计方案确定后、生成代码前，执行原型一致性检查（详见 `review-guide.md` CK-PT 章节）。

#### 2.4 调用 web-artifacts-builder 生成原型

- 调用 web-artifacts-builder skill（外部技能），由该 skill 的 `init-artifact.sh` 初始化 React 项目
- 基于 `prototype-design.md` 生成 React 组件代码
- 由 web-artifacts-builder skill 的 `bundle-artifact.sh` 打包为单文件 HTML
- **构建后修复**：运行 `node scripts/fix-bundle-fileproto.mjs prototype/bundle.html`，移除 `type="module"` 确保 Windows/macOS 双击直接打开（无需本地服务器）
- 输出到 `requirements/REQ-{date}-{name}/prototype/bundle.html`

#### 2.5 呈现原型

告知用户原型文件位置，可直接双击在浏览器中打开。

使用 `AskUserQuestion`（PT-03）：

> header: "原型结果"
> 问题: "原型已生成（`prototype/bundle.html`），双击即可在浏览器中打开体验。"
> 选项：
> - **满意，完成交付**：原型达到预期效果
> - **需要修改**：回到设计方案调整后重新生成
> - **有小问题，直接改代码**：不改设计方案，直接在代码层面微调

### 产出文件

```
requirements/REQ-{date}-{name}/
├── ... (已有 PRD 文件)
├── prototype-design.md         # 原型设计方案（迭代产物）
└── prototype/
    └── bundle.html             # 可交互原型（单文件 HTML）
```
