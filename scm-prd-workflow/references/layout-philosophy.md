# 布局哲学与引擎能力边界

本文档说明本 skill 在"画好一张图"这件事上的工程立场——什么样的图能被画好、什么样必须拆开、当前引擎止步于哪里、未来会如何演进。**所有写 YAML 的人和 AI 都应该先读这一份**。

---

## 1. 语义与布局解耦：为什么不直接给坐标

LLM 擅长理解逻辑结构（谁连接谁、哪个分支判断什么），但极不擅长像素级坐标定位。让模型直接吐 SVG 坐标是所有失败方案的共同点。

本 skill 的分层是固定的：

```
意图（自然语言）
    ↓ 阶段一：意图理解
故事（自然语言反讲）
    ↓ 阶段二：故事反讲
语义结构（YAML DSL：lanes / nodes / edges）
    ↓ 布局引擎
节点坐标 + 边路径
    ↓ 渲染器
.drawio / .svg / .png
```

**伙伴模式与工具模式都遵循这个分层**。"语义结构"是用户与 AI 对齐的最终媒介；坐标和像素不是。

---

## 2. 强制主方向

人脑扫读流程图的路径是有定向的：从上到下或从左到右。违背这个方向就是制造认知负担。

本引擎的硬约束：

- **泳道图（swimlane）**：泳道横排为列，流程**从上往下**走
- **流程图（flow）**：节点**从左到右**按拓扑排列
- **ER 图**：实体网格排列，关系连线是无向的（基数在端点标注）

**反向边会被 `validate()` 检测并告警**。如果用户的业务流程确实有"回退到供应商"这种循环，应该用一个独立的 `end` 节点（如"通知供应商补发"）作为该路径的终点，而不是让边指回起点——这能保留拓扑序的同时表达业务语义。

---

## 3. 节点尺寸规整

视觉节奏一旦乱，图就显得业余。

**规整的做法**：
- 同类型节点用同一尺寸（`process` 默认 120×60、`decision` 100×60、`start/end` 60×60）
- 差异通过**形状**（菱形/圆/矩形）和**颜色**（lane 色 / `style: error` 红 / `style: highlight` 强调）表达，**不靠文字撑开**

**反模式**：
- 把整段说明塞进节点 label
- 用超长标签让节点比同类宽出一倍

引擎会按 label 长度自动加宽节点，但这会触发布局微调。**label 控制在 12 个字符内**最稳。超出则改放到节点外的注释里（用 markdown 旁边章节描述）。

---

## 4. 减少边交叉永远比追求美观重要

一张交叉很多但配色精致的图，远不如一张交叉少但朴素的图易读。这也是为什么 Sugiyama 类布局算法的核心步骤就是 crossing minimization。

**当前引擎的局限**：本 skill 用拓扑分层 + 局部端口分配，**不做全局 crossing minimization**。意味着：

- 简单 DAG（≤12 节点）几乎不会有交叉
- 中等 DAG（13-20 节点）可能出现 1-2 处交叉
- 复杂 DAG（>20 节点）或密集跨泳道边场景，**交叉数量不可控**

**规则**：当画出来发现 ≥3 处明显交叉时，**立刻拆图**（见 §6），不要试图通过调整 YAML 顺序硬抠。

---

## 5. Schema 可校验

LLM 偶尔会幻觉出来：
- 孤立节点（没有边连接）
- 重复 ID（同一个 id 两个节点）
- 悬空边（`from` 或 `to` 引用不存在的节点）
- 自指环（节点连到自己）

`scripts/diagram_core.py::validate()` 把这些都接住，输出错误清单。**伙伴模式阶段三写完 YAML 后必须跑一次 validate**——这一步不能省，是 LLM 产物的最后一道防线。

---

## 6. 复杂图必拆：多信号柔性判断

"节点数超过 N 就拆图"是错的——N 是工程经验值，不是充分必要条件。本 skill 用**多信号组合**判断是否需要拆图。

### 拆图信号清单

| 信号 | 触发条件 | 含义 |
|---|---|---|
| **节点数偏多** | >12 | 评估信号，不是硬限。10 个节点全线性没问题；8 个节点有 6 个决策就该拆 |
| **跨泳道边占比高** | >50% 的边跨泳道 | 同一图在多系统间反复跳跃，叙事不聚焦 |
| **决策密度高** | 决策节点 ≥3 个 | 分支多说明流程复杂度高，拆开易读 |
| **嵌套子流程** | 某节点本身是另一个独立流程的起点 | 用 subprocess 节点占位，子流程拆为独立图 |
| **多叙事并存** | 一张图同时回答 ≥2 个独立问题（"概览 + 细节"、"主流程 + 异常"） | 强烈拆图信号 |
| **回环 / 反向边** | YAML 中存在 to 指向更早节点的边 | 引擎不支持，必须重构语义 |

### 决策规则

- **任 2 个信号命中** → 强烈建议拆图
- **回环信号命中** → 必须重构（引擎不支持）
- **嵌套子流程命中** → 必须拆（用 subprocess 节点 + 独立子图）

### 拆图结构（两层封顶）

- **概览图**（1 张）：subprocess 节点代表可展开的子流程，≤10 节点
- **子流程图**（2-3 张）：标明入口 / 出口节点，不再拆分
- 命名约定：概览 `{name}.diagram.yaml`，子流程 `{name}-{sub}.diagram.yaml`

## 7. 当前引擎能力边界

**清晰标注**——AI 在伙伴模式开始前就要对照这个表评估用户意图：

| 能力 | 状态 | 说明 |
|---|---|---|
| 严格 DAG（无环） | ✅ 完美支持 | 拓扑排序的前提 |
| ≤12 节点 | ✅ 完美支持 | 间距自适应、布局清晰 |
| ≤5 泳道 | ✅ 完美支持 | 测试覆盖到 5 泳道压力测试 |
| ≤2 出边决策 | ✅ 完美支持 | 全局端口分配优秀 |
| 13-20 节点 | ⚠️ 退化但可用 | 可能出现 1-2 处交叉，需手工调 YAML 节点顺序 |
| 长标签（>15 字符） | ⚠️ 退化 | 节点宽度变化会扰动布局；建议短化 |
| 节点 x 范围严重重叠 | ⚠️ 退化 | mid-point 避障可能找不到合法位置 |
| 跨 3+ 泳道密集边 | ⚠️ 退化 | 全局 crossing minimization 缺失 |
| **>20 节点** | ❌ 不推荐 | 视觉爆炸，强烈建议拆图 |
| **决策 3+ 出边同向** | ❌ 不支持 | 端口分配会冲突；改造决策语义或拆图 |
| **反向边 / 循环** | ❌ 不支持 | 拓扑排序失败；改用独立 end 节点 |
| **嵌套子流程** | ❌ 不支持 | 必须用 subprocess 节点 + 独立子图 |

## 8. 演进路径

本引擎是"为了解决问题而设计的"轻量级方案——核心架构通用（拓扑排序、端口分配、避障搜索），但有约 50% 的样本特化补丁（决策菱形几何处理、间距常数、`_recalc_entry` 魔数等）。

**短期内**：通过本文档与 `diagram-patterns.md` 引导用户在 YAML 写作阶段就避开能力边界，做到"问题在源头解决"而非"画完再改"。

**中长期**：如果未来需要支持反向边 / 循环 / 决策 3+ 出边 / 嵌套子流程，必须切换到工业级布局引擎。`feat/elkjs-layout` 分支正在做 ELK.js 的 spike 实验，目的是验证 Sugiyama 框架（layered placement + Brandes-Köpf node placement + crossing minimization + orthogonal edge routing）能否覆盖当前引擎的应力极限。

切换到 ELK 后：
- ✅ YAML DSL **不变**（用户接口稳定）
- ✅ 渲染器（drawio XML / SVG）**不变**（视觉风格稳定）
- ⚠️ 内部布局引擎从 `compute_layout` 切到 `elk_layout`
- ⚠️ 引入 Node 子进程依赖（违反当前零依赖原则——这是切换的代价）

spike 结果将记录在本文档末尾或独立 `spike-elkjs-report.md`。

## 9. ELK.js 切换实验记录（2026-05，结论：不切）

曾尝试 spike 把布局引擎从 builtin 切换到 ELK.js（Eclipse Layout Kernel），见 `feat/elkjs-layout` 分支（未 merge）。**最终结论：保留 builtin，不切**。

### 实验配置

- ELK 调用方式：Node 子进程（独立 Node bridge，spike 分支内）
- 算法：layered + Brandes-Köpf 节点放置 + LAYER_SWEEP 边交叉最小化 + ORTHOGONAL 边路由
- Python 端：`elk_layout` 模块（spike 分支内），把 YAML → ELK JSON → subprocess → 解析坐标
- SVG 渲染：新增 `generate_svg_elk()` 用 ELK edge sections 直接画 polyline

### 量化对比（scm-outbound-flow，17 节点 5 泳道）

| 引擎 | 边数 | 中位 | 最差 | >1.2 |
|---|---|---|---|---|
| builtin | 15 | 1.15 | 1.38 | 4 |
| elk | **11** | **1.00** | **1.29** | 2 |

ELK 中位 1.00、最差 1.29，看似显著优于 builtin。但**边数 15→11 说明 spike 边映射有 bug**——ELK 的 hierarchical box 算法在跨 lane container 的边上丢失 4 条。

### 视觉对比结论（用户走查 2 张图后判断）

> "builtin 明显好，只是偶尔有箭头是歪的"

builtin 在保持完整边数的前提下，视觉清晰度优于 spike 的 ELK 输出。ELK 的优势主要体现在中位数（接近直线），但泳道嵌套场景下 ELK 的 box layout + hierarchical edges 配合不好，跨泳道边可能丢失或路径异常。

### 为什么不修了再上

要让 ELK 达到 production-ready，至少还需：
- 修边映射 bug（边在 root vs lane container 的协调，~2-3 天）
- 调泳道 layoutOptions（让泳道横排而非自动布局，~1-2 天）
- 决策端口偏好控制（ELK 不直接支持"primary 走 BOTTOM"语义，~3-5 天）
- 标签位置精修（ELK 的 label 计算与我们的字体不一致，~1-2 天）
- 引入 Node 依赖（违反零依赖原则，体积 +25 MB）

总成本 2-3 周；收益是"可能更好的中位数"+ 解 reverse edges / loops / 嵌套子流程等当前不支持的 fundamental 场景。

**当前 4 张样本 + 2 张压力测试图都已经能正常画**（最差 1.41、>1.5 边 0 条），决策右突等系统性问题已修复，箭头方向也已对齐 TOP 端口。**ROI 不够，不切**。

### 何时重新评估

- 出现"必须画"的反向边 / 循环 / 嵌套子流程场景（builtin 不支持）
- 单图节点数稳定上升到 25+
- ELK.js 出现 Python 原生 binding（消除 Node 依赖）

满足任 1 条 → 重新评估 spike。届时直接 checkout `feat/elkjs-layout` 分支续做，不用从零开始。
