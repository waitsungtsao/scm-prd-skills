# 自主生成模式 — 操作指引

## 概述

自主生成模式是PRD工作台的默认工作模式。与交互模式（Phase 1-4逐步确认）不同，自主模式通过"先生成后审阅"的方式，在用户提供初始信息后一次性产出完整PRD，大幅提升效率。

**核心理念**：自主模式不是降低质量，而是将交互方式从"逐步阻塞式确认"变为"先生成后批量审阅"。

---

## 模式选择

### 默认行为

启动后通过模式确认门控（MC-01）让用户选择模式。详见 `SKILL.md` 中的"模式确认门控"章节。

### 模式触发关键词

| 模式 | 关键词 |
|------|--------|
| 轻量模式 | "轻量模式""快速出PRD""简版PRD""精简模式""简单需求""常规改动""小改动" |
| 交互模式 | "交互模式""逐步讨论""我想一步步来""详细讨论""切换到交互模式" |
| 自主模式 | "自主模式""快速出吧""直接帮我写""自主生成"或无特殊关键词 |

### 模式切换时机

| # | 时机 | 方向 | 触发方式 | AI提示语 |
|---|------|------|---------|---------|
| 1 | **模式确认门控** | →任意模式 | 用户在MC-01中选择 | 输出确认语后进入对应流程 |
| 2 | **Stage A 信息不足** | 自主→交互 | AI主动建议（硬性条件缺3+项） | "你提供的信息中，{缺失项}还不够明确。建议切换到交互模式逐步梳理，确保PRD质量。是否切换？" |
| 3 | **Stage C 审阅时** | 自主→交互（局部） | AI主动建议（某维度>50%假设需修改） | "关于{XX部分}有较多内容需要重新确认，建议对这部分切换到交互模式深入讨论，其余部分继续自主模式。" |
| 4 | **交互模式阶段边界** | 交互→自主 | 用户说"快速出吧""直接帮我写""自主生成" | "好的，基于已有的{Phase X}信息，我将自主生成剩余部分。生成后你可以集中审阅。" |
| 5 | **任何时候** | 双向 | 用户说"切换模式" | 自主→交互："好的，切换到交互模式。我们从{当前阶段对应的Phase}开始逐步确认。" / 交互→自主："好的，切换到自主模式。基于已收集的信息，我将一次性生成完整PRD。" |
| 6 | **任意阶段** | 任意→轻量 | 用户说"够了，出精简版就行""轻量模式" | "好的，基于已收集的信息，我将生成精简版PRD（7章）。" |
| 7 | **Stage L2 生成中** | 轻量→自主 | AI检测到复杂度超出（>3个待确认 / >5变更点 / 跨系统接口 / 数据迁移） | 使用AskUserQuestion建议升级（详见 `references/lite-mode.md` SL-02） |
| 8 | **Stage L3 审阅时** | 轻量→自主/交互 | 用户要求补充细节 / 连续2轮大改 | "好的，基于已有的轻量PRD内容，我将切换到{目标模式}补充完善。" |
| 9 | **自主/交互→轻量** | 任意→轻量 | 用户说"太细了""出精简版" | "好的，基于已收集的信息，我将生成精简版PRD。" |

### 切换时的状态保留

- **自主→交互**：Stage A 已产出的 intake.md 作为 Phase 1 的初始摘要，用户可在其基础上补充
- **交互→自主**：已完成的 Phase 产出文件直接作为 Stage B 输入，AI在此基础上补充推断并生成PRD
- **局部切换**（Stage C中）：仅对特定章节/维度进入交互讨论，讨论完毕后更新PRD，不影响其他章节
- **轻量→自主**：轻量PRD内容作为完整PRD的种子（第2章→Ch 2，第3章→Ch 4，第4章→Ch 6，第5章→Ch 5，第6章→Ch 9），未覆盖章节由自主模式补充生成
- **轻量→交互**：轻量PRD内容作为 Phase 1 摘要基础，用户进入 Phase 2 逐项澄清

#### 轻量→自主 Ch.4→Ch.6 转换示例

轻量模式第4章"功能说明"为简化格式，升级到自主模式时需重构为统一步骤-规则表。以下为最小化转换示例：

**Before（轻量模式第4章）**：
```markdown
### 变更点1：订单分配规则调整

**行为描述**：将手动选仓改为系统按距离自动分配

**业务规则**：
- 优先选择距离收货地最近且库存充足的仓库
- 如无可用仓库，通知运营人员
```

**After（自主模式 Ch.6 F-001）**：
```markdown
### F-001 自动分仓

**功能描述**：订单创建时系统自动按规则分配执行仓库
**操作角色**：系统自动  |  **前置条件**：订单已通过基础校验
**关联变更**：C-01

**操作步骤与规则**：

| 步骤 | 操作 | 业务规则 | 异常处理 |
|------|------|---------|---------|
| 1 | 系统获取收货地坐标 | BR-001: 地址必须含有效经纬度 | E-001: 地址无坐标→提示补充 |
| 2 | 计算各仓距离并排序 | BR-002: 距离最近+库存≥订单需求量 | E-002: 无可用仓→通知运营 |
| 3 | 锁定目标仓并分配 | BR-003: 分配后冻结对应库存 | E-003: 库存锁定失败→重试1次后通知 |
```
- **自主/交互→轻量**：基于已收集的全部信息，按轻量模式7章结构重新整理输出，已收集信息不丢失

## Stage A: 压缩录入

### 目标

用最少的对话轮次（1-2轮）获取足够生成PRD的信息。

### 操作流程

1. **分析用户输入**：用户可能一次性提供大段背景，也可能只说一句话
2. **读取知识库**：如存在 `knowledge-base/`，读取 `_index.md` 了解覆盖范围，读取相关知识卡片补充背景
3. **推断变更范围**（仅 `requirement_type = update|mixed`）：根据用户描述自动推断 `change_scope` 和 `unchanged_scope`。推断原则：
   - 用户明确提到的变更方面 → `change_scope`
   - 用户未提及但可能受影响的方面 → 保守归入 `change_scope`（宁可多写不可遗漏）
   - 用户明确说"不变""保持现有"的方面 → `unchanged_scope`
   - 完全不相关的方面 → `unchanged_scope`
   - 推断结果写入 intake.md §4.0，在 Stage C 审阅时供用户确认
4. **评估信息充分度**：按照下方标准评估
5. **补充提问**（如需）：最多追问1轮，聚焦缺失的硬性条件
6. **产出 intake.md**：使用 `templates/autonomous-intake-brief.md` 模板（含变更范围声明）

### 一次性结构化提问模板

当用户提供的信息不足以覆盖硬性条件时，采用混合提问方式：

**第一步**：对"涉及哪些系统/模块"使用 `AskUserQuestion`（multiSelect）：

> header: "系统模块"
> 问题: "本次需求涉及哪些系统模块？（可多选）"
> 选项：
> - **OMS订单系统**：订单管理、履约调度等
> - **WMS仓储系统**：入库、出库、库存管理等
> - **TMS运输系统**：运输计划、配送跟踪等
> - **BMS计费系统**：费用计算、账单结算等

**第二步**：对其余核心问题以自由文本方式提问：

> 请简要回答以下几个关键问题（可以简短回答，不需要长篇大论）：
>
> 1. **要解决什么问题？** 当前是怎么处理的？主要痛点是什么？
> 2. **谁来用？** 主要使用者是哪些角色？操作频率？
> 3. **期望什么效果？** 做完之后希望达到什么目标？
> 4. **有没有特别的约束或已知风险？**（可选）

### 信息充分度评估标准

**硬性条件**（4项，必须在进入Stage B前获得）：

| # | 条件 | 判断标准 |
|---|------|---------|
| 1 | 核心业务问题 | 用户明确说出了要解决什么问题、当前痛点 |
| 2 | 涉及系统模块 | 知道是哪些系统、哪些模块需要变更 |
| 3 | 主要用户角色 | 知道谁在用、在什么场景下用 |
| 4 | 期望目标 | 知道做完后期望达到什么效果（至少定性） |

**软性条件**（5项，缺失时AI可推断并标记）：

| # | 条件 | 缺失时的处理 |
|---|------|------------|
| 1 | 系统现状 | 基于知识库推断，标记 `[推断]` |
| 2 | 约束条件 | 不假设特殊约束，标记 `[待确认]` |
| 3 | 异常处理偏好 | 采用行业标准做法，标记 `[推断]` |
| 4 | 非功能性需求 | 采用通用标准值，标记 `[推断]` |
| 5 | 上下游影响 | 基于知识库推断，标记 `[推断]` 或 `[待确认]` |

**评估结果处理**：

| 硬性条件满足情况 | 处理方式 |
|----------------|---------|
| 4/4 全满足 | 直接进入 Stage B |
| 缺 1-2 项 | 使用 `AskUserQuestion` 让用户决定（见下方） |
| 缺 3+ 项 | 使用 `AskUserQuestion` 建议切换（见下方） |

**缺 1-2 项时**，使用 `AskUserQuestion`：

> header: "信息不足"
> 问题: "还缺少{缺失项}的信息，你希望？"
> 选项：
> - **继续生成**：基于已有信息和AI推断生成PRD，缺失部分标记[待确认]
> - **我来补充**：现在补充缺失信息（进入自由文本）
> - **切换交互模式**：切换到交互模式逐步梳理

**缺 3+ 项时**，使用 `AskUserQuestion`：

> header: "信息不足"
> 问题: "当前缺少{缺失项列表}等关键信息，建议切换到交互模式逐步梳理"
> 选项：
> - **切换交互模式**（推荐）：进入交互模式逐步确认，确保PRD质量
> - **继续自主（我立即补充）**：现在一次性补充所有缺失信息

### 知识库补充策略

1. 读取 `knowledge-base/_index.md` 确定覆盖范围
2. 根据用户提到的系统/模块，读取对应的知识卡片（如 `domain-oms.md`、`domain-wms.md`）
3. 读取 `glossary.yaml` 获取术语定义
4. 知识库中的信息视为"已确认事实"（无需标记），但如果知识库信息标注了 `[待确认]` 或 `[过时?]`，在PRD中保持该标注
5. 在 intake.md 的信息来源摘要中注明引用了哪些知识库文件

## Stage B: 自主生成

### 目标

一次性生成完整PRD及全部附属文件，全程无用户交互。

### 生成顺序

1. **整理信息** → 输出 `intake.md`（使用 `templates/autonomous-intake-brief.md`）
2. **推导假设** → 输出 `clarification.md`（AI基于信息缺口自主推导，标记假设依据）
3. **规划 PRD 大纲** → 输出大纲供用户确认（详见下方"PRD 大纲确认"），大纲中包含 ID 分配规划（含 BR-XXX 分段方案）
4. **撰写PRD** → 读取 `references/phase3-write.md` 和 `references/diagram-patterns.md`，严格按 ID 分配规划逐章撰写。当 `requirement_type = update|mixed` 时，只生成有实质内容的章节，未涉及变更的方面不独立成节，由PRD尾部一段话统一说明
5. **绘制流程图** → 保存到 `diagrams/` 目录
6. **一致性扫描** → 如 Python 可用，执行 `{python_cmd} scripts/check-prd-consistency.py` 验证 ID 交叉引用和模糊用语；发现关键问题先自动修复
7. **执行自检** → 读取 `references/phase4-review.md`，执行 CK-0 到 CK-8 全部检查
8. **生成报告** → 输出 `review-report.md`

### PRD 大纲确认

在推导假设完成后、正式撰写前，先输出 PRD 大纲（1页以内）供用户确认，避免生成后大幅返工：

```markdown
## PRD 大纲预览

### 功能点规划
- F-001: {功能名称} — {一句话描述}
- F-002: {功能名称} — {一句话描述}
- ...

### 业务规则分段规划
- F-001: BR-001 ~ BR-00X（{规则数}条）
- F-002: BR-0X1 ~ BR-0XX（{规则数}条）
- ...

### 接口规划
- IF-001: {接口名称}（{调用方}→{被调用方}）
- ...

### 图表规划
- 主流程泳道图（{参与角色}）→ YAML → draw.io
- {实体}状态流转图 → Mermaid
- 数据流向图 → Mermaid

### 各章节详略规划
| 章节 | 预计详略 | 说明 |
|------|---------|------|
| Ch.2 需求概述 | 标准 | — |
| Ch.5 业务流程 | 重点 | 涉及跨系统泳道 |
| Ch.6 功能明细 | 重点 | N个功能点 |
| Ch.7 接口集成 | 标准 | N个接口 |
| Ch.8 非功能性 | 简略 | 沿用现有标准 |
```

使用 `AskUserQuestion` 确认大纲：

> header: "PRD大纲"
> 问题: "以上是PRD的结构大纲，确认后我将开始撰写。"
> 选项：
> - **确认，开始撰写**：大纲无需调整
> - **需要调整**：进入自由文本，说明调整意见（增删功能点、调整详略等）

### 三级标记体系

#### 无标记（已确认事实）

内容来源于：
- 用户在对话中明确陈述的信息
- 知识库中已确认的信息（无质量标注的）
- 用户提供的文档中的事实性内容

#### `[推断]` — 专业推断

AI基于专业判断做出的推断，一个合格的供应链产品经理会做出相同判断。适用于：

- **行业标准做法**：接口超时重试策略、常见异常处理模式、标准非功能性要求
- **由事实推导的必然结论**：如用户说"需要实时库存"，推断需要并发控制
- **通用供应链业务模式**：标准的出入库流程、标准的单据流转规则
- **知识库中已有的标准流程**：直接引用并标注来源

格式：在相关内容末尾添加 `[推断]`

```markdown
系统应在接口调用超时（30秒）后自动重试，最多3次，超过后记录异常日志并触发告警。[推断]
```

#### `[待确认]` — 需用户确认

AI不确定的业务决策，存在多种合理选择。适用于：

- **业务边界决策**：某个场景是否在本次范围内
- **特定于用户系统的业务规则**：如审批流程、权限矩阵
- **多方案决策点**：存在两种以上合理方案，需要用户选择
- **与其他系统的集成细节**：接口协议、数据格式
- **权限和审批流程**：谁有权做什么操作
- **金额计算的精度和舍入**：不同业务场景可能不同

格式：使用引用块，包含假设和变更说明

```markdown
> [待确认] 当库存不足时，系统是否应自动触发采购申请？
> 当前假设：仅通知仓库主管，不自动触发采购流程。
> 如需变更此假设，请在审阅时指出。
```

#### 绝不假设的红线

以下信息**绝对不能假设**，必须在 Stage A 中从用户获得：

- 核心业务问题的定义（用户到底要解决什么）
- 系统模块范围（到底改哪些系统）
- 主要用户角色（到底谁在用）
- 基本期望目标（到底要达到什么效果）

如果这些信息在 Stage A 未获得，不得进入 Stage B。

### 质疑的文档化

在交互模式中，质疑通过对话进行。在自主模式中，质疑写入PRD文档：

**在PRD第2章"需求概述"§2.6，增加"需求合理性分析"段落**：

```markdown
### 2.6 需求合理性分析

以下是AI基于专业判断对需求的审视，供用户参考：

1. **{质疑点1}**：{描述} → {建议}
2. **{质疑点2}**：{描述} → {建议}
```

质疑内容不使用 `[建议]` 标记（那是交互模式的标记），而是作为PRD的正式分析章节。

### 知识库引用规范

引用知识库内容时，标注来源：

```markdown
根据已有业务知识，当前出库流程为"拣货→复核→打包→发运"四步流程。
（参考: knowledge-base/domain-wms.md §核心流程）
```

### clarification.md 的自主生成

在自主模式中，`clarification.md` 由AI自动生成。使用 `templates/clarification-template.md` 作为模板基础，设置 `mode: autonomous`、`status: ai-drafted`，按 phase2-clarify.md 中的维度结构逐维度填充 AI 的判断和依据。

## Stage C: 审阅迭代

### 目标

将完整PRD交给用户审阅，通过批量反馈高效迭代。

### 核心交互原则

**静默确认机制**：`[推断]` 项（基于行业标准的专业推断）在呈现后，若用户未在当轮对话中提出异议，自动视为确认。呈现假设总览表时必须明确告知此规则。这一机制大幅减少用户需要逐条确认的数量，将用户注意力聚焦在真正需要决策的 `[待确认]` 项上。

### 呈现顺序

Stage B 完成后，按以下顺序向用户呈现：

**第一步：假设总览表（分层呈现）**

假设按影响程度分层呈现，帮助用户聚焦关键决策，降低审阅认知负荷：

**第一层：高影响假设**（≤10个，优先呈现）

```markdown
## 假设总览 — 高影响项（需重点关注）

以下假设如果判断有误，将直接影响核心业务逻辑或数据安全。请逐一审阅。

| # | 类型 | 章节 | 假设内容 | 若判断有误的影响 | 状态 |
|---|------|------|---------|----------------|------|
| 1 | [待确认] | 5.2 | 库存不足时仅通知，不自动采购 | 可能导致缺货无法自动补货，影响履约 | 待审阅 |
| 2 | [待确认] | 6 F-002 | 审批金额阈值为5万元 | 阈值过高/过低直接影响风控 | 待审阅 |
```

**第二层：中低影响假设**（高影响项确认后再呈现）

```markdown
## 假设总览 — 中低影响项

以下假设基于行业标准或常见做法，如无异议将视为确认。

| # | 类型 | 章节 | 假设内容 | 若判断有误的影响 | 状态 |
|---|------|------|---------|----------------|------|
| 3 | [推断] | 6 F-003 | 接口超时重试3次后转人工 | 重试策略调整，影响可控 | 如无异议视为确认 |
| 4 | [推断] | 8 | 页面响应时间目标≤2秒 | 性能指标微调，不影响功能 | 行业标准值 |
```

**呈现规则**：
- 每个假设附加"若判断有误的影响"一句话说明，帮助用户快速判断是否需要关注
- `[待确认]` 按影响程度排序，高影响在前
- `[推断]` 按影响程度排序，高影响归入第一层

**第二步：选择审阅方式**

呈现假设总览表后，使用 `AskUserQuestion` 选择审阅方式（SC-01）：

> header: "审阅方式"
> 问题: "假设总览表中共有 X 项待确认、Y 项推断，你希望如何审阅？"
> 选项：
> - **逐条审阅**：逐条查看每个假设并确认
> - **挑选需改项**：默认全部确认，勾选需要修改的项（进入 multiSelect）
> - **全部确认**：所有假设无异议，全部确认

**选择"挑选需改项"时**，使用 `AskUserQuestion`（multiSelect），将所有假设列为选项，用户勾选需要修改的项：

> header: "选择需修改的假设"
> 问题: "以下假设默认全部确认。请勾选需要修改的项："
> 选项（multiSelect）：
> - **#{编号} {假设内容摘要}**：{所在章节}
> - ...（每个假设一个选项）

用户勾选后，仅对选中的假设逐条进入修改流程（SC-02/SC-03），其余视为确认。

**第三步：自检报告摘要**

简要呈现 CK-1 到 CK-8 的检查结果摘要。

**第四步：完整PRD**

提供完整PRD文档。用户可选择先看目录结构，再逐章阅读。

### 用户反馈处理规则

#### 逐条假设审阅（SC-02、SC-03）

选择"逐条审阅"时，对每个假设使用 `AskUserQuestion`：

**`[待确认]` 假设**（SC-02）：
> header: "假设确认"
> 问题: "关于此假设：{假设内容摘要}"
> 选项：
> - **确认**：假设正确，移除标记写入正文
> - **需修改**：进入自由文本，用户提供正确信息
> - **不确定（保持标记）**：暂时保留标记，后续再确认

**`[推断]` 假设**（SC-03）：
> header: "推断确认"
> 问题: "关于此推断：{推断内容摘要}"
> 选项：
> - **没问题**：推断合理，移除标记
> - **需修改**：进入自由文本，用户说明修正内容

#### 审阅后整体反馈（SC-04）

假设审阅完成后，使用 `AskUserQuestion` 收集整体反馈：

> header: "后续操作"
> 问题: "假设审阅完成，还需要什么调整？"
> 选项：
> - **章节细化**：指定某章节需要补充更多细节
> - **新增需求**：有PRD中未覆盖的新需求点
> - **整体满意**：PRD整体满意，准备最终交付

#### 其他反馈处理

| 反馈类型 | 处理方式 |
|---------|---------|
| **局部深入讨论** | 某个点需要多轮对话 → 对该维度进入交互讨论，讨论完更新PRD |
| **全局反馈** | "整体OK但第6章需要细化" → 对第6章进行补充细化 |
| **新增需求** | 用户提出PRD中没有的需求 → 补充到对应章节，标记为新增 |

### [推断] 标记的默认确认规则

- 用户审阅假设总览表后，未对某个 `[推断]` 项提出异议的，视为默认确认
- AI在第一轮呈现时告知此规则："标记为 [推断] 的项目，如您未提出异议，将视为确认"
- 用户可以在任何时候重新提出对已确认项的修改

### 变更摘要输出

每轮修改后、呈现更新的 PRD 之前，先输出变更摘要：

```markdown
## 本轮变更
- Ch.6 F-002: 修改了步骤2的业务规则（BR-003），原"人工审批"改为"系统自动审批"
- Ch.9 #3: 新增验收标准"自动审批在5秒内完成"
- Ch.2 §2.4 G-01: 目标值从"10分钟"调整为"5分钟"
```

### 迭代效率

- 每次修改后，仅对受影响的章节重新执行自检（不重跑全部CK检查）
- 假设确认后，批量移除标记（一次操作而非逐个移除）
- 支持"先确认大方向，再细化"的渐进式审阅

### 从自主模式回退到交互模式（SC-05）

当以下情况出现时，AI应使用 `AskUserQuestion` 主动建议局部或全局回退：

- **某一维度 >50% 的假设需要修改**
- **用户连续3次修改假设后仍不满意**
- **用户主动要求**：立即切换

使用 `AskUserQuestion`：

> header: "模式建议"
> 问题: "关于{维度名称}有较多内容需重新确认，建议切换到交互模式深入讨论"
> 选项：
> - **局部交互**：仅该维度切换到交互模式，其余保持自主
> - **全局交互**：整体切换到交互模式
> - **继续自主**：我补充信息后继续自主模式

### 最终交付（SC-06）

所有假设确认、自检通过后，使用 `AskUserQuestion` 并发确认最终交付选项（2个问题同时提出）：

> 问题1（header: "交付格式"）: "PRD最终输出格式？" → **Markdown + Word**（同时生成两种格式） / **仅Markdown**（仅保留Markdown格式）
> 问题2（header: "知识库"）: "是否有需要补充到知识库的新业务信息？" → **有，需要补充**（列出待补充内容） / **没有**（无需补充）

确认后执行：
1. 更新PRD版本号（移除所有标记后的版本）
2. 按选择的格式生成文件
3. 将所有产出文件列表呈现给用户
4. 如用户选择补充知识库，引导使用 scm-knowledge-curator 技能
