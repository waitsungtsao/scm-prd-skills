# 自主模式 — 操作指引

## 概述

**核心理念**：自主模式是本技能的标准工作模式。核心交互方式是"先生成后批量审阅"——AI 自适应深度地收集信息，一次性生成完整 PRD，用户集中审阅修改。

---

## 模式选择

### 默认行为

启动后通过模式确认门控（MC-01）让用户选择模式。详见 `SKILL.md` 中的"模式确认门控"章节。

### 模式触发关键词

详见 `SKILL.md` "模式触发关键词"章节。

### 模式切换时机

| # | 时机 | 方向 | 触发方式 | AI提示语 |
|---|------|------|---------|---------|
| 1 | **模式确认门控** | →任意模式 | 用户在MC-01中选择 | — |
| 2 | **任意阶段** | 自主→轻量 | 用户说"够了，出精简版就行""轻量模式" | "好的，基于已收集的信息，我将生成精简版PRD（7章）。" |
| 3 | **Stage L2 生成中** | 轻量→自主 | AI检测到复杂度超出（>3个待确认 / >5变更点 / 跨系统接口 / 数据迁移） | 使用AskUserQuestion建议升级（详见 `references/lite-mode.md` SL-02） |
| 4 | **Stage L3 审阅时** | 轻量→自主 | 用户要求补充细节 / 连续2轮大改 | "好的，基于已有的轻量PRD内容，我将切换到自主模式补充完善。" |

### 切换时的状态保留

- **局部深入**（Stage C中）：对特定章节/维度进行多轮讨论，讨论完毕后更新PRD，不影响其他章节
- **轻量→自主**：轻量PRD内容作为完整PRD的种子（第2章→Ch 2，第3章→Ch 4，第4章→Ch 6，第5章→Ch 5，第6章→Ch 9），未覆盖章节由自主模式补充生成

#### 轻量→自主 转换规则

AI 静默完成格式转换，用户只看到最终的完整版 PRD。**不向用户暴露"重构"过程**。

| 轻量章节 | 完整PRD章节 | AI 内部转换方式 |
|---------|-----------|----------------|
| 第2章 需求概述 | → Ch.2 | 直接复用，补充 §2.2 和 §2.6 |
| 第3章 业务变更点 | → Ch.4 | 直接复用 |
| 第4章 功能说明 | → Ch.6 | 扩展为步骤-规则表格式（补充步骤拆分和异常处理） |
| 第5章 流程与交互 | → Ch.5 | 直接复用，可选升级为泳道图 |
| 第6章 验收标准 | → Ch.9 | 直接复用，增加 G/F 关联列 |
| （无对应） | Ch.3/7/8/10 | 由自主模式补充生成 |

**面向用户的提示语**（简洁，不提"重构"）：
> "需求比预想复杂一些，我帮你补充了更详细的完整版本，请审阅。"

- **自主→轻量**：基于已收集的全部信息，按轻量模式7章结构重新整理输出，已收集信息不丢失

## Stage A: 需求录入（自适应深度）

**进度播报**：进入本阶段时输出 `▰▱▱▱▱ [1/{total}] 需求录入`。本阶段内的 AskUserQuestion header 使用 `"1/{total} 需求录入"`。

### 目标

像好的产品经理或咨询师一样，通过**鉴别诊断式**的信息收集，建立对需求的自洽理解。不是按清单收集信息，而是：形成假设 → 识别假设可能错误的关键鉴别点 → 定向探测 → 收敛。

收敛标准不是"清单填完了"，而是**"我的理解足够自洽，且主要的替代解释已被排除或标记为 `[待确认]`"**。

### 操作流程

1. **接收用户输入**：用户可能一次性提供大段背景，也可能只说一句话
2. **读取知识库**：如存在 `knowledge-base/`，读取 `_index.md` 了解覆盖范围，读取相关知识卡片补充背景
3. **形成初步假设**：基于用户输入 + 知识库，在内心构建一个对需求的完整假设（问题是什么、方案方向是什么、影响范围是什么）
4. **鉴别诊断**：识别假设中最脆弱的环节——哪些地方如果理解错了，会导致 PRD 方向性偏差？这些就是需要定向探测的鉴别点
5. **自适应深度收集**：根据假设的自洽程度决定收集深度（见下方）
6. **推断变更范围**（仅 `requirement_type = update|mixed`）：根据用户描述自动推断 `change_scope` 和 `unchanged_scope`
7. **产出 intake.md**：使用 `templates/autonomous-intake-brief.md` 模板

### 自适应深度

AI 在内部评估理解的自洽程度，自动选择收集路径。**不暴露评估过程给用户**，用户感知到的只是自然的对话。

#### 快速路径（1-2 轮）

**触发条件**：用户提供了充分的背景材料（文档、详细描述、manuscript），AI 的假设高度自洽，关键鉴别点很少。

**流程**：
1. 确认理解："我的理解是 {一段话复述需求本质}。"
2. 仅针对 1-2 个关键鉴别点提问（如"这个需求是改造现有流程还是新建？""库存不足时你们现在是怎么处理的？"）
3. 产出 intake.md

#### 定向补充路径（2-4 轮）

**触发条件**：用户描述有明确方向，但存在 2-3 个关键鉴别点（如：系统范围不确定、核心规则有多种可能的理解方式）。

**流程**：
1. 呈现已理解的部分："以下是我已经理解的：{要点列表}"
2. 针对鉴别点生成 3-5 个问题（优先问最可能导致方向性偏差的）
3. 对系统模块使用 `AskUserQuestion`（multiSelect），其余用自由文本
4. 产出 intake.md

#### 深度萃取路径（3-8 轮）

**触发条件**：用户的需求方向本身不清晰（"感觉XX有问题""想优化一下流程"），或存在多个相互关联的鉴别点。

**流程**：

**R1 — 邀请信息倾倒**：
> "请把你知道的关于这个需求的所有信息告诉我——背景、痛点、初步想法、相关文档、之前的讨论结论都可以。格式不限，我来整理。"

**R2 — 理解假设 + 鉴别问题**：
AI 消化全部输入后，输出：
```markdown
## 我的理解
{2-3 段复述需求本质，用自己的话}

## 我认为已经明确的
- ✅ {要点 1}
- ✅ {要点 2}

## 关键鉴别点（需要你帮我确认）
{5-8 个问题，每个问题附带 AI 当前的假设和替代可能}
1. {问题} — 我倾向于认为 {A}，但如果实际是 {B}，方案会完全不同
2. ...
```

**R3-N — 迭代深入**：
- 每轮更新理解摘要 + 新的鉴别问题
- 在对话中自然编织**合理性质疑**：
  - "你描述的这个问题，根因是 X 还是 Y？如果是 Y，也许有更简单的解决方式"
  - "这个方案的 ROI 如何？有没有考虑过 {更轻量的替代方案}？"
- 运用**已知背景推断**（反转知识诅咒）：
  - "我认为你的团队已经了解这些背景：{列表}。有不对的请指出。"
  - 用户只需否定不对的，不需要主动描述已知的

**收敛**：当 AI 判断理解足够自洽时：
> "我对需求的理解已经比较清晰了。{一句话总结}。还有 {N} 个鉴别点尚未完全确认，我会在 PRD 中标注 `[待确认]`，你审阅时重点看这些。我开始写？"

### 鉴别诊断的内部参考框架

以下 9 个维度是 AI **内部使用**的 gap 检测框架（来源于供应链领域最常见的需求盲区），用于确保鉴别诊断的全面性。**不逐项暴露给用户，不作为清单逐项提问**——仅在发现某维度存在可能导致方向性偏差的鉴别点时，才针对性地探测该维度。

1. **业务边界** — 哪些场景在范围内/外？需要留扩展口吗？
2. **数据流** — 数据从哪来、怎么处理、到哪去？时效性？量级？
3. **系统集成** — 现有接口变更？新增接口？触发时机？失败处理？
4. **异常处理** — 每步的失败场景？并发冲突？降级策略？
5. **状态流转** — 涉及哪些实体状态？触发条件？不可逆转换？
6. **权限控制** — 角色操作映射？审批流程？数据可见范围？
7. **非功能性** — 性能要求？并发量？安全？兼容性？
8. **风险** — 上线后可能出什么问题？外部依赖的信心？
9. **数据迁移** — 历史数据怎么处理？能否停机？（仅数据结构变更时）

### 一次性结构化提问模板

当 AI 判断需要一次性补充多项信息时（定向补充路径），采用混合提问方式：

**第一步**：对"涉及哪些系统/模块"使用 `AskUserQuestion`（multiSelect）：

> header: "系统模块"
> 问题: "本次需求涉及哪些系统模块？（可多选）"
> 选项：
> - **OMS订单系统**：订单管理、履约调度等
> - **WMS仓储系统**：入库、出库、库存管理等
> - **TMS运输系统**：运输计划、配送跟踪等
> - **BMS计费系统**：费用计算、账单结算等

**第二步**：对鉴别点以自由文本方式提问（只问 AI 认为最关键的 2-4 个）。

### 已知背景推断

在 intake.md 产出后、进入 Stage B 前，基于 knowledge-base、system-conventions、用户描述，推断一组"团队已知背景"，写入 intake.md 的 `known_context` 字段，标记 `[推断]`：

```yaml
known_context:   # [推断] AI 判断为团队共识，PRD 中引用但不展开
  - "现有 OMS 审批流程（domain-oms.md §3 流程部分）"
  - "字段命名规范（system-conventions.md §2.3）"
  - "当前退货流程 AS-IS 状态（用户在录入中已充分描述）"
```

**判断标准**：所有目标读者（撰写者 + 开发 + QA + BA + 决策者）都已知的才列入。作者知道但开发可能不知道的，不列入（PRD 仍需描述）。

Stage C 审阅时，用户可在假设总览表中修改 `known_context`（如"这个其实开发不太了解，需要展开"）。

### 知识库补充策略

1. 读取 `knowledge-base/_index.md` 确定覆盖范围
2. 根据用户提到的系统/模块，读取对应的知识卡片（如 `domain-oms.md`、`domain-wms.md`）
3. 读取 `glossary.yaml` 获取术语定义
4. 知识库中的信息视为"已确认事实"（无需标记），但如果知识库信息标注了 `[待确认]` 或 `[过时?]`，在PRD中保持该标注
5. 在 intake.md 的信息来源摘要中注明引用了哪些知识库文件

## Stage B: 自主生成

**进度播报**：叙事规划阶段输出 `▰▰▱▱▱ [2/{total}] 叙事规划`；PRD 撰写开始时输出 `▰▰▰▱▱ [3/{total}] PRD 撰写`。CD-01 确认后如原型选项变化导致 total 变化，**重新输出路线图**。

### 目标

一次性生成完整PRD及全部附属文件，全程无用户交互。

### 生成顺序

1. **整理信息** → 输出 `intake.md`（使用 `templates/autonomous-intake-brief.md`）
2. **推导假设** → 输出 `clarification.md`（AI基于信息缺口自主推导，标记假设依据）
3. **叙事规划 + PRD 大纲** → 输出叙事规划和大纲供用户确认（详见下方"PRD 大纲确认"）
4. **撰写PRD** → 读取 `references/writing-guide.md` 和 `references/diagram-patterns.md`，按叙事规划的详略策略逐章撰写。叙事规划中标为"省略"的维度不生成独立章节，标为"精简"的维度简练描述，标为"重点"的维度充分展开。在需要高亮关键信息处使用提示块（`[!INFO]`/`[!CAUTION]`/`[!WARNING]`/`[!TIP]`），`[推断]`和`[待确认]`标记在 Word 输出时自动渲染为对应提示块样式
5. **记录决策** → 撰写过程中遇到多方案选择时，同步写入 `decision-log.md`（使用 `templates/decision-log-template.md`）。当 `[推断]` 涉及多方案选择时，应同时创建决策记录
6. **绘制流程图** → 保存到 `diagrams/` 目录。`python_available = true` 时**默认生成** `.diagram.yaml` + `.drawio` + `.mermaid`；`python_available = false` 时按 `references/diagram-patterns.md` "draw.io 生成降级策略 DG-01" 处理（引导安装依赖或降级，不阻断流程）
7. **导出图表** → 调用 `scripts/export-diagrams.py` 统一处理 .drawio 生成 + .svg/.png 导出（.drawio 生成失败不阻断 PNG 导出）
8. **生成 Word** → `docx_engine = "js"` 时调用 `NODE_PATH=$(npm root -g) node scripts/md2docx.mjs`，`null` 时跳过并提示 `npm install -g docx`
9. **一致性扫描** → 如 Python 可用，执行 `{python_cmd} scripts/check-prd-consistency.py` 验证 ID 交叉引用和模糊用语；发现关键问题先自动修复
10. **执行自检** → 读取 `references/review-guide.md`，执行 CK-0 到 CK-9 全部检查
11. **生成报告** → 输出 `review-report.md`

### PRD 大纲确认（叙事规划）

在推导假设完成后、正式撰写前，先输出叙事规划 + PRD 大纲（1页以内）供用户确认，避免生成后大幅返工。**大纲的核心是表达策略，不是盘点数量。**

```markdown

### 核心叙事
{一段话概括：什么问题 → 怎么解决 → 预期价值}

### 表达维度与详略
| 维度 | 展开？ | 详略 | 理由 |
|------|-------|------|------|
| 业务流程 | 是 | 重点 | {为什么需要重点展开} |
| 功能规则 | 是 | 标准 | {概述} |
| 接口数据 | 简要 | 精简 | {为什么可以精简} |
| 非功能性 | 否 | — | {为什么可以省略} |

### 重点（读者必须理解的 2-3 件事）

### 功能点清单

- F-002: {功能名称} — {一句话描述}

- IF-001: {接口名称}（{调用方}→{被调用方}）

### 图表策略
- {图1}: {回答什么问题} → {格式}

- 第1章 文档信息 → 第2章 需求概述 → 第3章 {下一个保留章节} → ...

### 原型策略

> 这是原型的**早期思考**——PRD 尚未撰写，此处仅对齐"核心交互意图"。
> 如果屏幕清单或体验路径与你对需求的理解有偏差，现在是最好的纠偏时机。

| 屏幕 | 类型 | 来源 | 要表达什么 |

**不做什么**：{明确排除的屏幕/功能}

### ID 使用策略
- 第一层（始终）：G-XX、F-XXX

```

> **叙事规划自检**：大纲输出前，执行 NP-01 ~ NP-05 检查（规则详见 `writing-guide.md` "叙事规划自检"章节），静默修正后再呈现给用户。

使用 `AskUserQuestion`（multiSelect）强制章节详略选择（CD-01）：

> header: "章节详略选择"
> markdown: （展示上方叙事规划 + 大纲内容）
> 问题: "请选择需要**重点展开**的章节（可多选）。未选中的章节将按标准/精简描述或省略。"
> multiSelect:
> - **业务流程（Ch.5）**：完整流程图 + 步骤详解
> - **功能与规则（Ch.6）**：步骤-规则表逐项展开，含完整异常处理
> - **接口与数据（Ch.7）**：接口规格 + 字段级定义
> - **非功能性需求（Ch.8）**：性能/安全/可用性量化描述
> - **原型演示（PT-01）**：为关键交互生成可点击原型（AI 建议：{是/否}，理由：{触发条件}）

选择结果处理：选中→"重点"展开，未选中+相关→"标准/精简"，未选中+无关→省略（详见 `writing-guide.md` CD-01 说明）。原型演示选中→PRD 审阅后进入原型输出阶段。AI 据此更新叙事规划的详略表格后开始撰写。

### 三级标记体系

#### 无标记（已确认事实）

内容来源于：
- 用户在对话中明确陈述的信息
- 知识库中已确认的信息（无质量标注的）
- 用户提供的文档中的事实性内容

#### `[推断]` — 专业推断

AI基于专业判断做出的推断，一个合格的供应链产品经理会做出相同判断。适用于行业标准做法、由已知事实推导的结论、通用供应链模式、知识库已有内容。

格式：在相关内容末尾添加 `[推断]`

```markdown
系统应在接口调用超时（30秒）后自动重试，最多3次，超过后记录异常日志并触发告警。[推断]
```

#### `[待确认]` — 需用户确认

AI不确定的业务决策，存在多种合理选择。适用于业务边界决策、特定于用户系统的规则（审批流程、权限矩阵）、多方案决策点、系统集成细节、金额精度等。

格式：使用引用块，包含假设内容

```markdown
> [待确认] 当库存不足时，系统是否应自动触发采购申请？
> 当前假设：仅通知仓库主管，不自动触发采购流程。
```

#### 绝不假设的红线

以下信息必须在 Stage A 中从用户获得，不可假设：

- 核心业务问题的定义（用户到底要解决什么）
- 系统模块范围（到底改哪些系统）
- 主要用户角色（到底谁在用）

## Stage C: 审阅迭代

**进度播报**：进入本阶段时输出 `▰▰▰▰▱ [4/{total}] 审阅迭代`。多轮迭代期间步骤编号不变。

### 目标

将完整PRD交给用户审阅，通过批量反馈高效迭代。

### 核心交互原则

**静默确认机制**：`[推断]` 项在呈现后，若用户未在当轮对话中提出异议，自动视为确认。用户注意力聚焦在真正需要决策的 `[待确认]` 项上。

### 呈现顺序

Stage B 完成后，按以下顺序向用户呈现：

**第一步：假设总览表（分层呈现）**

假设按影响程度分层呈现。

**影响程度判断标准**：

| 等级 | 判断标准 | 典型场景 |
|------|---------|---------|
| 高 | 影响核心业务逻辑、数据安全或资金准确性，或涉及≥2个下游系统 | 库存扣减规则、金额计算精度、审批阈值、跨系统接口协议 |
| 中 | 影响次要流程、操作效率或单系统内部行为 | 异常处理策略、通知方式、日志级别、批量操作上限 |
| 低 | 仅影响展示/体验层面，不影响业务正确性 | 字段排序、默认排序方向、提示文案、页面布局细节 |

**第一层：高影响假设**（≤10个，优先呈现，**全部需显式确认**）

**重要**：高影响项无论类型（`[待确认]` 或 `[推断]`）均需用户显式确认，不适用静默确认机制。这避免了用户因"没仔细看就跳过"而让高风险假设生效。

```markdown
## 假设总览 — 高影响项（需逐一确认）

以下假设如果判断有误，将直接影响核心业务逻辑或数据安全。请逐一审阅。

| # | 类型 | 章节 | 假设内容 | 若判断有误的影响 | 状态 |
|---|------|------|---------|----------------|------|
| 1 | [待确认] | 5.2 | 库存不足时仅通知，不自动采购 | 可能导致缺货无法自动补货，影响履约 | 待审阅 |
| 2 | [推断] | 6 F-002 | 审批金额阈值为5万元 | 阈值过高/过低直接影响风控 | 待审阅 |
```

**第二层：中低影响假设**（高影响项确认后再呈现）

```markdown
## 假设总览 — 中低影响项

以下假设基于行业标准或常见做法，如无异议将视为确认。

| # | 类型 | 章节 | 假设内容 | 若判断有误的影响 | 状态 |
|---|------|------|---------|----------------|------|
| 3 | [推断] | 6 F-003 | 接口超时重试3次后转人工 | 重试策略调整，影响可控 | 如无异议视为确认 |
| 4 | [推断] | 8 | 页面响应时间目标≤2秒 | 性能指标微调，不影响功能 | 行业标准值 |
```

**呈现规则**：
- 每个假设附加"若判断有误的影响"一句话说明，帮助用户快速判断是否需要关注
- `[待确认]` 按影响程度排序，高影响在前
- `[推断]` 按影响程度排序，高影响归入第一层

**第二步：选择审阅方式**

呈现假设总览表后，使用 `AskUserQuestion` 选择审阅方式（SC-01）：

> header: "审阅方式"
> 问题: "假设总览表中共有 X 项待确认、Y 项推断，你希望如何审阅？"
> 选项：
> - **逐条审阅**
> - **挑选需改项**：默认全部确认，勾选需要修改的项
> - **全部确认**

**选择"挑选需改项"时**，使用 `AskUserQuestion`（multiSelect），将所有假设列为选项，用户勾选需要修改的项：

> header: "选择需修改的假设"
> 问题: "以下假设默认全部确认。请勾选需要修改的项："
> 选项（multiSelect）：
> - **#{编号} {假设内容摘要}**：{所在章节}
> - ...（每个假设一个选项）

用户勾选后，仅对选中的假设逐条进入修改流程（SC-02/SC-03），其余视为确认。

**第三步：自检报告摘要**

简要呈现 CK-0 到 CK-9 的检查结果摘要。

**第四步：完整PRD**

提供完整PRD文档。用户可选择先看目录结构，再逐章阅读。

### 用户反馈处理规则

#### 逐条假设审阅（SC-02、SC-03）

选择"逐条审阅"时，对每个假设使用 `AskUserQuestion`：

**`[待确认]` 假设**（SC-02）：
> header: "假设确认"
> 问题: "关于此假设：{假设内容摘要}"
> 选项：
> - **确认**：移除标记写入正文
> - **需修改**：用户提供正确信息
> - **不确定**：保留标记，后续再确认

**`[推断]` 假设**（SC-03）：
> header: "推断确认"
> 问题: "关于此推断：{推断内容摘要}"
> 选项：
> - **没问题**：移除标记
> - **需修改**：用户说明修正内容

#### 审阅后整体反馈（SC-04）

假设审阅完成后，使用 `AskUserQuestion` 收集整体反馈：

> header: "后续操作"
> 问题: "假设审阅完成，还需要什么调整？"
> 选项：
> - **章节细化**：指定需补充的章节
> - **新增需求**：补充PRD未覆盖的需求
> - **整体满意**：准备最终交付

#### 其他反馈处理

| 反馈类型 | 处理方式 |
|---------|---------|
| **局部深入讨论** | 某个点需要多轮对话 → 对该维度进入交互讨论，讨论完更新PRD |
| **全局反馈** | "整体OK但第6章需要细化" → 对第6章进行补充细化 |
| **新增需求** | 用户提出PRD中没有的需求 → 补充到对应章节，标记为新增 |

### [推断] 标记的默认确认规则

- 用户审阅假设总览表后，未对某个 `[推断]` 项提出异议的，视为默认确认
- AI在第一轮呈现时告知此规则："标记为 [推断] 的项目，如您未提出异议，将视为确认"
- 用户可以在任何时候重新提出对已确认项的修改

### 变更摘要输出与持久化

每轮修改后、呈现更新的 PRD 之前，先输出变更摘要并持久化到 `prd-changelog.md`（使用 `templates/prd-changelog-template.md`）。首次修改时创建文件，后续追加 RN 条目。

```markdown
## 本轮变更 (R{N})
- Ch.6 F-002: 修改了步骤2的业务规则，原"人工审批"改为"系统自动审批"
- Ch.9 #3: 新增验收标准"自动审批在5秒内完成"
- Ch.2 §2.4 G-01: 目标值从"10分钟"调整为"5分钟"

### 影响传播
- F-002 规则修改 → Ch.5 流程图已同步更新
- G-01 修改 → Ch.9 验收标准#1 已同步更新
```

### 迭代效率

- 每次修改后，追加变更记录到 prd-changelog.md（含影响传播分析）
- 每次修改后，仅对受影响的章节重新执行自检（不重跑全部CK检查）
- 假设确认后，批量移除标记（一次操作而非逐个移除）
- 支持"先确认大方向，再细化"的渐进式审阅

### 局部深入讨论（SC-05）

当以下情况出现时，AI 应主动建议对特定维度进行多轮深入讨论：

- **某一维度 >50% 的假设需要修改** — 说明 AI 对该领域的理解存在系统性偏差
- **用户连续3次修改假设后仍不满意** — 需要更深入的上下文交换

使用 `AskUserQuestion`：

> header: "深入讨论"
> 问题: "关于{维度名称}有较多内容需重新确认，建议逐项深入讨论"
> 选项：
> - **逐项讨论**：对该维度的假设逐条确认和修正
> - **我来补充**：我一次性说明这部分的背景，你重新生成
> - **先跳过**：先处理其他部分，回头再来

### 最终交付（SC-06）

交付流程的完整定义（交付选项、Word 生成、交付精要、知识库导入）见 `references/review-guide.md` "最终交付"章节。自主模式额外增加一步：

7. 如果叙事规划的 CD-01 中用户选中了"原型演示"，进入 Stage D
8. **更新约束索引**：执行 `references/review-guide.md` "约束索引提取"中定义的提取流程

## Stage D: 原型设计与生成（可选）

**进度播报**：原型设计阶段输出 `▰▰▰▰▰▱▱ [5/7] 原型设计`；原型生成阶段输出 `▰▰▰▰▰▰▱ [6/7] 原型生成`。PT-02 多轮迭代期间步骤编号不变（始终是"原型设计"步）。

### 前置条件

- CD-01 中用户选中了"原型演示（PT-01）"
- Stage C 审阅迭代已完成，PRD 内容已确认

### 目标

基于已确认的 PRD，与用户共同打磨原型设计方案，然后生成可交互原型。

### 操作流程

1. **读取参考文档**：读取 `references/prototype-planning.md` 获取详细操作指引
2. **生成设计方案初稿**：使用 `templates/prototype-design-template.md` 模板，从 PRD 提取状态映射、数据契约、交互流、校验规则，补充组件结构和设计决策，生成 `prototype-design.md`
3. **设计方案迭代**（PT-02）：呈现设计方案，用户审阅并反馈。可多轮迭代，每轮更新 `prototype-design.md` 并追加迭代记录
4. **CK-PT 检查**：设计方案确定后，验证与 PRD 的一致性
5. **生成原型**：调用 web-artifacts-builder skill 生成 React 代码并打包
6. **呈现与微调**（PT-03）：用户在浏览器中体验原型，可选择回到设计方案调整或直接微调代码
