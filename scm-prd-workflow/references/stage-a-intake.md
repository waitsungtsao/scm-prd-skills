# Stage A: 需求录入（自适应深度）

读取时机：进入 Stage A 时。Stage A 完成后可释放。

**进度播报**：进入本阶段时输出 `▰▱▱▱▱ [1/{total}] 需求录入`。本阶段内的 AskUserQuestion header 使用 `"1/{total} 需求录入"`。

## 目标

像好的产品经理或咨询师一样，通过**鉴别诊断式**的信息收集，建立对需求的自洽理解。不是按清单收集信息，而是：形成假设 → 识别假设可能错误的关键鉴别点 → 定向探测 → 收敛。

收敛标准不是"清单填完了"，而是**"我的理解足够自洽，且主要的替代解释已被排除或标记为 `[待确认]`"**。

## 操作流程

1. **接收用户输入**：用户可能一次性提供大段背景，也可能只说一句话
2. **读取知识库**：如存在 `knowledge-base/`，读取 `_index.md` 了解覆盖范围，读取相关知识卡片补充背景
3. **形成初步假设**：基于用户输入 + 知识库，在内心构建一个对需求的完整假设（问题是什么、方案方向是什么、影响范围是什么）
4. **鉴别诊断**：识别假设中最脆弱的环节——哪些地方如果理解错了，会导致 PRD 方向性偏差？这些就是需要定向探测的鉴别点
5. **自适应深度收集**：根据假设的自洽程度决定收集深度（见下方）
6. **推断变更范围**（仅 `requirement_type = update|mixed`）：根据用户描述自动推断 `change_scope` 和 `unchanged_scope`
7. **产出 intake.md**：使用 `templates/autonomous-intake-brief.md` 模板

## 自适应深度

AI 在内部评估理解的自洽程度，自动选择收集路径。**不暴露评估过程给用户**，用户感知到的只是自然的对话。

### 快速路径（1-2 轮）

**触发条件**：用户提供了充分的背景材料，AI 的假设高度自洽，关键鉴别点很少。

**流程**：
1. 确认理解："我的理解是 {一段话复述需求本质}。"
2. 仅针对 1-2 个关键鉴别点提问
3. 产出 intake.md

### 定向补充路径（2-4 轮）

**触发条件**：用户描述有明确方向，但存在 2-3 个关键鉴别点。

**流程**：
1. 呈现已理解的部分："以下是我已经理解的：{要点列表}"
2. 针对鉴别点生成 3-5 个问题（优先问最可能导致方向性偏差的）
3. 对系统模块使用 `AskUserQuestion`（multiSelect），其余用自由文本
4. 产出 intake.md

### 深度萃取路径（3-8 轮）

**触发条件**：用户的需求方向本身不清晰，或存在多个相互关联的鉴别点。

**流程**：

**R1 — 邀请信息倾倒**：
> "请把你知道的关于这个需求的所有信息告诉我——背景、痛点、初步想法、相关文档、之前的讨论结论都可以。格式不限，我来整理。"

**R2 — 理解假设 + 鉴别问题**：
AI 消化全部输入后，输出理解摘要 + 已明确项 + 关键鉴别点（5-8 个问题，每个附带 AI 当前假设和替代可能）。

**R3-N — 迭代深入**：
- 每轮更新理解摘要 + 新的鉴别问题
- 在对话中自然编织**合理性质疑**和**已知背景推断**（反转知识诅咒）

**收敛**：当 AI 判断理解足够自洽时：
> "我对需求的理解已经比较清晰了。{一句话总结}。还有 {N} 个鉴别点尚未完全确认，我会在 PRD 中标注 `[待确认]`，你审阅时重点看这些。我开始写？"

## 鉴别诊断的内部参考框架

以下 9 个维度是 AI **内部使用**的 gap 检测框架，**不逐项暴露给用户，不作为清单逐项提问**——仅在发现某维度存在可能导致方向性偏差的鉴别点时，才针对性地探测。

1. **业务边界** — 哪些场景在范围内/外？需要留扩展口吗？
2. **数据流** — 数据从哪来、怎么处理、到哪去？时效性？量级？
3. **系统集成** — 现有接口变更？新增接口？触发时机？失败处理？
4. **异常处理** — 每步的失败场景？并发冲突？降级策略？
5. **状态流转** — 涉及哪些实体状态？触发条件？不可逆转换？
6. **权限控制** — 角色操作映射？审批流程？数据可见范围？
7. **非功能性** — 性能要求？并发量？安全？兼容性？
8. **风险** — 上线后可能出什么问题？外部依赖的信心？
9. **数据迁移** — 历史数据怎么处理？能否停机？（仅数据结构变更时）

## 假设检验式自问协议

Stage A 的每一轮对话，AI 在内部执行**倾听→解释→澄清**循环（不暴露给用户）。

### 每轮自问（收到用户回复后、输出前）

1. 倾听：用户刚才的回复告诉了我什么新信息？修正了我的哪个假设？
2. 解释：基于全部已知信息，我当前的理解是什么？列出 2-3 个并行假设，标注各自可能性。
3. 区分：这些假设的关键分歧点在哪里？什么问题能一次区分不同 condition？
4. 优先级：哪个分歧点对方案方向影响最大？先问这个。

### 收敛自问（决定是否进入 Stage B 前）

1. 开发者测试：如果现在把我的理解说给开发，他知道要做什么吗？
2. 替代方案测试：我能说出至少一个考虑过但排除了的替代方案吗？
3. 标记测试：打算标的 [待确认] 里，有没有方向性的？方向层面有 [待确认] 则继续收集。

### 供应链需求 Red Flags（高频误判模式）

| Red Flag | 信号 | 区分性问题 |
|----------|------|-----------|
| "加字段"实为接口变更 | 新字段的值需从其他系统获取 | "这个字段的值从哪来？谁还用？" |
| "优化流程"实为重建 | 用户说不清当前流程 | "当前具体是怎么操作的？走几步？" |
| "改规则"牵连状态机 | 规则变更涉及有状态流转的实体 | "这个规则变了，哪些状态转换会受影响？" |
| "加报表"实为缺数据 | 想看的指标没有直接数据源 | "这个指标的原始数据在哪个系统的哪张表？" |
| "参考竞品"实为未定义 | 用"像XX那样"代替具体描述 | "具体来说，你期望的操作步骤是？" |
| "小改动"跨系统 | 看似单模块改动但涉及上下游 | "这个改动，下游系统需要感知吗？" |

## 一次性结构化提问模板

当 AI 判断需要一次性补充多项信息时，采用混合提问方式：

**第一步**：对"涉及哪些系统/模块"使用 `AskUserQuestion`（multiSelect）。
**第二步**：对鉴别点以自由文本方式提问（只问最关键的 2-4 个）。

## 已知背景推断

在 intake.md 产出后、进入 Stage B 前，推断一组"团队已知背景"，写入 intake.md 的 `known_context` 字段，标记 `[推断]`。

**判断标准**：所有目标读者（撰写者 + 开发 + QA + BA + 决策者）都已知的才列入。作者知道但开发可能不知道的，不列入。

Stage C 审阅时，用户可修改 `known_context`。

## 知识库补充策略

1. 读取 `knowledge-base/_index.md` 确定覆盖范围
2. 根据用户提到的系统/模块，读取对应的知识卡片
3. 读取 `glossary.yaml` 获取术语定义
4. 知识库中的信息视为"已确认事实"，但保持知识库中已有的 `[待确认]` 或 `[过时?]` 标注
5. 在 intake.md 的信息来源摘要中注明引用了哪些知识库文件
