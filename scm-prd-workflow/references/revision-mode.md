# 修订模式 (Revision Mode)

## 目标

对已交付的 PRD 进行版本升级（V1→V2），支持评审返工、需求追加、上游变更三种修订场景。

## 设计原则

1. **修订不是重写**：修订模式的核心是增量更新——只改需要改的，保留未变更章节原文
2. **正交入口**：修订模式不是第 4 种生产模式，而是独立的入口。进入修订后，内部使用自主模式（默认）或交互模式执行。轻量模式不适用修订（修订必然涉及 AS-IS/TO-BE 对比）
3. **可追溯**：每次修订产出 `revision-diff.md`，结构化记录所有变更，供评审者快速理解"改了什么、为什么改"
4. **跨 PRD 感知**：通过 `_constraints-index.yaml` 检测本次修订对其他 PRD 的潜在影响

## 阶段结构

| 阶段 | 名称 | 核心动作 | 产出 |
|------|------|---------|------|
| Stage RV-A | 读取与理解 | 读取原 PRD + 约束索引 + 收集修订意图 | 修订范围（内部） |
| Stage RV-B | 增量规划 | 仅对变更部分做叙事规划 + 确认详略 | 增量写作计划（内部） |
| Stage RV-C | 增量撰写 | 重写受影响章节 + 生成 revision-diff | 更新后的 PRD + revision-diff.md |
| Stage RV-D | 修订审查 | 增量 CK 检查 + 跨 PRD 影响分析 + 交付 | review-report.md（增量）+ 更新后的约束索引 |

## Stage RV-A: 读取与理解

**进度播报**：`▰▱▱▱ [1/4] 读取原 PRD`

### 操作流程

1. **读取原 PRD 全文**：用户在初始化阶段选中的 PRD 文件
2. **读取关联文件**（如存在）：
   - `prd-changelog.md` → 了解历史修订记录
   - `review-report.md` → 了解上次自检结果
   - `decision-log.md` → 了解历史设计决策
   - `knowledge-discoveries.md` → 了解已发现的业务知识
3. **读取约束索引** `requirements/_constraints-index.yaml`：
   - 提取本 PRD 的当前约束条目
   - 提取其他 PRD 中引用了本 PRD 接口/实体的条目（用于 RV-D 影响分析）
4. **收集修订意图**（RV-01）

### 修订类型确认（RV-01）

使用 `AskUserQuestion`：

> header: "修订类型"
> 问题: "本次修订的触发原因是？"
> 选项：
> - **评审返工**：评审团队提出了修改意见，需要根据反馈调整 PRD
> - **需求追加**：业务方提出了新的功能点或变更点，需要在现有 PRD 基础上扩展
> - **上游变更**：依赖的其他 PRD 或外部系统发生了变化，需要同步调整本 PRD

用户选择后，进入自由文本描述修订内容。

**评审返工场景**的额外引导：
> "请提供评审反馈内容（可粘贴评审纪要或逐条列出修改意见）。"

**需求追加场景**的额外引导：
> "请描述需要追加的需求（新功能、新规则、新接口等）。如有相关文档可一并提供。"

**上游变更场景**的额外引导：
> "请说明哪个上游 PRD 或系统发生了什么变更，以及对本 PRD 的预期影响。"

### 修订范围分析

根据用户描述，AI 分析并输出修订范围：

```markdown
## 修订范围分析

**修订类型**：{评审返工 / 需求追加 / 上游变更}
**修订摘要**：{一句话概括}

### 受影响章节

| 章节 | 影响类型 | 说明 |
|------|---------|------|
| Ch.6 功能与规则 | 重写 | F-003 规则需修改 |
| Ch.5 业务流程 | 更新 | 主流程图需同步调整 |
| Ch.9 验收标准 | 更新 | F-003 对应验收标准需修改 |
| Ch.7 接口与数据 | 保留 | 不受影响 |
| ... | ... | ... |

### 功能点变更清单

| 操作 | ID | 名称 | 说明 |
|------|-----|------|------|
| 修改 | F-003 | {名称} | {修改说明} |
| 新增 | F-008 | {名称} | {新增说明} |
| 删除 | F-005 | {名称} | {删除原因} |
```

将修订范围呈现给用户确认后，进入 Stage RV-B。

## Stage RV-B: 增量叙事规划

**进度播报**：`▰▰▱▱ [2/4] 增量规划`

### 修订叙事

与新建 PRD 的叙事规划不同，修订叙事聚焦于**变更本身**：

```markdown
## 修订叙事

### 核心变化
{一句话：这次修订的本质是什么？例如："拆单规则从单一维度扩展为多维度，影响下游接收逻辑"}

### 表达维度与处理方式

| 维度 | 原章节 | 处理方式 | 说明 |
|------|--------|---------|------|
| 功能与规则（Ch.6） | V1.2 Ch.6 | **重写** | F-003 规则大幅调整 |
| 业务流程（Ch.5） | V1.2 Ch.5 | **局部更新** | 仅更新涉及 F-003 的流程节点 |
| 验收标准（Ch.9） | V1.2 Ch.9 | **局部更新** | 同步更新 F-003 的验收标准 |
| 接口与数据（Ch.7） | V1.2 Ch.7 | **保留** | 不受影响 |
| 需求概述（Ch.2） | V1.2 Ch.2 | **保留** | 不受影响 |
| ... | ... | ... | ... |

### 新增内容
- F-008: {简述}

### 图表影响
- `flow-main.diagram.yaml`：需更新（F-003 节点变更）
- 其余图表：保留
```

### 叙事规划自检

执行 NP-01 ~ NP-05 检查（规则详见 `phase3-write.md` "叙事规划自检"章节），但有以下修订模式适配：

- **NP-01 泛化叙事**：核心变化必须指向具体的修订原因和影响，不能写"优化拆单功能"
- **NP-04 脱离 intake**：改为检查是否脱离了 RV-A 确认的修订范围
- **NP-02/NP-03/NP-05**：正常执行

### 修订章节详略选择（RV-02）

使用 `AskUserQuestion`（multiSelect），仅列出受影响的章节：

> header: "修订章节详略"
> markdown: （展示修订叙事内容）
> 问题: "以下章节将被修改或新增，请选择需要**重点展开**的（可多选）："
> multiSelect:
> - **{受影响章节1}**：{处理方式说明}
> - **{受影响章节2}**：{处理方式说明}
> - ...

未受影响的章节不出现在选项中（它们将被逐字保留）。

## Stage RV-C: 增量撰写

**进度播报**：`▰▰▰▱ [3/4] 增量撰写`

### 核心原则

1. **只改需要改的**
   - **重写章节**：完整重写，按 `phase3-write.md` 对应章节的撰写指引执行
   - **局部更新章节**：仅修改受影响的段落/表格/图表，保留其余内容原文
   - **保留章节**：逐字保留，**禁止**做任何"顺手优化"（包括措辞润色、格式调整、补充说明等）

2. **关联更新最小化**
   - 修改 F-XXX 的规则 → 检查并更新 Ch.9 对应验收标准
   - 修改 IF-XXX → 检查并更新 Ch.7 接口规格
   - 新增/删除 F-XXX → 更新 Ch.2 范围列表、Ch.9 验收标准
   - 修改业务规则 → 检查 Ch.5 流程图是否需同步

3. **ID 连续性**
   - 新增功能点：在已有 F-XXX 序列之后递增（如已有 F-001~F-007，新增为 F-008）
   - 删除功能点：**不重新编号**已有 ID（保持外部引用稳定）
   - 修改功能点：保留原 ID

### 版本号递增

| 修订幅度 | 版本递增 | 判断标准 |
|---------|---------|---------|
| 重大修订 | V{X}.{Y} → V{X+1}.0 | 新增/删除功能点 ≥3，或修改了核心流程 |
| 一般修订 | V{X}.{Y} → V{X}.{Y+1} | 修改功能点 ≤2，或仅调整规则/参数 |

### 撰写流程

1. **读取撰写指引**：读取 `references/phase3-write.md` + `references/diagram-patterns.md`（如涉及图表更新）
2. **逐章处理**：按修订叙事中确定的处理方式（重写/局部更新/保留）逐章执行
3. **标记体系**：
   - 使用自主模式标记（`[推断]` / `[待确认]`）标注新增或修改的内容
   - 原文保留部分不添加任何新标记
4. **生成 revision-diff.md**：使用 `templates/revision-diff-template.md`，结构化记录所有变更
5. **更新 prd-changelog.md**：追加本轮修订记录（格式与 `templates/prd-changelog-template.md` 一致）
6. **更新 PRD front matter**：递增 `version`、更新 `updated` 日期

### 图表更新

- **受影响的图表**：重新生成 `.diagram.yaml` 或 `.mermaid`，运行 `scripts/export-diagrams.py` 导出
- **未受影响的图表**：保留原文件，不重新生成
- **新增图表**：按 `diagram-patterns.md` 规范创建

## Stage RV-D: 修订审查

**进度播报**：`▰▰▰▰ [4/4] 修订审查`

### 增量 CK 检查

仅对受影响的章节执行 CK-0 ~ CK-9（详见 `references/phase4-review.md`）。未变更章节不重复检查。

**额外检查项**：

| 检查项 | 方法 | 严重性 |
|--------|------|--------|
| CK-RV.1 | 新增/修改的 F-XXX 在 Ch.9 有对应验收标准 | 关键 |
| CK-RV.2 | 删除的 F-XXX 在 Ch.9 的验收标准已同步清理 | 关键 |
| CK-RV.3 | revision-diff.md 中的变更条目与 PRD 实际变更一致 | 关键 |
| CK-RV.4 | 未变更章节确实未被修改（与原文比对） | 警告 |
| CK-RV.5 | 新 ID 编号连续、无跳号或重复 | 警告 |

### 跨 PRD 影响分析

读取 `requirements/_constraints-index.yaml`，对本次修订的每个变更点检查：

1. **接口变更影响**：本 PRD 修改/删除的 IF-XXX，是否出现在其他 PRD 的 `interfaces` 或 `scope_in` 中？
2. **实体变更影响**：本 PRD 修改/删除的数据实体，是否出现在其他 PRD 的 `entities` 中？
3. **规则冲突检查**：本 PRD 修改的业务规则，是否与其他 PRD 的 `key_rules` 存在矛盾？

**影响分析输出**（写入 review-report.md 的新章节）：

```markdown
## 跨 PRD 影响分析

### 影响汇总
| 受影响 PRD | 影响内容 | 影响类型 | 严重性 | 建议操作 |
|-----------|---------|---------|--------|---------|
| PRD-inventory-alert (V1.0) | IF-001 key_fields 新增 carrier_code | 接口 | 高 | 需同步修订 |

### 详细说明
1. **PRD-inventory-alert**：该 PRD 的 §7 引用了 IF-001（OMS→WMS 拆单推送），本次修订新增了 carrier_code 字段。如 inventory-alert 的接收逻辑未适配新字段，可能导致数据丢失。建议在本 PRD 交付后对 PRD-inventory-alert 发起修订。
```

如约束索引不存在或为空，跳过影响分析并提示："约束索引尚未建立，无法进行跨 PRD 影响分析。建议在所有已有 PRD 交付后补充约束索引。"

### 审查与交付

审查流程复用 `references/phase4-review.md` "最终交付"章节，但有以下修订模式特有行为：

1. 交付选项增加 `revision-diff.md`（始终包含，无需用户选择）
2. 如选中 Word 输出，`revision-diff.md` 也生成 `.docx` 版本
3. 更新 `_constraints-index.yaml` 中本 PRD 的条目（版本号、变更后的约束）

**修订交付确认（RV-03）**：

使用 `AskUserQuestion`：

> header: "修订交付"
> 问题: "修订后的 PRD 已通过自检，以下为本次修订产出："
> markdown: （展示 revision-diff.md 摘要 + 跨 PRD 影响分析摘要）
> 选项：
> - **确认交付**：所有修订内容确认，生成最终文件
> - **需要调整**：返回 Stage RV-C 继续修改

### 产出文件

```
requirements/REQ-{date}-{name}/
├── PRD-{name}.md                  # 更新后的 PRD（版本号已递增）
├── PRD-{name}.docx                # Word 版本（如选中）
├── revision-diff.md               # 修订摘要（本次新增）
├── revision-diff.docx             # 修订摘要 Word 版（如选中 Word）
├── review-report.md               # 自检报告（含跨 PRD 影响分析）
├── prd-changelog.md               # 版本历史（追加本轮记录）
├── decision-log.md                # 设计决策（追加本轮决策）
├── knowledge-discoveries.md       # 知识发现（追加本轮发现）
└── diagrams/                      # 更新后的图表文件
```

## 执行方式选择

进入修订模式后，AI 默认以**自主模式**执行修订（一次性输出修订后的内容供审阅）。如用户需要逐步确认，可在 RV-A 阶段声明"我想逐步确认修改"，切换为交互模式执行：

| 执行方式 | 适用场景 | RV-B 行为 | RV-C 行为 |
|---------|---------|-----------|-----------|
| **自主执行**（默认） | 修订范围清晰、改动明确 | 叙事规划输出后直接进入 RV-C | 一次性输出所有修改，集中审阅 |
| **交互执行** | 修订范围大、需逐章讨论 | 叙事规划后逐章确认修改方案 | 逐章撰写、逐章确认 |

切换方式：用户在任意阶段说"切换到交互/自主"即可。切换时保留已有产出。

## 模式互通

| 场景 | 处理方式 |
|------|---------|
| 修订过程中发现需要新建独立 PRD | 暂存当前修订进度，启动新 PRD 流程（MC-01），完成后可回来继续修订 |
| 修订幅度过大（新增功能 ≥5 或重写章节 ≥6） | 建议终止修订模式，改为新建 PRD 并引用原 PRD 作为基线 |
| 修订触发了其他 PRD 的连锁修订 | 在 RV-D 跨 PRD 影响分析中标注，由用户决定是否在本会话中继续修订受影响的 PRD |
