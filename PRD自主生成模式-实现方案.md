# PRD自主生成模式 — 实现方案

## Context

当前PRD工作流（4阶段：录入→澄清→撰写→自检）采用全交互模式，每个阶段都需要大量用户问答。对于用户已有较完整背景信息或希望快速出初稿的场景，效率不够理想。

**目标**：新增"自主生成模式"，在用户提供初始信息后，AI可一次性自主生成完整PRD，仅在关键决策点提问，完成后交由用户审阅迭代。

**核心设计理念**：自主模式不是"跳过质量"，而是将交互从"逐步阻塞式确认"变为"先生成后批量审阅"。

---

## 整体架构

```
用户启动PRD技能
       │
       ▼
   启动初始化（读取知识库、检查未完成需求）
       │
       ▼
   ┌─ 模式选择 ─────────────────────────┐
   │                                     │
   ▼                                     ▼
自主生成模式（默认）             交互模式（用户主动切换）
Stage A → B → C                  Phase 1→2→3→4
压缩录入→自主生成→审阅迭代       逐阶段Q&A确认
```

**自主模式为默认模式。两种模式并行存在，可随时切换。**

---

## 模式选择与切换机制

### 默认行为

自主模式为默认。启动后AI的开场提示：

> 我将以**自主模式**为你生成PRD。请描述你的需求背景，我会在收集关键信息后一次性输出完整PRD供你审阅修改。
>
> 如你希望逐步讨论每个细节，随时说"**切换到交互模式**"。

### 可切换的5个时机及提示语

| # | 时机 | 方向 | 触发方式 | AI提示语 |
|---|------|------|---------|---------|
| 1 | **启动时** | 自主→交互 | 用户说"交互模式""逐步讨论""我想一步步来" | （直接进入Phase 1，无额外提示） |
| 2 | **Stage A 信息不足时** | 自主→交互 | AI主动建议（硬性条件缺3+项） | "你提供的信息中，{缺失项}还不够明确。建议切换到交互模式逐步梳理，确保PRD质量。是否切换？" |
| 3 | **Stage C 审阅时** | 自主→交互（局部） | AI主动建议（某维度>50%假设需修改） | "关于{XX部分}有较多内容需要重新确认，建议对这部分切换到交互模式深入讨论，其余部分继续自主模式。" |
| 4 | **交互模式任意阶段边界** | 交互→自主 | 用户说"快速出吧""直接帮我写""自主生成" | "好的，基于已有的{Phase X}信息，我将自主生成剩余部分。生成后你可以集中审阅。" |
| 5 | **任何时候** | 双向 | 用户说"切换模式" | 自主→交互："好的，切换到交互模式。我们从{当前阶段对应的Phase}开始逐步确认。"<br>交互→自主："好的，切换到自主模式。基于已收集的信息，我将一次性生成完整PRD。" |

### 切换时的状态保留

- **自主→交互**：Stage A 已产出的 intake.md 作为 Phase 1 的初始摘要，用户可在其基础上补充，不需从零开始
- **交互→自主**：已完成的 Phase 产出文件（intake.md / clarification.md）直接作为 Stage B 的输入，AI在此基础上补充推断并生成PRD
- **局部切换**（Stage C中）：仅对特定章节/维度进入交互讨论，讨论完毕后AI将结论更新到PRD中，不影响其他章节

## 自主生成模式：三个阶段

### Stage A: 压缩录入（1-2轮对话）

**目标**：用最少对话获取足够生成PRD的信息。

**操作**：
1. 如用户已提供大段背景 → 直接分析提取，评估信息充分度
2. 如信息不足 → 发送一次性结构化提问（5个核心问题）
3. 读取 `knowledge-base/` 补充领域背景
4. 评估信息充分度，最多追问1轮
5. 产出 `intake.md`（标注信息来源：用户输入/知识库/AI推断）

**信息充分度标准**：
- **硬性条件**（4项必备）：核心业务问题、涉及系统模块、主要用户角色、期望目标
- **软性条件**（5项可推断）：系统现状、约束条件、异常处理偏好、非功能性需求、上下游影响
- 硬性条件全满足 → 进入 Stage B
- 缺1-2项硬性 → 追问1轮后进入
- 缺3+项硬性 → 建议切换交互模式

### Stage B: 自主生成（无用户交互）

**目标**：一次性生成完整PRD及全部附属文件。

**生成顺序**：
1. 整理信息 → 输出 `intake.md`
2. AI自主推导假设 → 输出 `clarification.md`（标记假设依据）
3. 撰写15章PRD → 输出 `PRD-{名称}.md` + 流程图
4. 自动执行自检 → 输出 `review-report.md`

**内容决策规则**（三级标记体系）：

| 标记 | 含义 | 适用场景 | 示例 |
|------|------|---------|------|
| （无标记） | 已确认事实 | 用户明确说的、知识库中的 | 用户说"涉及OMS和WMS" |
| `[推断]` | 专业推断 | 行业标准做法、由事实推导 | 接口超时重试3次后转人工 |
| `[待确认]` | 需用户确认 | 业务边界、特定规则、多方案决策 | 库存不足时是否自动触发采购 |

**`[待确认]`** 的格式：
```markdown
> [待确认] 当库存不足时，系统是否自动触发采购申请？
> 当前假设：仅通知仓库主管，不自动触发。
> 如需变更，请在审阅时指出。
```

**原则合规方式**：

| 原则 | 交互模式 | 自主模式 |
|------|---------|---------|
| 禁止臆想 | 不确定就问 | 不确定就标 `[待确认]` 并给假设 |
| 质疑精神 | 对话中质疑 | PRD第2章末增"需求合理性分析"段落 |
| 证据驱动 | 对话中引用 | 标注来源（知识库/行业实践/推断） |
| 授权写入 | 逐条确认 | 全部写入+分级标记，审阅时批量确认 |
| 增量发现 | 实时提醒 | intake.md末尾汇总知识库补充建议 |

### Stage C: 审阅迭代（用户主导）

**目标**：用户审阅PRD，批量反馈，AI迭代修改。

**呈现顺序**：
1. **假设总览表** — 汇总所有 `[待确认]` 和 `[推断]`，按影响程度排序
2. **自检报告摘要** — CK-1~CK-8 检查结果
3. **完整PRD** — 可先看目录再看全文

**用户反馈方式**：
- 批量确认："假设1-5确认"
- 逐条修改：针对某条假设给出正确信息
- 局部深入讨论：某个复杂点进入多轮对话
- 全局反馈："整体OK但第8章需要细化"

**迭代规则**：
- 修改后仅重新自检受影响章节
- 确认后批量移除标记
- 如大量假设需修改 → AI建议切换到交互模式处理该部分

## 验证方式

1. **交互模式回归**：使用 scm-prd-workflow 技能，走标准交互流程，确认行为与修改前一致
2. **自主模式触发**：对技能说"帮我快速出一份PRD"，确认进入自主模式流程
3. **压缩录入**：提供一段需求背景，确认AI发送结构化提问（或直接评估充分度）
4. **自主生成**：确认AI一次性产出 intake.md、clarification.md、PRD、review-report.md，且包含三级标记
5. **审阅迭代**：确认AI呈现假设总览表，用户可批量确认/修改，AI能迭代更新
6. **回退测试**：在审阅阶段大量修改假设，确认AI建议切换交互模式
